# 3.1 데이터베이스 기본
- DBMS 아키텍처·구축 방법론
    - DBMS (Database Management System, 데이터베이스 관리 시스템): 데이터 정의·조작·제어·무결성·동시성·회복을 통합 제공하는 소프트웨어.
    - 3계층 스키마 (Three-Schema Architecture, 3단계 스키마): 외부 스키마(사용자 뷰)·개념 스키마(논리 구조)·내부 스키마(물리 저장)로 데이터 독립성 확보.
    - DB구축 방법론 v4.0 공정 — 현황조사→구축대상 선정→계획수립→실행계획→구축→검수 순서
        - 실행계획 세그먼트: 메타데이터 입력항목 협의·다국어 지원 협의·표준안 수립·작업지침 작성 등 구체적 실행 준비 태스크 집결.
        - 계획수립 세그먼트: 전체 프로젝트 일정·인력·범위 계획. 구체적 표준안 작성은 실행계획 단계 소관.
- 릴레이션 저장·레코드 관리
    - 최대 투플 수 (Maximum Cardinality, 최대 카디널리티): 기본키 구성 속성들의 도메인 크기 곱. 비주요 속성은 계산에서 제외.
    - 고정 길이 레코드 블로킹 (Fixed-length Record Blocking, 고정 길이 블로킹 인수): blocking factor = floor((블록 크기 − 블록 헤더) / 레코드 크기). 블록별 잔여 공간 = 블록 크기 − 헤더 − 레코드 수×레코드 크기.
    - Heap File (비순서 힙 파일): 무작위 삽입, 레코드 정렬 없음. 평균 검색 비용 = (총 블록 수 / 2) × 블록 접근 비용.
    - ROWID (Row Identifier, 행 식별자): 오브젝트 번호·데이터 파일 번호·블록 번호로 구성. 블록 읽기 순서 ROWID→해시값→해시체인 탐색→(미스 시) LRU 빈 버퍼→Dirty 버퍼 플러시→디스크 읽기.
- 인덱스 구조
    - B-트리 (B-Tree, 균형 다진 탐색 트리) — 차수 m일 때 노드당 최소 ⌈m/2⌉−1개, 최대 m−1개 키
        - 삽입 동작: 오버플로우 시 노드 분할(split)·중간 키 부모 승격. 루트 분할 시 트리 높이 증가.
        - 삭제 동작: 언더플로우 시 재분배(redistribution) 또는 병합(merge). 삽입에서는 병합 발생 안 함.
    - B+-트리 (B+ Tree, B+ 트리): 모든 키가 리프 노드에 저장되고 내부 노드는 탐색용 키만 보유. 리프끼리 연결 리스트로 연결되어 범위 질의에 유리.
    - Extendible Hashing (확장 해싱, 확장 가능 해싱) — 디렉터리와 버킷을 동적으로 확장
        - Global Depth (전역 깊이, d): 디렉터리 항목 수 = 2^d. 해시값 상위 d비트로 디렉터리 인덱싱.
        - Local Depth (지역 깊이, d'): 각 버킷이 사용하는 비트 수. d' ≤ d, 2^(d−d')개의 디렉터리 항목이 동일 버킷 공유.
        - 삽입 시 버킷 초과: 지역 깊이 증가·분할. 지역 깊이가 전역 깊이 초과하면 디렉터리 2배 확장.
        - 삭제 시 버디 합병: 버디 버킷과 지역 깊이 같고 합병 후 크기 ≤ 최대면 합병. 모든 지역 깊이 < 전역 깊이면 디렉터리 축소.

# 3.2 데이터베이스 설계
- 함수적 종속성·후보키
    - Functional Dependency (함수적 종속, FD): X → Y는 X 값이 같으면 Y 값도 같아야 함. 부분 함수 종속·이행 함수 종속 식별이 정규화의 출발점.
    - Armstrong's Axioms (암스트롱 공리, 추론 규칙): 반사율(X⊇Y → X→Y), 확장율(X→Y → XZ→YZ), 이행율(X→Y, Y→Z → X→Z). 유도 규칙으로 합집합·분해율·의사이행율.
    - Candidate Key (후보키): 속성 집합의 클로저(closure)가 전체 속성을 포함하는 최소 슈퍼키. BD의 클로저에 C·F·G 없으면 후보키 아님.
    - Superkey (슈퍼키): 튜플을 유일하게 식별하는 속성 집합. 후보키를 포함한 모든 상위집합이 슈퍼키.
    - Closure Computation (속성 클로저 계산): 주어진 FD 집합에서 속성 집합이 결정 가능한 모든 속성 도출. 후보키 판별의 핵심 도구.
- 정규화 (Normalization, 정규형 단계)
    - 1NF (First Normal Form, 제1정규형): 모든 속성이 원자값(atomic). 다치·복합 속성 제거.
    - 2NF (Second Normal Form, 제2정규형): 1NF + 기본키에 대한 부분 함수 종속 제거. 복합 기본키의 일부에만 종속되는 비주요 속성 분리.
    - 3NF (Third Normal Form, 제3정규형): 2NF + 비주요 속성의 이행 함수 종속(transitive FD) 제거.
    - BCNF (Boyce-Codd Normal Form, 보이스-코드 정규형): 모든 비자명 FD의 결정자가 후보키. 3NF보다 엄격. zipcode→city처럼 비후보키 결정자가 있으면 위반.
    - 4NF (Fourth Normal Form, 제4정규형): BCNF + 비자명 다치 종속(MVD) 결정자가 슈퍼키. 하나의 테이블에 두 개 이상 독립적 MVD 공존 시 위반.
    - Multivalued Dependency (다치 종속, MVD, X →→ Y): X 값이 결정하는 Y 값 집합이 다른 속성과 독립. {직원이름→→학위명, 직원이름→→형제이름} 공존은 4NF 위반.
    - Lossless Decomposition (무손실 분해, 손실없는 조인 분해): R1∩R2 → R1−R2 또는 R1∩R2 → R2−R1 중 하나 만족(Heath 정리). 속성 커버리지(R1∪R2 = R)가 선행 조건.
    - Dependency Preservation (종속성 보존): 분해 후에도 원본 FD를 각 릴레이션에서 검증 가능. BCNF 분해는 무손실은 보장하나 종속성 보존은 보장되지 않음.
- E-R 모델링·제약조건
    - Weak Entity Type (약한 엔티티 타입): 자체 키 부족, 소유 엔티티(strong/owner entity)의 기본키 + 부분 키(partial key)로 식별. 식별 관계에 반드시 전체 참여(total participation).
    - Identifying Relationship (식별 관계): 약한 엔티티와 소유 엔티티를 연결. ER 다이어그램에서 이중 마름모로 표기.
    - Category / Union Type (범주, 유니온 타입): EER에서 이종(heterogeneous) 엔티티들(PERSON·BANK·COMPANY)의 합집합으로 슈퍼타입 OWNER 구성. 일반화(공통 속성 추출)와 구분.
    - Aggregation (집단화): 관계를 상위 엔티티로 추상화하는 EER 개념. 범주와 혼동 주의.
    - Participation Constraint (참여 제약) — 부분 참여 vs 전체 참여
        - Total Participation (전체 참여): 모든 엔티티가 관계 참여 필수. DDL에서 외래키 NOT NULL 제약으로 구현.
        - Partial Participation (부분 참여): 일부 엔티티만 관계 참여. 외래키 NULL 허용.
- ER → 관계 모델 변환
    - 1:1 관계: 한쪽 엔티티에 상대 PK를 FK로 추가하거나 두 엔티티 병합. 관계 속성은 한쪽 엔티티로 이동(양쪽 중복 저장은 무결성 위반).
    - 1:N 관계: N쪽 엔티티에 1쪽 PK를 FK로 추가. 별도 릴레이션 불필요.
    - N:M 관계: 반드시 별도 관계 릴레이션 생성. 양쪽 PK를 FK로 포함, 복합키가 기본키. 관계 속성(Grade 등)도 포함.
    - Multivalued Attribute (다치 속성, 이중 타원): 별도 릴레이션으로 분리. 예: PROJECT_LOCATION(PNO, Location).
- 무결성 제약
    - Domain Constraint (도메인 제약): 속성 값이 지정 도메인 범위 이내.
    - Key Constraint (키 제약): 기본키는 릴레이션 내 유일.
    - Entity Integrity (엔티티 무결성): 기본키 속성은 NOT NULL.
    - Referential Integrity (참조 무결성): 외래키 값은 참조 릴레이션의 기본키 값 존재 또는 NULL. 삽입/삭제/갱신 시 위배 가능.
    - Foreign Key Nullability (외래키 NULL 허용): FK는 NULL 가능(선택적 참조). 한 릴레이션에 여러 FK 존재 가능.

# 3.3.1 관계대수·질의 최적화
- 관계대수 기본 연산
    - Selection (σ, 셀렉션): 조건을 만족하는 튜플 추출.
    - Projection (π, 프로젝션): 지정 속성만 추출, 중복 제거.
    - Cartesian Product (×, 카티전 곱): 두 릴레이션 튜플의 모든 조합. |R|×|S| 고정.
    - Set Operations (집합 연산): 합집합 상한 |R|+|S|, 교집합 상한 min(|R|,|S|), 차집합 상한 |R|.
    - Rename (ρ, 리네임): 속성·릴레이션 이름 변경.
- 조인 연산·카디널리티
    - Natural Join (⋈N, 자연 조인): 공통 속성으로 동등 조인 후 중복 속성 제거. 매칭되는 튜플만 반환.
    - Theta Join (⋈θ, 세타 조인): 임의 조건 θ로 카티전 곱 필터. 조건 느슨할수록 카디널리티 증가.
    - Outer Join (외부 조인) — 매칭 없는 튜플에 NULL 채움
        - Left Outer Join (⋈+, 좌외부 조인): 왼쪽 튜플 전부 유지, 오른쪽 매칭 없으면 NULL.
        - Right Outer Join (우외부 조인): 오른쪽 튜플 전부 유지.
        - Full Outer Join (완전 외부 조인): 양쪽 모두 유지. 매칭 없는 쪽에 NULL. 매칭 튜플이 있으면 해당 쪽은 NULL 생성 불가.
    - Semi Join (⋉, 세미 조인, R⋉S): R과 S 조인 조건 만족하는 R 튜플만 R 스키마로 반환. S 속성 미포함. 분산 환경 데이터 전송 감소.
- Division (÷, 나눗셈, 디비전)
    - 정의: R÷S = S의 모든 B 값에 대해 (A,b) 튜플이 R에 존재하는 A 값 집합. "모든 ~에 대해" 전칭 한정 질의에 사용.
    - 대수적 분해: R÷S = πR−S(R) − πR−S((πR−S(R)×S) − R). 필요 연산은 프로젝트·차집합·카티전 곱 세 가지.
    - SQL 대응: NOT EXISTS(A EXCEPT B) 또는 이중 NOT EXISTS 패턴.
    - 완전 집합 최소 부분집합: {π, −, ×}만으로 디비전 표현 가능. σ·∪·ρ는 불필요.
- 질의 최적화 동등 규칙 (Equivalence Rules, 등가 변환 규칙)
    - 선택 분해: σθ1∧θ2(E) = σθ1(σθ2(E)). 복합 조건을 단일 조건 중첩으로 분해.
    - 선택-조인 흡수: σθ1(E1 ⋈θ2 E2) = E1 ⋈θ1∧θ2 E2. 조기 선택(push-down)으로 튜플 수 감소.
    - 선택-차집합 분배: σθ(E1 − E2) = σθ(E1) − E2. E2는 이미 필터 역할 수행.
    - 프로젝션-합집합 분배 주의: π L1∪L2(E1 ∪ E2) 분해는 스키마 동일(L1=L2) 조건 필요. 일반적으로 성립 안 함.

# 3.3.2 SQL 질의·조인·집계·서브쿼리
- COUNT·집계 함수·GROUP BY
    - COUNT(*): NULL 포함 전체 행 수. 중복 포함.
    - COUNT(속성명): 해당 속성 NULL 제외 행 수. 중복 포함.
    - COUNT(DISTINCT 속성명): NULL 제외 + 중복 제거한 유일값 수.
    - DISTINCT와 NULL: SQL 표준에서 NULL도 하나의 고유값으로 취급. DISTINCT 결과에 NULL 그룹 포함.
    - GROUP BY와 NULL: NULL도 별도 그룹으로 집계. COUNT(*)는 NULL 그룹에서도 해당 행 수 반환.
    - HAVING 절: GROUP BY 후 집계 결과 필터. HAVING SUM(...) >= 0은 NULL 그룹을 UNKNOWN으로 제외. HAVING COUNT(*) >= 0은 항상 참.
- JOIN 종류
    - INNER JOIN: 양쪽 매칭 튜플만 반환. NULL FK는 조인 키로 매칭 실패 → 결과에서 제외.
    - LEFT OUTER JOIN: 왼쪽 전체 행 유지. 오른쪽 매칭 없으면 NULL. 1:N 매칭 시 N쪽 매칭 수만큼 행 증가.
    - RIGHT OUTER JOIN: 오른쪽 전체 행 유지.
    - FULL OUTER JOIN: 양쪽 전체 행 유지. 매칭 없는 쪽 NULL.
    - Self Join (자체 조인): 동일 테이블을 별칭으로 참조. 직원-상사 관계(E.MANAGER = M.ENO) 등. NULL FK 튜플은 조인 제외.
    - JOIN 문법 구분: JOIN...ON (명시적 조건), NATURAL JOIN (동일 컬럼 자동), JOIN...USING (지정 컬럼). NATURAL JOIN과 ON/USING 혼용은 문법 오류.
- 서브쿼리·집합 비교
    - EXISTS / NOT EXISTS (상관 서브쿼리): 외부 쿼리 행마다 내부 쿼리 실행. NULL 안전, NOT IN 대신 권장.
    - IN / NOT IN: 값 집합 비교. 서브쿼리에 NULL 포함 시 NOT IN은 모든 행에 UNKNOWN 반환 → 공집합.
    - ALL: 서브쿼리 모든 값보다 조건 만족. "> ALL"은 최댓값보다 큰 조건. "모든 사원보다" 패턴.
    - ANY / SOME: 하나 이상의 값과 조건 만족. "> ANY"는 최솟값보다 큰 조건. "어느 한 사원보다" 패턴.
    - EVERY / ONE: SQL 표준 비교 연산자 아님.
    - 전칭 한정 패턴: NOT EXISTS(A EXCEPT B) = "A의 모든 원소가 B에 포함" = A ⊆ B. 디비전(÷)과 동치. "모든 X를 reserve한 Sailor" 질의의 표준 구현.
    - 교집합 구현: "모든 color 조건 만족"은 AND가 아닌 Self-Join 또는 INTERSECT. 다중 행 AND는 자기 조인으로 처리.
- 정렬·결과 해석
    - ORDER BY 다중 기준: 첫 기준(예: CNO DESC)으로 정렬 후 동일 값이면 다음 기준(GRADE ASC)으로 정렬.
    - Degree (차수): SELECT 절 컬럼 수.
    - Cardinality (카디널리티): 결과 행 수. GROUP BY·HAVING·DISTINCT 적용 후 값 계산.

# 3.3.3 DDL·무결성·View·Trigger·JDBC
- View (뷰, 가상 테이블)
    - CREATE VIEW 구문: SELECT 기반 가상 릴레이션. 집계·GROUP BY·조인 포함 가능하나 갱신 제한.
    - 갱신 불가 조건: ①두 개 이상 테이블 조인, ②DISTINCT/GROUP BY 사용, ③기본키 미포함, ④상수·산술식·집계 함수로 만든 열.
    - 갱신 가능성 동심원: 이론적 갱신 가능 뷰 ⊂ DBMS 허용 뷰 ⊂ 사용자 정의 가능한 모든 뷰.
    - Materialized View (실체화 뷰, 구체화 뷰): 물리적으로 저장된 뷰. CREATE MATERIALIZED VIEW 구문 사용. 일반 뷰(CREATE VIEW)와 구분.
    - Query Modification (쿼리 수정, 뷰 질의 치환): 뷰 정의를 외부 쿼리에 인라인 확장. 조인 조건·필터 조건·SELECT 대상 모두 원본 테이블 기준으로 재작성.
- 참조 무결성 액션 (ON DELETE / ON UPDATE)
    - CASCADE (연쇄): 부모 행 변경·삭제 시 자식 FK도 동일하게 변경·삭제.
    - SET NULL: 부모 행 삭제 시 자식 FK를 NULL로 설정. 자식 FK가 NOT NULL이면 사용 불가.
    - SET DEFAULT: 부모 행 삭제 시 자식 FK를 DEFAULT 값으로 설정. NULL 아님.
    - NO ACTION / RESTRICT: 참조 중인 부모 행 삭제·변경 금지.
    - 방향성: 제약은 부모 테이블 변경 시 자식에 적용. 자식 테이블에 FOREIGN KEY...REFERENCES...ON DELETE/UPDATE 명시.
- Trigger vs Stored Procedure
    - Trigger (트리거): 테이블/뷰에 할당. INSERT/DELETE/UPDATE 이벤트 발생 시 DBMS가 자동 실행. 입력 인자 없음.
    - Stored Procedure (저장 프로시저): 입력 매개변수·결과 반환 가능. CALL/EXECUTE로 명시적 호출. DB에 컴파일되어 저장.
    - 공통점: 둘 다 DB에 저장·컴파일 상태로 보관.
- JDBC / SQL-CLI 호출 순서
    - 호출 순서: 환경변수 설정 → SQLPrepare → SQLBindParameter → SQLExecute → SQLBindCol.
    - SQLPrepare (문장 준비, 파싱·최적화): SQL 서버로 전송하여 컴파일.
    - SQLBindParameter (매개변수 바인딩): 준비된 SQL의 ? 자리를 프로그램 변수에 연결.
    - SQLExecute (실행): 바인딩된 값으로 SQL 실행.
    - SQLBindCol (결과 열 바인딩): 결과 집합의 각 열을 C 프로그램 변수에 연결하여 Fetch 준비.
    - Prepared Statement (준비된 문장): 동일 SQL 반복 실행 시 파싱 비용 절감. SQL Injection 방지 효과.
- GRANT / REVOKE (권한 제어)
    - GRANT SELECT, UPDATE ON 테이블 TO 사용자: 권한 부여.
    - WITH GRANT OPTION: 부여받은 사용자가 제3자에게 동일 권한 재부여 가능. OPTION GRANT·REGRANT는 비표준.
    - REVOKE...CASCADE: 연쇄 부여된 권한까지 회수.

# 3.4 데이터베이스 관리
- 조인 구현 알고리즘
    - Nested Loop Join (중첩 루프 조인): 외부 릴레이션 각 튜플에 대해 내부 릴레이션 전체 스캔. O(n×m). 정렬·인덱스 불필요.
    - Index Nested Loop Join (인덱스 중첩 루프 조인): 내부 릴레이션 조인 속성에 인덱스 있을 때 가속.
    - Sort-Merge Join (정렬-합병 조인): 양쪽 릴레이션을 조인 속성으로 정렬 후 동시 스캔·병합. O(n+m), 이미 정렬된 경우 매우 효율적.
    - Hash Join (해시 조인): 조인 속성에 해시 함수 적용해 버킷 분류. 대용량에 효율적. 메모리 필요.
    - Partition-Hash Join (분할-해시 조인): 해시 조인 변형. 파티션 나눠 처리.
    - k-NN Join: 공간·유사도 검색 특화, 일반 관계형 조인 알고리즘 아님.
- 트랜잭션 동시성 이상 현상 & 격리 수준 (Isolation Level)
    - **격리 수준별 이상 현상 방지표** (O=발생, X=방지)

        | 격리 수준 | Dirty Read | Non-repeatable Read | Phantom Read |
        |---|:---:|:---:|:---:|
        | Read Uncommitted | O | O | O |
        | Read Committed | **X** | O | O |
        | Repeatable Read | **X** | **X** | O |
        | Serializable | **X** | **X** | **X** |

        > **Lost Update(갱신 손실)**는 모든 격리 수준에서 발생 가능. 별도 잠금·낙관적 동시성 제어로 해결.

    - **Dirty Read (오손 데이터 읽기)**: 커밋되지 않은 트랜잭션 변경 데이터를 다른 트랜잭션이 읽음. T1 ROLLBACK 시 읽은 데이터 무효화.
        - **방지 격리 수준**: Read Committed 이상.
    - **Non-repeatable Read (반복불가 읽기)**: 동일 트랜잭션 내 동일 행 재조회 시 값이 달라짐. 다른 트랜잭션의 UPDATE/DELETE로 발생.
        - **방지 격리 수준**: Repeatable Read 이상.
    - **Phantom Read (유령 데이터 읽기)**: 동일 트랜잭션 내 동일 조건 재조회 시 새 행 출현·소멸. 다른 트랜잭션의 INSERT/DELETE로 발생.
        - **방지 격리 수준**: Serializable (또는 범위 잠금·MVCC 구현에 따라 Repeatable Read에서 부분 방지 가능).
    - **Lost Update (갱신 손실)**: 두 트랜잭션이 동일 데이터를 각자 읽고 갱신 시 하나의 갱신이 덮어써짐.
        - **방지 방법**: 비관적 잠금(SELECT FOR UPDATE), 낙관적 잠금(버전 번호·타임스탬프), Serializable 격리 수준.
- Locking (로킹, 잠금 제어)
    - 로킹 단위 (Granularity, 단위 크기) — DB > 테이블 > 페이지 > 레코드 > 필드
        - 단위 작을수록: 동시성 ↑, 로킹 오버헤드 ↑.
        - 단위 클수록: 동시성 ↓, 오버헤드 ↓.
    - 2PL (Two-Phase Locking, 2단계 잠금): 확장 단계(잠금 획득) + 축소 단계(잠금 해제). 직렬가능성 보장.
    - Strict 2PL (엄격 2단계 잠금): 독점(X-lock)을 트랜잭션 완료 시까지 유지. 연쇄 롤백(cascading rollback) 방지. 실무에서 널리 사용.
    - Rigorous 2PL (엄정 2단계 잠금): 공유(S-lock)·독점(X-lock) 모두 완료 시까지 유지. Strict 2PL보다 엄격.
    - Multiple Granularity Locking (다중 단위 로킹) — 의도 로크로 계층적 충돌 사전 감지
        - IS (Intention Shared, 의도 공유): 하위에 S-lock 의도.
        - IX (Intention eXclusive, 의도 독점): 하위에 X-lock 의도.
        - SIX (Shared + Intention eXclusive, 공유+의도 독점): 현재 S + 하위 X 의도.
        - 양립성: IX+IX 허용. IX+SIX, SIX+IX, SIX+SIX는 S와 하위 X 충돌로 대기.
    - Optimistic vs Pessimistic Locking (낙관적 vs 비관적 로킹)
        - Pessimistic (비관적): 즉시 잠금. 충돌 빈번 환경·특정 행 집중 접근에 적합.
        - Optimistic (낙관적): 작업 후 커밋 시점 충돌 검사. 웹·인터넷처럼 충돌 적은 환경에 효율적. 버전 번호·타임스탬프로 구현.
- 교착상태 (Deadlock, 데드락)
    - **발생 필요조건 4가지** — Coffman 조건: 4가지 **모두** 동시에 성립해야 교착상태 발생
        - **상호 배제 (Mutual Exclusion)**: 한 번에 하나의 트랜잭션만 자원(락) 점유 가능. 다른 트랜잭션은 대기.
        - **점유와 대기 (Hold and Wait)**: 이미 하나 이상의 자원을 점유한 채 다른 자원을 추가 요청하며 대기.
        - **비선점 (Non-preemption)**: 트랜잭션이 점유한 락을 강제로 빼앗을 수 없음. 스스로 해제해야만 반납.
        - **순환 대기 (Circular Wait)**: T1→T2→T3→T1처럼 트랜잭션 간 자원 요청이 원형(사이클)을 형성.
    - **해결 방법**
        - **예방 (Prevention)** — 4가지 조건 중 하나를 원천 차단
            - 점유와 대기 제거: 트랜잭션 시작 전 필요한 모든 락을 한 번에 획득(일괄 획득). 자원 낭비·기아(starvation) 발생 가능.
            - 비선점 제거: 대기 중인 트랜잭션의 락을 강제 해제. 롤백 발생.
            - 순환 대기 제거: 자원에 전역 순서(번호) 부여 후 오름차순으로만 잠금 요청(잠금 순서 규약).
        - **회피 (Avoidance)** — 타임스탬프 기반 선점 알고리즘 (교착상태 자체를 허용하지 않음)
            - **Wait-Die** (비선점 방식, Non-preemptive)
                - 기본 원칙: **오래된 트랜잭션(TS 낮음)이 인내**, 젊은 트랜잭션은 죽음
                - 요청 트랜잭션(Ti)이 보유 트랜잭션(Tj)보다 **오래됨(TS_i < TS_j)** → Ti **대기(Wait)**
                - 요청 트랜잭션(Ti)이 보유 트랜잭션(Tj)보다 **젊음(TS_i > TS_j)** → Ti **롤백(Die)**
                - 예시: T1(TS=1) vs T2(TS=5) — T1이 T2 락 요청 시 T1 대기(O), T2가 T1 락 요청 시 T2 롤백
                - 특징: **선점 없음**. 젊은 트랜잭션이 항상 희생.
            - **Wound-Wait** (선점 방식, Preemptive)
                - 기본 원칙: **오래된 트랜잭션(TS 낮음)이 선점**, 젊은 트랜잭션이 기다림
                - 요청 트랜잭션(Ti)이 보유 트랜잭션(Tj)보다 **오래됨(TS_i < TS_j)** → Tj **강제 롤백(Wound·상처)**
                - 요청 트랜잭션(Ti)이 보유 트랜잭션(Tj)보다 **젊음(TS_i > TS_j)** → Ti **대기(Wait)**
                - 예시: T1(TS=1) vs T2(TS=5) — T1이 T2 락 요청 시 T2 강제 롤백(O), T2가 T1 락 요청 시 T2 대기
                - 특징: **선점 있음**. 오래된 트랜잭션이 항상 관철.
            - **비교 요약**

                | 구분 | Wait-Die | Wound-Wait |
                |---|---|---|
                | 방식 | 비선점 | 선점 |
                | 오래된 Ti → 젊은 Tj 요청 | Ti **대기** | Tj **강제 롤백** |
                | 젊은 Ti → 오래된 Tj 요청 | Ti **롤백** | Ti **대기** |
                | 롤백 대상 | 젊은 트랜잭션(요청자) | 젊은 트랜잭션(보유자) |
                | 오버헤드 | 낮음(강제 선점 없음) | 높음(보유자 강제 롤백) |

            - **공통점**: ①재시작 시 원래 타임스탬프(TS) 유지 → 기아(Starvation) 방지. ②젊은 트랜잭션이 희생자. ③교착상태 자체가 발생하지 않음(회피, 탐지 불필요).
            - **시험 함정**: "Wait-Die에서 오래된 트랜잭션이 젊은 트랜잭션 락 요청 시" → **대기(Wait)**. "Wound-Wait에서 동일 상황" → 젊은 보유자 **강제 롤백(Wound)**. 두 방식 혼동 주의.

        - **탐지 및 복구 (Detection & Recovery)** — 교착상태 허용 후 사후 처리
            - 탐지: 대기 그래프(Wait-for Graph)에서 사이클 존재 여부 주기적으로 검사. 사이클 = 교착상태.
            - 복구: 사이클 내 트랜잭션 중 하나를 **희생자(Victim)** 선정하여 강제 롤백. 선정 기준: 진행 비용(undo 비용)·우선순위·점유 자원 수 등.
            - 타임아웃(Timeout): 일정 시간 이상 대기 트랜잭션을 자동 롤백. 단순하지만 교착 여부 불확실.
    - **시험 함정**: 4가지 조건 중 하나만 부정되어도 교착상태 발생 불가. "상호 배제를 허용하면 교착상태가 발생할 수 없다" → 오답(상호 배제는 제거 불가한 경우가 많음).

- 장애 회복 기법
    - Deferred Update (지연 갱신): 커밋 전까지 DB 미반영, 로그만 기록. 커밋 전 실패 시 UNDO 불필요, 커밋 후 장애 시 REDO 필요할 수 있음.
    - Immediate Update (즉시 갱신): 트랜잭션 중 DB에 즉시 반영. 커밋 전 실패 시 UNDO 필요, 커밋 후 장애 시 REDO 필요.
    - WAL (Write-Ahead Logging, 선기록 로그): 데이터 페이지 기록 전 로그를 먼저 디스크에 기록. 회복성 보장의 기본 원칙.
    - AFIM (After Image, 변경 후 값) → REDO 로그: 커밋된 트랜잭션 재수행에 사용.
    - BFIM (Before Image, 변경 전 값) → UNDO 로그: 미완료 트랜잭션 취소에 사용.
    - Checkpoint (검사시점, 체크포인트) — 회복 범위 축소를 위한 동기화 시점
        - 회복 순서: 변경 버퍼 디스크 출력 → 로그 스캔으로 UNDO-list/REDO-list 분류 → UNDO → REDO.
        - 체크포인트 이전 완료 트랜잭션(T1): 이미 디스크 반영, 회복 불필요.
        - 체크포인트 이후 완료 트랜잭션: REDO.
        - 장애 시점 미완료 트랜잭션: UNDO.
- DB 보안 (Access Control, 접근 제어)
    - DAC (Discretionary Access Control, 임의 접근 제어): 데이터 소유자가 권한 부여. GRANT/REVOKE SQL 명령으로 제어.
    - MAC (Mandatory Access Control, 강제 접근 제어, 다단계 보안): 객체에 비밀 등급·사용자에 허가 등급 부여. Bell-LaPadula 모델 기반.
    - 보안 등급 체계: TS (Top Secret) > S (Secret) > C (Confidential) > U (Unclassified).
    - Simple Security (읽기 규칙): 사용자 허가 등급 ≥ 객체 비밀 등급 (no read up).
    - *-property (쓰기 규칙, 스타 속성): 사용자 허가 등급 ≤ 객체 비밀 등급 (no write down). 상위 등급의 하향 기록 금지로 정보 유출 방지.

# 3.5.1 데이터 마이닝·정보검색
- Association Rule Mining (연관 규칙 마이닝)
    - Apriori Algorithm (아프리오리 알고리즘): 빈발 항목집합(frequent itemset)을 하위→상위 순서로 탐색하며 가지치기.
    - Anti-monotone Property (비-단조 성질, 반단조 성질): 어떤 항목집합이 비빈발이면 그 모든 상위집합도 비빈발. 빈발 항목집합의 모든 부분집합은 반드시 빈발.
    - Support (지지도): 항목집합 등장 거래 수 / 전체 거래 수. minsup 0.5 = 10건 중 5건 이상 등장.
    - Confidence (신뢰도, X→Y): support(X∪Y) / support(X). LHS를 줄이면 신뢰도 감소·동일. BCD→A가 미충족이면 BD→AC, D→ABC 등도 미충족(가지치기).
- Classification (분류 기법)
    - Naive Bayes (나이브 베이즈 분류기): 베이즈 정리 P(C|X) = P(X|C)P(C)/P(X). 특성 독립 가정. P(X) = Σ P(X|Ci)P(Ci) 전체 확률 법칙.
    - Conditional Probability (조건부 확률) 계산: P(X|C) = 클래스 C에서 X 발생 빈도 / 클래스 C 전체 빈도. 훈련 데이터의 클래스별 분포로 산출.
    - Decision Tree (의사결정 트리): Gini 불순도가 낮은 방향으로 분기 기준 선택.
    - Gini Impurity (지니 불순도, Gini Index): 1 − Σ(p_i)². 이진 분류 최대 0.5(50:50 분포), 최소 0(완전 순수). 절대 1 아님.
    - Classification Error (분류 오류): 1 − max(p_i). 이진 분류 최대 0.5.
- Clustering (군집화) & 유사도
    - k-Means Clustering (k-평균 군집화): 초기 중심점 임의 선택 → 할당 → 중심 갱신 반복. 구형 군집에 최적, 이상치에 민감, k 사전 지정 필요.
    - k-Means 거리 계산 횟수: n개 점 × k개 중심 × t회 반복. 100점·k=5·5회 반복이면 기본 2500회.
    - k-Medoids (k-메도이드): 평균 대신 실제 점을 중심으로 사용. 이상치에 강건.
    - DBSCAN (밀도 기반 군집): 비구형 군집과 이상치 탐지에 적합.
    - k-Means++ (초기 중심 개선 기법): 초기 중심점 의존성 완화.
    - Cosine Similarity (코사인 유사도): A·B / (|A|×|B|). 문서 벡터 유사도. 값 1에 가까울수록 유사.
- Information Retrieval (정보 검색)
    - TF-IDF (Term Frequency - Inverse Document Frequency, 용어 빈도-역문서 빈도): TF × IDF. IDF = log(|D| / df). 모든 문서 등장 용어(stop word)는 IDF=0 → TF-IDF=0.
    - Stemming (어간 추출): 단어의 접두사·접미사를 제거해 어근(stem) 추출. 규칙 기반, 언어학적 정확도 낮을 수 있음.
    - Lemmatization (표제어 추출): 사전적 기본형으로 변환. 스테밍보다 정확.
    - 텍스트 전처리 단계: 토큰화 → 불용어(stopword) 제거 → 스테밍/표제어 → 색인 생성.
    - Matching Technique (매칭 기법): 특징 벡터 추출 → 유사도 함수(거리·코사인) 계산 → 임계값 이상 반환. 멀티미디어 검색의 핵심.
- 성능 평가 척도
    - Confusion Matrix (혼동행렬): TP/FP/TN/FN 4분할 표.
    - Precision (정밀도): TP / (TP + FP). 양성 예측 중 실제 양성 비율.
    - Recall (재현율, Sensitivity, 민감도): TP / (TP + FN). 실제 양성 중 감지 비율.
    - Specificity (특이도): TN / (TN + FP). 실제 음성 중 음성 예측 비율.
    - Accuracy (정확도): (TP + TN) / 전체.
    - F-score (F-스코어, F1): 2 × Precision × Recall / (Precision + Recall). 조화 평균 → 두 값 중 작은 쪽에 가까워짐(더 큰 쪽이 아님).
    - Average Precision (평균 정확도): 랭킹 검색 결과에서 관련 문서가 나타날 때마다의 Precision을 평균.

# 3.5.2 NoSQL·CAP·분산DB·객체DB·품질
- NoSQL 데이터 모델 4분류
    - Key-Value Store (키-값 저장소): 단순 Key-Value 쌍 저장, 고속 검색. Redis, DynamoDB. 관계 표현·복잡 검색 부적합.
    - Document-based (문서 기반): JSON/BSON 형태 저장, 중첩·계층 구조 자유. MongoDB, CouchDB. 스키마 유연.
    - Column-family (컬럼 기반): 열 단위 저장, 대용량 분산 처리. Cassandra, HBase. 클러스터 환경 적합.
    - Graph-based (그래프 기반): 노드(엔티티)+엣지(관계). ACID 지원, 클러스터 부적합, 단일 서버 최적. 소셜 네트워크 친구 탐색·추천·지식 그래프. Neo4j, Neptune.
- CAP Theorem (CAP 정리, CAP 이론)
    - Consistency (일관성): 모든 노드가 동일 시점에 동일 데이터 보유.
    - Availability (가용성): 모든 요청이 응답을 받음.
    - Partition Tolerance (분할 감내): 네트워크 분할 상황에서도 시스템 동작.
    - 핵심 원칙: 세 성질 중 최대 두 가지만 동시 보장 가능. NoSQL은 CA/CP/AP 중 둘 선택.
    - Eventual Consistency (궁극적 일관성): AP 시스템이 채택하는 약한 일관성. 일정 시간 후 수렴.
    - BASE (Basically Available, Soft state, Eventually consistent): NoSQL의 트랜잭션 모델. ACID와 대비.
- NoSQL 특성·RDB 비교
    - Horizontal Scalability (수평 확장성, Scale-out): 노드 추가로 저장 공간·처리 능력 증가. NoSQL 핵심 장점.
    - Fault Tolerance (고장 허용성): 복제(replication)로 특정 노드 장애 시에도 서비스 지속.
    - Sharding (샤딩): 데이터를 여러 노드에 분산 저장하여 부하 분산.
    - 복제와 쓰기 성능: 복제는 읽기 성능 향상에 기여하나 쓰기는 모든 복제본 동기화 필요로 성능 향상 제한.
    - RDB vs NoSQL: 엄격한 스키마 vs 유연한 스키마, 수직 확장 vs 수평 확장, SQL 복잡 질의 vs 단순 검색, ACID 트랜잭션 vs eventual consistency.
- Distributed DB Transparency (분산 데이터베이스 투명성)
    - Location Transparency (위치 투명성): 데이터 물리적 위치 모르고 논리적 이름으로 접근.
    - Naming Transparency (이름부여 투명성): 단일 전역 네임스페이스, 사이트 무관 동일 이름 접근.
    - Replication Transparency (중복 투명성): 데이터 복제본 존재를 사용자에게 숨김.
    - Fragmentation Transparency (단편화 투명성): 릴레이션이 여러 단편으로 나뉘어도 단일 릴레이션처럼 접근.
    - Failure Transparency (장애 투명성): 일부 노드·링크 장애에도 시스템 동작.
    - Concurrency Transparency (동시성 투명성): 다중 트랜잭션 동시 수행 시 일관성 유지. 장애 투명성과 혼동 주의.
- ORDB/OODB (객체 데이터베이스)
    - OID (Object Identifier, 객체 식별자): 시스템 자동 부여, 불변, 속성값 무관, 객체 제거 시에만 소멸·재사용 불가.
    - OID vs 기본키: OID는 포인터/참조 기반 식별, 기본키는 값 기반 식별. 본질적으로 다름.
- 데이터 품질 (ISO 8000)
    - 유효성 (Validity): 사실성(Accuracy)·필수성(Completeness)·정합성(Consistency). 데이터 자체의 정확성 검증.
    - 활용성 (Portability): 충분성(Sufficiency). 목적 달성에 충분한 데이터.
    - 적합성 (Conformity): 표준 코드값·표준 도메인값·정해진 형식 준수 여부.
- 모바일 DB 캐싱
    - Invalidation Report (무효 보고서): 서버가 연결된 모바일 호스트에 변경된 캐시 항목 브로드캐스트.
    - Read-only Cache (판독 전용 캐시): 변경 없으므로 일관성 유지 보장.
    - 단절 상태 처리: 모바일에서 분할(단절)은 정상 운영 일부. 재연결 시 전체 캐시 무효화는 상당한 비용 수반.

# 3.5.3 빅데이터 인프라·데이터 포맷·DW
- HDFS (Hadoop Distributed File System, 하둡 분산 파일 시스템)
    - NameNode (네임노드): 메타데이터(파일 경로·블록 위치) 관리. 네임스페이스·블록 매핑을 메모리 유지.
    - DataNode (데이터노드): 실제 데이터 블록 저장·제공. 주기적 Heartbeat·블록 리포트 전송.
    - Secondary NameNode: NameNode 백업 노드.
    - 기본 복제 인수 3 (3-way replication): 블록을 3개 DataNode에 분산.
    - 읽기 동작: NameNode에서 블록 위치 조회 → 가까운 DataNode에서 직접 읽기(Rack-aware). 빠름.
    - 쓰기 동작: 복제 파이프라인(DN1→DN2→DN3) ACK 대기. 읽기보다 느림.
- Apache Spark
    - RDD (Resilient Distributed Dataset, 탄력적 분산 데이터셋): Spark 핵심 데이터 추상화. 관계형 DB의 릴레이션에 대응.
    - RDD 특성: 불변성(immutable), 지연 평가(lazy evaluation), Lineage 기반 장애 복구(fault tolerance).
    - Transformation vs Action: Transformation은 새 RDD 생성·지연 실행, Action은 결과 반환·즉시 실행.
    - HDFS 연동·반복 계산 적합: 메모리 캐싱으로 MapReduce 대비 반복 작업 고속화.
- 데이터 포맷
    - JSON (JavaScript Object Notation, 자바스크립트 객체 표기법): 속성-값 쌍, 트리/중첩 구조 지원. REST API 등 네트워크 통신에 주로 사용.
    - XML (eXtensible Markup Language, 확장 마크업 언어): 태그 기반 계층적 구조. 문서 저장·교환.
    - XML DTD (Document Type Definition, 문서 타입 정의): 요소 구조 정의. (a,b,c)는 순서 필수, (b|c)는 선택, *은 0회 이상, ?는 0 또는 1회, +는 1회 이상.
    - CSV (Comma Separated Values, 쉼표 구분값): 행·열의 평면(flat) 구조. 중첩·계층 표현 불가. 정형 데이터·스프레드시트 호환.
    - Key-Value Store (키-값 저장소): 단순 Key-Value 쌍. NoSQL에서 고속 검색 용도.
- Data Warehouse (데이터 웨어하우스, DW)
    - Inmon의 4대 특성: 주제 지향성(Subject-oriented)·통합성(Integrated)·시간 가변성(Time-variant)·비휘발성(Nonvolatile).
    - Subject-oriented (주제 지향성): 업무 처리 중심의 운영 DB와 달리 의사결정 주제(고객·제품)별 구성.
    - Integrated (통합성): 이기종 소스에서 ETL (Extract·Transform·Load)로 통합.
    - Time-variant (시간 가변성): 과거~현재 이력 데이터를 시계열로 보존. Time-invariant가 아님에 주의.
    - Nonvolatile (비휘발성): 적재 후 삭제·갱신 없이 읽기 전용 유지.
    - OLAP (Online Analytical Processing, 온라인 분석 처리): DW 기반 다차원 분석·의사결정 지원.
