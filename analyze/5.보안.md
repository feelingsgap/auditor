# 5.1 정보보호 이론

- 정보보호 3대 목표 (CIA Triad)
    - Confidentiality (기밀성): 인가된 주체만 정보에 접근 가능하도록 보장하는 속성.
    - Integrity (무결성): 정보가 허가되지 않은 방법으로 변경되지 않음을 보장하는 속성.
    - Availability (가용성): 인가된 사용자가 필요할 때 정보·서비스를 사용할 수 있도록 보장하는 속성.
- 접근통제 (Access Control)
    - 접근통제 모델
        - DAC (Discretionary Access Control, 임의적 접근통제): 데이터 소유자가 재량으로 다른 사용자에게 접근 권한을 부여하는 신원 기반 모델.
        - MAC (Mandatory Access Control, 강제적 접근통제): 시스템이 정한 보안 등급(레이블) 기반으로 접근을 강제 통제하는 모델. 신원이 아닌 등급이 기준.
        - RBAC (Role-Based Access Control, 역할 기반 접근통제): 사용자에게 역할을 부여하고 역할에 권한을 할당하여 대규모 환경의 관리 효율을 높이는 모델.
    - ACL (Access Control List, 접근제어 목록): 객체에 접근 가능한 주체와 권한을 나열한 자료구조. 주체 수 증가 시 관리 복잡도 상승.
    - BLP·Biba 모델: 대표적 보안 모델은 5.3 시스템보안 참조.
- 사용자 인증 (Authentication)
    - 인증 요소
        - 지식 기반 (Something you know): 패스워드·PIN. 저비용이나 추측·사전 공격에 취약.
        - 소유 기반 (Something you have): OTP 토큰·스마트카드. 분실·복제 위험 존재.
        - 생체 기반 (Something you are): 지문·홍채·망막. 사전에 생체 템플릿 등록(Enrollment) 필수.
        - 2FA (Two-Factor Authentication, 이중 인증): 서로 다른 유형의 인증 요소 2개 결합. MFA는 2개 이상 결합.
    - OTP (One-Time Password, 일회용 패스워드) 생성 방식
        - S/KEY 방식: 사전에 생성한 해시 체인을 역순으로 사용하는 OTP 방식.
        - 시간 동기화 방식 (TOTP, Time-based OTP): 서버·클라이언트가 동기화된 시간을 기반으로 동일 OTP 생성.
        - 이벤트 동기화 방식 (HOTP, HMAC-based OTP): 카운터 값을 증가시키며 OTP 생성.
        - 도전-응답 방식 (Challenge-Response): 서버가 난수(Challenge)를 전송, 클라이언트가 이를 이용해 응답(Response) OTP 생성.
- 대칭키 암호 (Symmetric Key Cryptography)
    - 블록 암호 알고리즘
        - DES (Data Encryption Standard, 데이터 암호화 표준): Feistel 구조, 블록 64비트, 키 56비트. 현재는 취약.
        - AES (Advanced Encryption Standard, 고급 암호화 표준): NIST 표준 SPN 구조 블록 암호. 블록 128비트 고정, 키 128/192/256비트 (AES-128/192/256). 라운드 수 10/12/14.
        - AES 라운드 함수 — 각 라운드의 4개 서브 연산
            - SubBytes: 각 바이트를 GF(2^8) 기반 S-Box로 독립 치환하는 비선형 연산.
            - ShiftRows: 상태 행렬의 각 행을 서로 다른 오프셋만큼 순환 이동.
            - MixColumns: 각 열에 대해 GF(2^8) 다항식 곱셈으로 열 혼합 (마지막 라운드 제외).
            - AddRoundKey: 라운드 키와 상태 행렬을 비트 단위 XOR.
        - SEED / ARIA: 한국 표준 대칭키 블록 암호 (각각 KISA·국정원 개발).
    - 블록 암호 운용 모드 (Block Cipher Mode of Operation)
        - ECB (Electronic Code Book, 전자부호책 모드): 블록을 독립 암호화. 동일 평문 → 동일 암호문 (패턴 노출). IV 없음.
        - CBC (Cipher Block Chaining, 암호 블록 연쇄 모드): 이전 암호문을 XOR 후 암호화. IV 사용, 암호화 순차(병렬 불가), 복호화는 병렬 가능, 오류 전파 2블록 한정(현재+다음 블록).
        - CFB (Cipher Feedback, 암호 피드백 모드): 이전 암호문을 암호화하여 키스트림 생성 후 평문과 XOR → 스트림 암호처럼 동작. IV 사용. 암호화 순차(병렬 불가), 복호화는 병렬 가능. 오류 전파: 오류 발생 블록과 다음 블록까지(CBC와 유사).
        - OFB (Output Feedback, 출력 피드백 모드): 암호화 함수 출력(키스트림)을 피드백하여 키스트림만 생성 후 평문과 XOR → 스트림 암호처럼 동작. IV 사용. 암호화·복호화 모두 병렬 불가(키스트림 순차 의존). **비트 오류가 해당 비트에만 영향(오류 전파 없음)** — 잡음 많은 채널에 적합.
        - CTR (Counter, 카운터 모드): 카운터 값을 암호화한 키스트림을 평문과 XOR. 암호화·복호화 모두 병렬 처리 가능. 임의 블록 접근(random access) 가능. 오류 전파 없음(해당 블록만 영향).
        - GCM (Galois/Counter Mode, 갈루아/카운터 모드): CTR 암호화 + GHASH (갈루아 필드 해시) 기반 인증 태그. AEAD 모드로 기밀성·무결성 동시 제공, 병렬 처리 가능. TLS 1.2/1.3 권장.
        - **병렬 처리 요약**: 암호화·복호화 모두 가능 → CTR·GCM. 복호화만 가능 → CBC·CFB. 모두 불가 → OFB.

- 공개키 암호 (Asymmetric / Public-Key Cryptography)
    - 수학적 난제 기반 알고리즘
        - RSA (Rivest-Shamir-Adleman, 리베스트·샤미르·애들먼 암호): 큰 수의 소인수분해 난이도 기반. 암호화 C = M^E mod N (공개키 E,N), 복호화 M = C^D mod N (개인키 D,N). 서명·키 교환에도 활용.
        - ECC (Elliptic Curve Cryptography, 타원곡선 암호): 타원곡선 이산대수 문제(ECDLP) 기반. 짧은 키(256비트)로 RSA 3072비트와 동등 강도. 모바일·IoT에 적합. ECDH (키 교환)·ECDSA (서명) 응용.
        - ElGamal (엘가말 암호): 이산대수 문제 기반 공개키 암호. 암호문이 (C1, C2) 쌍이므로 평문 대비 2배 크기(메시지 확장) 단점.
        - Diffie-Hellman Key Exchange (디피-헬만 키 교환): 이산대수 난이도 기반 공개키 교환 프로토콜. 인증 기능 없음(MITM 취약).
    - 공개키 암호 특성: 암복호화에 서로 다른 키 사용. 대칭키 대비 계산 속도 느림 → 키 교환·서명에 주로 활용, 대용량 데이터 암호화는 대칭키 담당.
- 해시·메시지 인증 (Hash & Message Authentication)
    - 암호학적 해시함수 보안 요건
        - Preimage Resistance (역상 저항성): 해시값 h로부터 h=H(m)을 만족하는 m을 찾기 어려움 (일방향성).
        - Second Preimage Resistance (제2역상 저항성): 주어진 m1에 대해 H(m1)=H(m2)인 다른 m2를 찾기 어려움.
        - Collision Resistance (충돌 저항성): H(m1)=H(m2)를 만족하는 임의의 (m1, m2) 쌍을 찾기 어려움.
    - 대표 해시 알고리즘
        - SHA-2 (Secure Hash Algorithm 2): Merkle-Damgard 구조, SHA-224/256/384/512. SHA-256은 256비트 출력.
        - SHA-3 (Secure Hash Algorithm 3, Keccak): 스펀지 구조(Sponge Construction). SHA-1/SHA-2와 설계가 완전히 다름. SHA3-224/256/384/512 및 가변 길이 SHAKE128/256. 임의 길이 입력 가능, 의사난수 생성기·스트림 암호로도 활용.
    - 해시함수 용도: 무결성 검증·전자서명·패스워드 저장·MAC. 기밀성은 제공 불가(단방향이므로 복원 불가지만 암호화가 아님).
    - MAC (Message Authentication Code, 메시지 인증 코드): 공유 대칭키로 생성한 인증 태그. 무결성·송신자 인증 제공, 부인 방지 불가(양당사자 모두 생성 가능).
    - HMAC (Hash-based MAC, 해시 기반 메시지 인증 코드): 비밀키와 해시함수를 결합한 MAC (예: HMAC-SHA256). 부인 방지 미제공.
    - CBC-MAC / CMAC (Cipher-based MAC): 블록 암호의 CBC 모드를 이용한 MAC 구현.
- 전자서명·키 관리
    - Digital Signature (전자서명): 개인키로 메시지 해시를 서명 → 공개키로 검증. 제공 서비스는 인증(Authentication)·무결성(Integrity)·부인 방지(Non-repudiation). 기밀성은 제공하지 않음. 서명은 메시지 해시에 종속되어 다른 문서에 재사용 불가.
    - Hybrid Cryptosystem (하이브리드 암호 시스템): 대칭 세션키로 메시지 암호화 + 수신자 공개키로 세션키 암호화 전달. 속도와 키 배포 장점 결합. PGP·TLS가 대표 적용 사례.
    - PBE (Password-Based Encryption, 패스워드 기반 암호화): 패스워드 + 솔트 → KDF → KEK (Key Encryption Key) 생성 → CEK (Contents Encryption Key) 암호화. 솔트는 레인보우 테이블·사전 공격 방어용 (MITM 방어 아님).
    - KEK (Key Encryption Key, 키 암호화 키): 다른 키(CEK)를 암호화할 때 사용하는 상위 키.
    - CEK (Contents Encryption Key, 콘텐츠 암호화 키): 실제 메시지·파일을 암호화하는 키.
    - Salt (솔트): 동일 패스워드라도 다른 해시·KEK를 생성하도록 더하는 난수. 사전 공격·레인보우 테이블 방어.
- 부채널 공격 (Side Channel Attack)
    - 공격 유형 — 암호 알고리즘 자체가 아닌 구현·하드웨어 부가정보 이용
        - Power Analysis Attack (전력 분석 공격): 암호 연산 중 전력 소비 패턴을 분석하여 키 추출.
        - Timing Attack (타이밍 공격): 연산 수행 시간 차이를 측정하여 키 추론.
        - Electromagnetic Attack (전자기 분석 공격): 장비에서 방사되는 전자파를 분석하여 키 추출.
    - 비부채널 대조 — CCA (Chosen Ciphertext Attack, 선택 암호문 공격): 임의 암호문에 대한 복호화 결과를 이용하는 수학적 공격으로 부채널에 해당하지 않음.
    - 대응책: 일정 시간 실행(constant-time), 전력 소비 균일화, 전자기 차폐, 연산 순서 랜덤화.

# 5.2 네트워크보안

- 보안 프로토콜 — 전송·메일·원격 접속
    - SSL/TLS 핸드셰이크
        - SSL (Secure Sockets Layer): Netscape가 개발한 전송 계층 보안 프로토콜의 초기 이름. 현재는 TLS로 대체.
        - TLS (Transport Layer Security, 전송 계층 보안): SSL의 후속 표준 프로토콜. 서버·클라이언트 인증, 키 교환, 세션 암호화로 종단 간 통신 보호.
        - 핸드셰이크 단계: ClientHello (암호 스위트·난수 제안) → ServerHello (선택·난수 응답) → Certificate (서버 인증서) → Key Exchange → ChangeCipherSpec → Finished.
        - Cipher Suite (암호 스위트): 클라이언트가 ClientHello로 지원 가능한 암호·해시 알고리즘 목록 전송.
        - Record Protocol (레코드 프로토콜): 핸드셰이크 이후 데이터 전송 단계에서 애플리케이션 데이터에 대해 MAC·암호화·압축 처리. MAC 계산은 핸드셰이크가 아닌 Record 단계 업무.
    - HTTPS over HTTP Proxy — HTTP 프록시 환경에서 HTTPS 전달
        - HTTPS (HTTP over TLS): HTTP를 TLS 위에서 전송하는 보안 웹 프로토콜.
        - CONNECT 메서드: 브라우저가 프록시에게 TCP 터널 설정을 요청하는 HTTP 메서드. HTTPS 사용 시 GET/POST 대신 CONNECT 호출.
        - 프록시 역할: CONNECT 터널 수립 후 브라우저-목적지 서버 간 단일 SSL/TLS 핸드셰이크를 TCP 레벨에서 투명 중계(암호화 해독 없음).
    - 메일 보안
        - PGP (Pretty Good Privacy, 프리티 굿 프라이버시): 대칭키·공개키 결합 하이브리드 전자우편 보안. 기밀성·무결성·인증·부인방지 제공.
        - S/MIME (Secure/Multipurpose Internet Mail Extensions, 보안 다목적 인터넷 메일 확장): MIME 기반 전자우편 보안 표준. 기업 환경에서 인증서 기반 메일 암호화·서명.
    - 무선 LAN 보안
        - WPA2 (Wi-Fi Protected Access 2): CCMP/AES 기반 무선 보안 표준. Personal (PSK)은 모든 단말이 사전 공유 키 사용.
        - CCMP (Counter Mode with CBC-MAC Protocol): AES 기반 기밀성·무결성 제공 무선 보안 프로토콜.
        - WPA-EAP (WPA-Enterprise, 기업용 WPA): 802.1X + RADIUS 인증으로 사용자별 세션마다 고유 동적 키 생성.
        - 802.1X: 포트 기반 네트워크 접근 제어 표준. EAP 프로토콜 전달.
        - RADIUS (Remote Authentication Dial-In User Service): 중앙집중식 AAA 서버로 사용자 인증·권한부여·계정관리 담당.
- VPN 터널링 프로토콜 (Virtual Private Network)
    - Layer 2 VPN
        - L2F (Layer 2 Forwarding, 계층 2 포워딩): Cisco가 개발한 초기 데이터링크 계층 터널링 프로토콜.
        - PPTP (Point-to-Point Tunneling Protocol, 지점 간 터널링 프로토콜): Microsoft 개발. PPP 프레임을 GRE로 캡슐화하는 Layer 2 터널링. MS-CHAPv2 인증, RC4 암호화 (현재 취약).
        - L2TP (Layer 2 Tunneling Protocol, 계층 2 터널링 프로토콜): PPTP와 L2F의 장점을 결합한 IETF 표준. 자체 암호화 없음 → IPSec과 결합(L2TP/IPSec, UDP 1701). Remote Dial-Up 사용자의 사설망 연결에 활용.
    - Layer 3 VPN
        - IPSec (Internet Protocol Security, 인터넷 프로토콜 보안): IP 네트워크 계층 보안 표준. 인증·암호화·키 관리 제공.
        - AH (Authentication Header, 인증 헤더): 무결성·인증 제공, 암호화는 없음.
        - ESP (Encapsulating Security Payload, 보안 페이로드 캡슐화): 기밀성·무결성·인증 제공.
        - Tunnel Mode (터널 모드): 원본 IP 패킷 전체(헤더+페이로드)를 암호화하고 새 IP 헤더로 캡슐화. 게이트웨이 간 VPN에 사용.
        - Transport Mode (전송 모드): IP 페이로드만 암호화, 원본 헤더 유지. 호스트 간 통신에 사용.
    - Layer 4+ VPN
        - SSL/TLS VPN: 애플리케이션 계층 보호. 웹 브라우저만으로 접속 가능, 방화벽 친화적.
        - SSTP (Secure Socket Tunneling Protocol, 보안 소켓 터널링 프로토콜): Microsoft 개발. SSL/TLS (포트 443) 기반으로 방화벽·NAT 우회 용이.
        - OpenVPN: SSL/TLS 기반 오픈소스 VPN.
    - 활용 구도 — Site-to-Site VPN (본사·지사 연결, 주로 IPSec) / Remote Access VPN (재택·외부 근무자, 주로 SSL VPN).
- 방화벽·IDS/IPS
    - Firewall (방화벽) 패킷 필터링: ACL 규칙 테이블(From/To/Service/Action)로 출발지·목적지·포트·프로토콜 기반 Allow/Deny 결정. 포트 예: SMTP 25, POP3 110, IMAP 143, HTTP 80, HTTPS 443, FTP 21, DNS 53.
    - IDS (Intrusion Detection System, 침입 탐지 시스템): 공격 탐지 후 경고. 능동 차단 없음.
    - IPS (Intrusion Prevention System, 침입 방지 시스템): 탐지와 동시에 트래픽 차단.
    - Snort 규칙 문법 — [action] [proto] [srcIP] [srcPort] -> [dstIP] [dstPort] (옵션)
        - Action: alert(탐지·알림), drop(차단+로그), log, pass.
        - Proto: tcp/udp/icmp/ip.
        - !CIDR: 해당 대역 제외(부정). 화살표(->)는 단방향, <>는 양방향.
        - Option: msg:"...", sid, content 등.
- DDoS 방어·IP 필터링
    - Ingress Filtering (인그레스 필터링): 자신의 AS로 들어오는 패킷의 출발지 IP가 할당 대역인지 검사하여 위조 IP 차단.
    - Egress Filtering (이그레스 필터링): 자신의 AS 밖으로 나가는 패킷의 출발지 IP가 내부 대역인지 검사하여 스푸핑 방지.
    - uRPF (Unicast Reverse Path Forwarding, 유니캐스트 역경로 전달): 라우터가 수신 패킷의 출발지 IP로 역경로 조회, 비정상 경로 패킷 폐기.
    - RTBH (Remotely Triggered Black Hole, 원격 트리거 블랙홀 라우팅): BGP 커뮤니티로 업스트림 ISP에 트리거 전달 → 특정 IP로 향하는 트래픽을 Null0 인터페이스로 라우팅하여 폐기. 공격·정상 트래픽 함께 희생되나 대역폭 절감 가능.
    - Source-based RTBH (SRTBH, 출발지 기반 RTBH): 출발지 IP 기반 차단 변형.
- 네트워크 접근 제어·분리
    - NAC (Network Access Control, 네트워크 접근 제어): 내부망 접속 단말의 MAC 주소·사용자 인증·보안정책(백신·패치 상태) 검사 후 격리·차단. 에이전트 기반 또는 에이전트리스.
    - VLAN (Virtual LAN, 가상 랜): 물리 스위치를 논리적으로 분할해 브로드캐스트 도메인 분리. ARP Request·NetBIOS 브로드캐스트가 다른 VLAN으로 전파되지 않음. IEEE 802.1Q 태깅 표준, VLAN 간 통신은 L3 라우터/스위치 필요.
    - ARP Poisoning 방어: 정적 ARP 테이블, DAI (Dynamic ARP Inspection, 동적 ARP 검사)로 DHCP Snooping 바인딩 테이블 기반 비정상 ARP 차단, 외부 PC 로컬 접근 제한. MAC 주소 주기적 변경은 방어책 아님.
- 통합 인증 (Single Sign-On)
    - SSO (Single Sign On, 통합 인증): 한 번의 인증으로 다수 시스템에 접근. 중앙 SSO 서버가 토큰을 발급하여 연계 시스템이 검증.
    - 구현 표준 — Kerberos (MIT 개발 티켓 기반 네트워크 인증, Windows Active Directory 핵심) / SAML (Security Assertion Markup Language) / OAuth 2.0 / OpenID Connect.
    - Step-up Authentication (스텝업 인증): 민감 자원 접근 시 추가 인증 요구하는 조건부 강화.
    - 단점: SSO 서버가 Single Point of Failure (단일 장애점)로 작동, 침해 시 전체 시스템 노출.

# 5.3 시스템보안

- 보안 모델 (Security Models)
    - BLP (Bell-LaPadula Model, 벨-라파둘라 모델): 기밀성 중심 강제접근통제 모델. 군사 기밀등급 체계 기반.
        - Simple Security Rule (No Read Up, 단순 보안 규칙): 주체는 자신의 등급보다 높은 등급 객체를 읽을 수 없음.
        - Star Security Rule (No Write Down, ★-보안 규칙): 주체는 자신의 등급보다 낮은 등급 객체에 쓸 수 없음 → 기밀 유출 방지.
        - Strong Star Property (강한 ★-보안 규칙): 주체는 동일 등급의 객체에만 읽기·쓰기 허용.
    - Biba Model (비바 모델): 무결성 중심 모델. BLP의 무결성 버전.
        - No Read Down (단순 무결성 규칙): 주체는 자신보다 낮은 무결성 등급 객체를 읽지 못함 (오염 방지).
        - No Write Up (무결성 ★-규칙): 주체는 자신보다 높은 무결성 등급 객체에 쓰지 못함.
- 데이터 유출·권한 관리 솔루션
    - DLP (Data Leakage Prevention / Data Loss Prevention, 데이터 유출 방지): 내부 민감 정보의 외부 유출을 내용 기반으로 탐지·차단. 이메일·웹·USB·블루투스·테더링 등 다양한 채널 모니터링. Network DLP / Endpoint DLP / Storage DLP로 분류.
    - DRM (Digital Rights Management, 디지털 저작권 관리): 콘텐츠 암호화·접근 권한·사용 권한(열람·편집·인쇄·캡처) 집행. 사용자별·그룹별·프로세스별 세밀한 권한 제어 가능.
    - E-DRM (Enterprise DRM, 기업용 DRM): 기업 내 문서의 생성·유통·폐기 전 단계 보호. IRM (Information Rights Management)과 동의어.
    - DRM 3요소: 콘텐츠 암호화, 라이선스 관리, 사용 권한 집행(Enforcement).
    - Digital Watermarking (디지털 워터마킹): 콘텐츠에 식별 정보 삽입. 유출 차단이 아닌 사후 추적 목적.
- 데이터베이스 보안
    - DB 보안 요구사항: 논리적 일관성, 기밀성, 접근통제, 트랜잭션 병행처리 무결성.
    - Inference Attack (추론 공격): 집계·통계 데이터에서 개별 레코드를 역추적하는 공격. 통계 DB 보안의 주요 방어 대상.
    - DB 암호화 방식
        - Plug-In 방식: DB 서버·엔진에 암복호화 모듈 설치. 애플리케이션 수정 최소화(투명 암호화). DB 성능에 부하.
        - API 방식: 애플리케이션에서 암복호화 라이브러리 호출. 앱 코드 수정 필요, DB 서버 부하 최소.
        - Hybrid (하이브리드) 방식: Plug-In·API 절충. 대용량 트랜잭션은 Plug-In, 성능 영향 민감 부분은 API.
        - TDE (Transparent Data Encryption, 투명 데이터 암호화): DB 파일·스토리지 레벨 암호화. Oracle·SQL Server 등 지원.
- IDS 탐지 결과 분류
    - True-Positive (TP, 진양성): 실제 공격을 공격으로 정확히 탐지.
    - True-Negative (TN, 진음성): 정상 트래픽을 정상으로 정확히 판단.
    - False-Positive (FP, 위양성/오탐): 정상 트래픽을 공격으로 오인하여 차단 → 가용성 침해.
    - False-Negative (FN, 위음성/미탐): 실제 공격을 정상으로 판단하여 통과 → 기밀성·무결성 침해.
- Linux OS 보안 설정
    - iptables 기본 구조 — iptables -[A|I] [체인] [조건] -j [액션]
        - 체인: INPUT(유입), OUTPUT(유출), FORWARD(경유).
        - Action: DROP(응답 없이 폐기), REJECT(거절 응답 전송), ACCEPT, LOG, RETURN.
        - -s (source): 출발지 IP. -d (destination): 목적지 IP. -p (protocol): 프로토콜.
        - 예시 — `iptables -A INPUT -p icmp --icmp-type echo-request -j DROP` (Ping of Death 방어: ICMP echo-request 차단).
        - 예시 — `iptables -A INPUT -s 192.168.20.22 -j DROP` (특정 IP 차단).
    - root 원격 로그인 차단 설정
        - /etc/securetty: root가 로그인 가능한 tty 목록. pts(원격) 항목 제거 → 원격 root 로그인 차단.
        - /etc/pam.d/login: PAM 기반 로그인 인증 설정. 콘솔 로그인 제한에 관여.
        - /etc/ssh/sshd_config: `PermitRootLogin no` 설정으로 SSH 원격 root 금지.
        - /etc/passwd: root 셸을 /sbin/nologin으로 변경 시 모든 로그인 차단(원격 포함).
- 모바일 OS 보안
    - iOS 보안 특성: BSD Unix 기반, 샌드박스(Sandbox)로 앱 격리, 원격 쉘·SSH 앱 App Store 배제, 코드 서명은 Apple Developer Certificate 필수.
    - Android 보안 특성: 샌드박스 기반이나 Intent를 통한 앱 간 통신·데이터 전달이 iOS 대비 유연, 모든 앱에 코드 서명 요구하나 개발자 자체 서명(self-signed) 허용 (CA 인증 불필요), 앱은 기본적으로 일반 사용자 권한으로 실행, 권한은 매니페스트에 명시.

# 5.4.1 시큐어 코딩·웹 취약점

- 웹 입력 검증 미흡 취약점
    - Injection 계열
        - SQL Injection (SQL 삽입 공격): 사용자 입력이 SQL 질의에 병합되어 의도치 않은 질의가 실행되는 취약점. DB 데이터 탈취·변조·인증 우회 가능.
        - Command Injection (명령어 삽입): 입력값에 OS 명령어를 삽입하여 서버에서 실행시키는 공격.
        - XPath Injection (XPath 삽입): XML 쿼리(XPath) 구문에 악성 코드 삽입하여 데이터 탈취.
        - Format String Injection (포맷 스트링 삽입): printf() 등에 사용자 입력이 포맷 문자열로 직접 전달될 때 %x·%s·%n으로 메모리 읽기·쓰기 가능. 수정: printf(name) → printf("%s", name).
        - SQL Injection 방어 우선순위 — Parameterized Query (Prepared Statement, 파라미터화 쿼리) > Stored Procedure (저장 프로시저) > 화이트리스트 입력 검증 > 최소 권한 DB 계정.
    - XSS (Cross-Site Scripting, 교차 사이트 스크립팅) — 공격자가 삽입한 악성 스크립트가 피해자 브라우저에서 실행되는 웹 취약점
        - Stored XSS (저장형 XSS / Persistent XSS): 악성 스크립트가 DB·게시판 등에 저장되어 해당 페이지를 조회하는 모든 사용자에게 실행됨.
        - Reflected XSS (반사형 XSS): URL 파라미터 등으로 전달된 스크립트가 서버 응답에 그대로 반사되어 즉시 실행. 피싱·쿠키 탈취 유도에 활용.
        - DOM-based XSS (DOM 기반 XSS): 서버 응답과 무관하게 클라이언트 스크립트가 DOM을 조작하여 실행.
        - 대응: 출력 인코딩(HTML Entity Encoding), CSP (Content Security Policy, 콘텐츠 보안 정책), HttpOnly 쿠키.
    - CSRF (Cross-Site Request Forgery, 교차 사이트 요청 위조): 인증된 사용자의 세션을 악용하여 의도하지 않은 요청(계좌이체·정보변경)을 서버에 전송하는 공격. 스크립트 실행이 아닌 위조 요청 전송.
    - Unvalidated Redirects and Forwards (신뢰되지 않은 URL 자동 접속 연결): 입력 URL을 검증 없이 리다이렉트에 사용하여 피싱 사이트로 유도하는 취약점.
- 메모리·코드 취약점
    - Buffer Overflow (버퍼 오버플로우) — 입력 데이터 크기 검증 없이 버퍼 경계를 초과하여 인접 메모리를 덮어쓰는 취약점.
        - Stack Overflow (스택 오버플로우): 스택 지역변수 영역 초과. RET 주소·EBP 덮어써 셸코드 실행.
        - Heap Overflow (힙 오버플로우): malloc 등으로 할당된 힙 영역 초과.
        - 안전 코드: scanf("%99s", buf), fgets() 등 크기 제한 함수 사용.
    - CPU 레지스터 (스택 프레임 관련)
        - ESP (Extended Stack Pointer, 확장 스택 포인터): 스택 최상단(Top)을 가리키는 레지스터. 스택 증감에 따라 동적으로 변함.
        - EBP (Extended Base Pointer, 확장 베이스 포인터): 현재 스택 프레임의 베이스 주소. 지역변수는 EBP-n, 함수 인자는 EBP+n으로 참조. 스택 오버플로우 공격의 주요 탈취 대상.
        - Return Address (RET): 함수 종료 시 복귀할 주소. 오버플로우로 덮어쓰면 실행 흐름 조작 가능.
    - Race Condition (경쟁 조건): 두 개 이상의 프로세스·스레드가 공유 자원에 동시 접근하여 발생하는 타이밍 취약점. 입력 검증이 아닌 동시성 제어 문제. TOCTOU (Time-of-Check to Time-of-Use, 점검-사용 시점 차이) 취약점이 대표적.
    - Path Traversal (경로 탐색 / Directory Traversal): "../" 등 상대 경로로 웹 루트 밖 파일에 접근.
    - Null Pointer Dereference (널 포인터 역참조): NULL 상태의 포인터를 참조하여 비정상 종료·DoS 유발.
    - Hardcoded Password (하드코딩 비밀번호): 소스코드에 암호를 고정 삽입하여 노출.
    - Integer Overflow (정수 오버플로우): 산술 연산에서 정수형 표현 범위 초과로 발생하는 취약점. 입력 크기 초과로 발생하는 버퍼 오버플로우와 구분.
- 파일 업로드·웹 서버 보안
    - 파일 업로드 취약점 방어
        - 화이트리스트 확장자 검증: 허용된 확장자·MIME 타입만 업로드 허용.
        - 실행 속성 제거: 업로드 파일의 실행 권한 제거로 웹셸 실행 차단.
        - 웹 루트 외부 저장: 업로드 파일을 문서 디렉터리 외부에 저장해 URL 직접 접근 차단. (내부 저장은 오히려 위험)
        - 파일명 난수화: 사용자가 유추 불가한 임의 문자열로 저장.
        - Magic Byte 검증: Content-Type 헤더만이 아닌 파일 내용(매직 바이트)도 검사.
    - 웹 서버 강화
        - WebDAV (Web-based Distributed Authoring and Versioning): IIS의 원격 파일 공동 편집 기능. 불필요 시 비활성화.
        - Least Privilege (최소 권한 원칙): 웹 서버 프로세스 계정에 서비스 운영 최소 권한만 부여.
        - Directory Listing (디렉토리 리스팅) 비활성화: 디렉토리 조회 시 파일 목록 노출 차단.
        - 파일 업로드·다운로드 최대 크기 제한으로 DoS·대용량 악성파일 방지.
- OWASP Top 10 및 통합 가이드 (대표 항목)
    - Broken Access Control (접근 통제 결함): 권한 확인 미흡으로 수평·수직 권한 상승.
    - Cryptographic Failures (암호화 실패): 민감 데이터의 미암호화·약한 암호화.
    - Injection (삽입 공격): SQL/NoSQL/OS Command/LDAP Injection 포괄.
    - Insecure Design (안전하지 않은 설계): 위협모델링·보안 설계 패턴 부재.
    - Security Misconfiguration (보안 설정 오류): 기본 계정·불필요 서비스·디렉토리 리스팅 노출.
    - Vulnerable and Outdated Components (취약·구버전 구성요소): 패치 미적용 라이브러리 사용.
    - Identification and Authentication Failures (식별·인증 실패): 세션 고정·취약한 패스워드.
    - Software and Data Integrity Failures (소프트웨어·데이터 무결성 실패): 무결성 검증 없는 업데이트·역직렬화.
    - Security Logging and Monitoring Failures (로깅·모니터링 실패): 침해 탐지 지연.
    - Server-Side Request Forgery (SSRF, 서버 측 요청 위조): 서버가 임의 URL에 요청을 대신 보내도록 유도.
- 보안 취약점 진단 도구
    - 정적 분석 도구 (SAST, Static Application Security Testing)
        - SpotBugs (구 FindBugs): 자바 바이트코드 정적 분석 오픈소스. XSS·HTTP 응답 분할·Path Traversal·하드코딩 비밀번호·Null 포인터 역참조 탐지. Find Security Bugs 플러그인으로 OWASP 범위 확장.
        - SonarQube / Checkmarx: 상용·오픈소스 SAST 플랫폼.
    - 동적 분석 도구 (DAST, Dynamic Application Security Testing)
        - Burp Suite (버프 스위트): 웹 애플리케이션 보안 테스트 통합 플랫폼. HTTP 프록시·스파이더·스캐너·인트루더 포함.
        - OWASP ZAP (Zed Attack Proxy): OWASP 오픈소스 웹 취약점 진단 프록시.
        - Paros (파로스): 자바 기반 HTTP/HTTPS 프록시. 현재 OWASP ZAP으로 발전.
        - WebScarab (웹스카랩): OWASP 개발 웹 애플리케이션 취약점 분석 프레임워크.
        - Nikto / Acunetix: 웹 서버·애플리케이션 취약점 스캐너.
    - 도구 구별 주의
        - WAF (Web Application Firewall, 웹 방화벽) — ModSecurity는 런타임 트래픽 필터링 WAF이며 정적 분석 도구 아님.
        - Network Vulnerability Scanner — OpenVAS는 네트워크 취약점 스캐너로 소스·바이트코드 분석 도구 아님.
        - Dynamic Debugger — OllyDbg는 Windows 실행파일 동적 디버거(리버스 엔지니어링)로 Java 바이트코드 분석 도구 아님.
        - Password Cracker — Hydra는 FTP·SSH·HTTP 등 프로토콜 대상 브루트포스 도구로 웹 취약점 진단 도구 아님.
- 스니핑 관련 용어 주의
    - Sniffing (스니핑): 네트워크 트래픽을 수동적으로 도청하는 행위.
    - ARP Sniffing (ARP 스니핑): 네트워크 상의 ARP 패킷을 수동적으로 도청.
    - ARP Spoofing / Poisoning (ARP 스푸핑): IP-MAC 매핑을 능동적으로 위조하여 트래픽 가로채기. 스니핑과 혼동 금지.
    - 스니핑 방어: SSL/TLS 등 통신 암호화, 스위칭 네트워크 사용, OTP 활용.
- 애플리케이션 취약성 vs. 암호 공격 구분
    - 애플리케이션 취약성 공격: Buffer Overflow, SQL Injection, XSS, Remote Code Execution, 역직렬화 취약점 등 소프트웨어 코드 결함 직접 악용.
    - Rainbow Table Attack (레인보우 테이블 공격): 사전 계산된 해시 테이블로 패스워드 해시 역추적. 애플리케이션 코드 취약성이 아닌 패스워드 크래킹 기법. Salt 추가로 방어.

# 5.4.2 공격기법·악성코드

- DoS / DDoS (Denial of Service / Distributed DoS, 서비스 거부 / 분산 서비스 거부)
    - TCP 기반 공격
        - SYN Flooding (SYN 플러딩): 위조 출발지 IP로 TCP SYN을 대량 전송하여 서버 TCB (Transmission Control Block) 큐를 포화시켜 정상 연결 차단. 3-way handshake의 미완료 상태 누적 악용. 대응: SYN Cookie, 백로그 큐 증가, 타임아웃 단축.
        - Land Attack (랜드 공격): 패킷의 출발지 IP와 목적지 IP를 동일하게 위조하여 시스템 무한 응답 루프 유발.
    - IP 단편화 기반 공격
        - TearDrop Attack (티어드롭 공격): IP 분할(fragment) 패킷의 오프셋을 중첩되게 조작 → 수신측 재조합 시 버퍼 오버플로·크래시.
        - Bonk/Boink (봉크 공격): UDP 패킷 시퀀스 번호를 비정상 조작하여 수신측 재조합 오류 유발 (TearDrop 변형).
    - 증폭/반사 공격
        - Smurf Attack (스머프 공격): 피해자 IP로 위장한 ICMP Echo Request를 브로드캐스트 주소로 송신 → 네트워크 전 호스트가 피해자에게 Reply 폭주 (증폭 DDoS).
        - DNS Amplification (DNS 증폭 공격): 위조 출발지 IP로 DNS 재귀 질의 → DNS 서버가 피해자에게 대용량 응답 송신.
        - HTTP GET Flooding (HTTP GET 플러딩): 대량 HTTP GET 요청으로 웹 서버 자원 고갈.
        - Ping of Death (죽음의 핑): 비정상적으로 큰 ICMP 패킷으로 시스템 오동작·충돌 유발. ICMP echo-request 차단으로 방어.
    - TCB Dequeue 조건 (SYN Flooding 대조) — 3-way handshake 완료 / 타임아웃 / RST 수신 시 TCB 큐에서 제거. 위조 IP SYN 지속 전송은 Dequeue가 아닌 큐 포화 유발.
- 스푸핑 (Spoofing) — 신분·출처 위조
    - IP Spoofing (IP 스푸핑): 출발지 IP를 신뢰 호스트로 위조하여 접근 제어 우회.
    - ARP Spoofing / Poisoning (ARP 스푸핑 / 캐시 오염): 위조 ARP 응답으로 피해자 ARP 캐시에 공격자 MAC을 게이트웨이 IP에 연결 → MITM.
    - ARP Redirect Attack (ARP 리다이렉트 공격): 위조 ARP 응답을 브로드캐스트하여 네트워크 호스트들이 공격자를 라우터로 오인하게 만드는 공격.
    - DNS Spoofing / Cache Poisoning (DNS 스푸핑 / 캐시 포이즈닝): DNS 질의 스니핑 후 Transaction ID 위조한 가짜 응답을 먼저 전송, 피해자 캐시에 가짜 IP 주입. DNS 캐시 '삭제'가 아닌 '오염'이 핵심. 대응: DNSSEC, TLS 강제.
    - ICMP Redirect Attack (ICMP 리다이렉트 공격): 위조된 ICMP Redirect 메시지로 라우팅 경로를 공격자 경유로 변경.
    - Evil Twin AP (이블 트윈 AP): 정상 AP와 동일 SSID·강한 신호로 가짜 AP 설치 → 사용자 접속 유인 후 MITM. 대응: WIPS, AP 인증서 검증, VPN·HTTPS 강제.
- 스위치·TCP 세션 공격
    - Switch Jamming / MAC Flooding (스위치 재밍 / MAC 플러딩): 위조 MAC 주소 대량 전송으로 스위치 CAM 테이블 오버플로 → 허브처럼 브로드캐스트 → 스니핑 가능.
    - TCP Session Hijacking (TCP 세션 하이재킹): 이미 수립된 세션의 시퀀스 번호를 예측·조작하여 가로채는 공격.
        - 탐지 방법: 시퀀스 번호 비동기화 감지, 패킷 유실·재전송 증가 탐지, 예상치 못한 RST 발견.
        - 비탐지 방법: SYN Storm (SYN Flooding 탐지용)은 세션 하이재킹 탐지 지표가 아님.
- 악성코드 (Malware)
    - 유형별 분류
        - Virus (바이러스): 다른 프로그램(숙주)에 기생하여 자기 복제·전파. 숙주 필요.
        - Worm (웜): 네트워크를 통해 자기 복제·전파. 숙주 불필요.
        - Trojan Horse (트로이 목마): 정상 프로그램으로 위장한 악성 코드. 자기 복제 안 함, 원격 제어·정보 탈취.
        - Spyware (스파이웨어): 사용자 동의 없이 개인정보·브라우징 습관 수집하여 외부 전송.
        - Adware (애드웨어): 원치 않는 광고 표시, 탐색 정보 수집.
        - Keylogger (키로거): 키보드 입력을 몰래 기록하여 패스워드·카드번호 탈취.
        - Ransomware (랜섬웨어): 파일·시스템 암호화 후 복구 대가로 금전 요구.
        - PUP (Potentially Unwanted Program, 잠재적 원치 않는 프로그램): 형식상 사용자 동의 설치, 성능 저하·결제 유도.
    - 페이로드 전달자 (Loader 계열)
        - Dropper (드로퍼): 내부에 압축·암호화된 악성코드를 포함하여 실행 시 추출·설치(drop). 백신 패턴 탐지 우회에 유리. 자기 복제 없음.
        - Downloader (다운로더): 외부에서 악성코드를 다운로드하여 실행. Dropper와 구분.
        - Packer (패커): UPX 등으로 실행 파일을 압축·난독화. 다형성(Polymorphic)·변성(Metamorphic) 탐지 회피 기법.
        - Stager / Loader: 메모리 상 페이로드 전달·로드.
- 무선·블루투스 공격
    - 블루투스 공격
        - Bluejacking (블루재킹): 블루투스로 수신자 동의 없이 스팸성 메시지·이미지 전송.
        - BlueSnarfing (블루스나핑): 블루투스 장치 펌웨어 취약점으로 연락처·캘린더·SMS 등 저장 데이터를 무단 접근·탈취.
        - BlueBorne (블루본): 블루투스 프로토콜 취약점으로 원격 코드 실행·악성코드 전파 가능한 심각한 공격.
        - Blueprinting (블루프린팅): 블루투스 장치 정보(모델·제조사) 수집 (Reconnaissance).
- 기타 대표 공격 기법
    - Salami Attack (살라미 공격): 금융 프로그램에서 반올림 끝수 등 미세 금액을 지속적으로 탈취. 개별 피해는 미미하나 누적 피해 큼. 예: 이자 끝수 이체.
    - Data Diddling (데이터 디들링): 입력 데이터를 처리 전·중에 변조.
    - Exploit Attack (익스플로잇 공격): 소프트웨어 취약점을 이용한 일반적 공격 기법.
    - finger 명령어: 유닉스/리눅스 등록 사용자 정보(로그인명·실명·홈·최근 로그인) 외부 조회 도구. 권한 탈취 전 Reconnaissance (사전 정보 수집) 용도. 대응: fingerd 비활성, 포트 79 차단.
    - Reconnaissance (정찰 단계): 공격 전 목표 시스템 정보 수집 (포트 스캔, 사용자 열람, 서비스 탐지).

# 5.4.3 침해대응·포렌식·리버스 엔지니어링

- 디지털 포렌식 원칙 (Digital Forensics Principles)
    - 신속성의 원칙 (Timeliness): 디지털 증거는 덮어쓰기·소멸 위험이 크므로 전 과정을 지체 없이 신속히 진행.
    - 재현의 원칙 (Reproducibility): 동일 조건·방법으로 동일 결과가 도출되어야 함 (검증 가능성).
    - 무결성의 원칙 (Integrity): 수집된 증거가 위변조되지 않았음을 증명 (해시값·봉인 관리).
    - 연계보관성의 원칙 (Chain of Custody): 증거 획득·이송·분석·보관·법정 제출 각 단계의 담당자·책임자를 명확히 기록.
    - 정당성의 원칙 (Legitimacy): 적법 절차에 따라 증거 수집 — 위법 수집 증거는 수사관 동행 여부와 무관하게 증거 능력 배제(위법 수집 증거 배제 원칙).
- 파일 시스템 포렌식 대상 공간
    - File Slack (파일 슬랙): 파일에 할당된 디스크 클러스터 끝과 파일 데이터 실제 끝 사이의 빈 공간. 이전 삭제 파일 내용 잔존 가능 → 포렌식 복구 대상.
    - RAM Slack (램 슬랙): 파일 데이터가 끝난 섹터 내부의 남은 공간. 0으로 채워지거나 메모리 잔존물 포함.
    - Disk Slack (디스크 슬랙): 클러스터 마지막 섹터 이후의 미사용 섹터 공간.
    - Metadata (메타데이터): 파일명·크기·타임스탬프·권한 등 파일 부가 정보 (슬랙과 구분).
    - Journal (저널): 파일 시스템 변경 사항 로그. 장애 복구용, 슬랙 공간과 별개.
- 리버스 엔지니어링·분석 도구
    - Reverse Engineering (리버스 엔지니어링, 역공학): 완성된 장치·시스템·소프트웨어를 분해·분석하여 설계 원리와 동작 방식 파악. 취약점 분석·악성코드 분석·호환 SW 개발·프로토콜 분석에 활용. 지적재산권 침해 법적 이슈 고려 필요.
    - 실행파일 분석 도구
        - OllyDBG (올리디버거): x86 PE 실행파일 대상 대표적 동적 디버거 (어셈블리 수준 역공학).
        - x64dbg / IDA Pro / Ghidra / Radare2: 정적·동적 분석 역공학 플랫폼.
    - 용도 혼동 주의 도구
        - Wireshark (와이어샤크): 네트워크 패킷 캡처·분석 도구(pcap). 실행파일 역공학 아님, 패킷 생성·재전송도 아님 (재전송은 hping3·Scapy·tcpreplay 담당).
        - OpenVAS (Open Vulnerability Assessment System): 오픈소스 네트워크 취약점 스캐너. 역공학 아님.
        - Snort (스노트): NIDS (네트워크 침입 탐지 시스템). 역공학 아님.
- 네트워크 보안 점검 명령어
    - netstat: 네트워크 연결 상태·포트 사용·라우팅 테이블 조회 (대부분 OS 내장).
    - tcpdump: CLI 기반 패킷 캡처 도구. 구간 간 통신 확인.
    - nmap (Network Mapper): 포트 스캔·OS 탐지·서비스 버전 탐지 네트워크 스캐너.
    - Wireshark: GUI 패킷 캡처·분석 도구로 pcap 파일 분석.

# 5.5 표준,법령,지침

- 정보보호 관리체계·거버넌스
    - Security Governance (보안 거버넌스): 조직이 적합한 보안 정책을 수립하고, 정책에 따라 보안 조직을 구성하여 일련의 보안 활동을 지휘·감독하는 체계. IT 거버넌스의 하위 개념으로 이사회·경영진 수준의 방향 설정 포함.
    - ISO/IEC 27001 (ISMS 국제 표준): 정보보호 관리체계(Information Security Management System) 요구사항 규격. PDCA 순환 기반.
    - ISO/IEC 27014 (보안 거버넌스): 정보보안 거버넌스 프레임워크 국제 표준.
    - ISMS-P (Information Security Management System - Personal information, 정보보호 및 개인정보보호 관리체계): 한국 국내 ISMS + 개인정보보호 통합 인증.
    - ISMS-P 위험관리 절차
        - 1단계: Risk Identification (위험 식별) / 현황·흐름 분석 및 정보자산 식별.
        - 2단계: Risk Analysis·Assessment (위험 분석·평가) — 식별 자산 대상 위협·취약성·영향도 종합 평가, 조직 환경 분석 기반 방법 선정.
        - 3단계: Risk Treatment Plan (위험 대응 계획) — 위험별 회피·전가·완화·수용 전략 수립, 보호대책 선정.
        - 4단계: Risk Monitoring & Control (위험 모니터링·통제) — 대응 실행 및 신규 위험 지속 감시.
    - 정보자산 식별 원칙: 관리체계 범위 내 모든 중요 정보자산을 식별하고 중요도 산정. 일부 자산만 식별하는 것은 오류.
- 개인정보 비식별화 기법 (De-identification)
    - k-Anonymity (k-익명성): 준식별자(quasi-identifier) 조합이 동일한 레코드가 최소 k개 존재하도록 일반화·은닉. 동질성·배경지식 공격에 취약.
    - l-Diversity (ℓ-다양성): 동일 준식별자 그룹 내 민감 속성이 최소 l개의 다양한 값을 갖도록 보완. 쏠림·유사성 공격에 취약.
    - t-Closeness (t-근접성): 그룹 내 민감 속성 분포가 전체 데이터셋 분포와 t 이하 차이가 나도록 보완. l-Diversity의 한계 보완.
    - Data Reduction (데이터 삭제): 개인 식별 불필요 속성·레코드 삭제.
    - Data Suppression/Generalization (데이터 범주화): 값을 상위 범주로 대체 (예: 나이 → 20대).
    - Aggregation / 총계처리: 개별 데이터를 합계·평균 등으로 집계.
    - Data Masking (데이터 마스킹): 일부 식별 문자를 특수문자로 치환.
    - 비식별화 기법 아닌 것 — Data Synthesis (데이터 합성): 실제 데이터 통계 특성만 유지한 가상 데이터 생성 (원본 비식별화가 아닌 신규 데이터 생성). m-Clustering (m-군집화)도 비식별화 표준 기법 아님.
- 개인정보보호법 핵심 조항
    - 개인정보 보호 원칙 (법 제3조)
        - 목적 명확화·최소 수집 원칙: 처리 목적을 명확히 하고 최소한의 개인정보만 적법하게 수집 (명확화 생략 불가).
        - 정확성·완전성·최신성 보장 원칙: 처리 목적 범위 내 정확성·완전성·최신성 유지.
        - 안전성 확보 원칙: 위험 정도를 고려하여 개인정보를 안전하게 관리.
        - 사생활 침해 최소화 원칙: 정보주체의 사생활 침해를 최소화하는 방법으로 처리.
        - 처리방침 공개·열람 청구권 보장 원칙: 처리방침을 공개하고 열람 청구권 등 정보주체 권리 보장 (비공개 금지).
        - 익명·가명 처리 우선 원칙: 익명처리로 목적 달성 가능 시 익명처리, 불가 시 가명처리.
        - 책임 준수 원칙: 법령상 책임·의무를 준수·실천하여 정보주체 신뢰 획득 노력.
    - 개인정보 제3자 제공 동의 고지 사항 (법 제17조 제2항)
        - 개인정보를 제공받는 자.
        - 개인정보를 제공받는 자의 개인정보 이용 목적.
        - 제공하는 개인정보의 항목.
        - 개인정보를 제공받는 자의 개인정보 보유·이용 기간.
        - 동의를 거부할 권리가 있다는 사실 및 거부에 따른 불이익.
        - 포함되지 않음 — 정보제공 시 사용하는 보안조치는 별도 내부 관리 의무 사항.
    - 개인정보 유출 통지·신고 사항 (법 제34조)
        - 유출된 개인정보의 항목.
        - 유출 등이 된 시점과 그 경위.
        - 유출로 인한 피해를 최소화하기 위해 정보주체가 할 수 있는 방법.
        - 개인정보처리자의 대응 조치 및 피해 구제 절차.
        - 피해 발생 시 신고 등을 접수할 수 있는 담당 부서·연락처.
        - 포함되지 않음 — 유출된 개인정보 규모는 포함되나 '피해액'은 법정 통지 의무 사항 아님.
        - 1천 명 이상 유출 시 보호위원회 또는 전문기관 신고 의무.
    - 가명정보 처리 특례 (법 제3절, 제28조의2~5)
        - 가명정보 (Pseudonymized Information): 추가정보 없이는 특정 개인을 알아볼 수 없게 처리한 정보.
        - 동의 면제 목적: 통계작성·과학적 연구·공익적 기록보존 목적 시 정보주체 동의 없이 가명정보 처리 가능.
        - 재식별 금지: 특정 개인을 알아보기 위한 목적으로 가명정보 처리 금지.
        - 재식별 가능 정보 생성 시 조치: 즉시 처리 중지 + 해당 정보 삭제 (통지 후 계속 처리 불가).
        - 처리 기록 보관: 처리 목적·가명처리 항목·이용내역·제3자 제공 받는 자를 기록·보관.
        - 가명정보·추가정보 분리 보관 원칙 (물리적 분리 어려울 시 논리적 분리 허용), 접근권한 최소화(최소 권한 원칙, 균등 부여 금지), 접근통제 강화, 접속기록 보관·점검.
    - PIA (Privacy Impact Assessment, 개인정보 영향평가) — 법 제33조
        - 공공기관의 대규모 개인정보 처리 시스템 구축·변경 시 수행.
        - 고려사항: 처리하는 개인정보의 수, 개인정보의 제3자 제공 여부, 정보주체의 권리를 해할 가능성 및 그 위험 정도, 민감정보·고유식별정보 처리 여부(시행령 제37조).
        - 포함되지 않음 — 보유기간은 처리방침 기재·동의 고지·파기 의무 등 다른 조항에서 규율되며 영향평가 고려사항 아님.
    - 개인정보 안전성 확보조치 기준 (암호화 적용)
        - 주민등록번호: AES-256 (양방향 암호화, 복호화 필요).
        - 비밀번호: SHA-256 (단방향 해시, 복호화 불필요).
        - 바이오정보: AES-256 (양방향 암호화, 복호화 필요).
        - 일반 개인정보: AES-128 이상 양방향 암호화.
- OECD 개인정보보호 8대 원칙
    - Collection Limitation Principle (수집 제한 원칙): 적법·공정한 방법으로 수집, 정보주체 인지·동의 필요.
    - Data Quality Principle (정보 정확성 원칙): 이용 목적 범위 내 정확성·완전성·최신성 확보.
    - Purpose Specification Principle (목적 명확화 원칙): 수집 시점에 이용 목적 명확화.
    - Use Limitation Principle (이용 제한 원칙): 명시된 목적 외 이용·제공 제한.
    - Security Safeguards Principle (안전성 확보 원칙): 물리적·조직적·기술적 안전조치 확보.
    - Openness Principle (공개 원칙): 처리 정책·담당자 정보 등 투명 공개 (비공개 원칙 아님).
    - Individual Participation Principle (개인 참가 원칙): 정보주체의 열람·정정 권리 보장.
    - Accountability Principle (책임 원칙): 원칙 준수에 대한 책임.
- 보안쳬계 보안가이드라인 1.0
    - 정보 등급 분류 단계 : 기밀정보(C), 민감정보(S), 공개정보(O)
# 5.6 기타(신기술)

- 블록체인 (Blockchain)
    - Blockchain (블록체인): 중앙 서버 없이 다수 노드가 동일 원장(Ledger)을 분산 보관하는 분산원장기술. 각 블록이 이전 블록 해시로 연결되어 무결성 보장.
    - 4대 특성 — Decentralization (분산성), Transparency (투명성), Immutability (불변성), Security (보안성).
    - 블록 구조
        - Block Header (블록 헤더): 이전 블록 해시, Merkle Root, 타임스탬프, Nonce 등 메타데이터 저장.
        - Block Body (블록 바디): 실제 트랜잭션 원본 저장.
        - Merkle Root (머클 루트): 블록 내 모든 트랜잭션의 해시를 트리 구조로 집약한 단일 해시값. 헤더에 포함.
    - 트랜잭션 불변성: 기록된 트랜잭션은 삭제·수정 불가. 취소는 역방향 트랜잭션(Reverse Transaction) 생성으로 처리, 예외적으로 하드포크 거버넌스 합의.
    - 트랜잭션 보호 기술: 공개키 암호(ECDSA 등) 서명, 해시함수(SHA-256) 블록 연결, PoW/PoS 합의 메커니즘.
    - Smart Contract (스마트 계약): 블록체인 트랜잭션 DB에 포함되어 트랜잭션과 함께 실행되는 비즈니스 규약 코드.
    - 블록체인 분류 — Public (퍼블릭, 비트코인·이더리움 등 개방·투명·성능 제약) / Private (프라이빗) / Consortium (컨소시엄).
- 블록체인 포크·보안 공격
    - Soft Fork (소프트 포크): 후방 호환성(backward-compatible) 지원. 규칙을 더 엄격하게 제한하므로 미업그레이드 노드도 새 블록을 유효로 인식. 예: 비트코인 SegWit.
    - Hard Fork (하드 포크): 후방 호환성 없음. 모든 클라이언트 필수 업그레이드, 체인이 영구 분리될 수 있음. 예: 이더리움 클래식 분리.
    - 블록체인 대상 공격
        - Double Spending Attack (이중 지불 공격): 동일 암호화폐를 두 번 이상 사용. 51% 공격, Race Attack, Finney Attack으로 시도.
        - Eclipse Attack (이클립스 공격 / 허위 정보 전파): 특정 노드의 모든 P2P 연결을 공격자 노드로 교체하여 네트워크에서 격리·허위 정보 주입.
        - Reentrancy Attack (재진입 공격): 스마트 컨트랙트에서 함수 완료 전 재귀 호출로 잔액 반복 인출 (이더리움 DAO 해킹 사례).
        - Sybil Attack (시빌 공격): 다수의 가짜 신원(노드) 생성으로 네트워크·합의 교란.
        - 51% Attack (51% 공격): 전체 해시 파워의 과반 장악 후 체인 재작성.
        - 블록체인 공격 아님 — Smurf Attack (스머프)은 ICMP 브로드캐스트 기반 DDoS로 블록체인 특화 공격 아님.
- AI 보안
    - Deepfake (딥페이크): 딥러닝과 Fake 합성어. GAN (Generative Adversarial Network, 생성적 적대 신경망) 등으로 진위 구별이 어려운 가짜 이미지·영상 생성. 가짜 뉴스·불법 성인물·사기·정치 조작 등 부작용 존재 (부작용 없다는 설명은 오류). 탐지: CNN 기반 모델, 디지털 워터마킹, 블록체인 기반 콘텐츠 인증.
    - AI/ML 공격 유형
        - Poisoning Attack (데이터 오염 공격): 학습 단계에서 훈련 데이터에 악의적 데이터 주입하여 모델 예측 왜곡.
        - Evasion Attack (회피 공격): 추론 단계에서 적대적 예제(adversarial example)로 오분류 유도.
        - Inversion Attack (모델 역전 공격): 반복적 쿼리·결과 분석으로 학습 데이터 재구성·추출. 대응: Differential Privacy (차분 프라이버시), 쿼리 횟수 제한, 출력 결과 제한.
        - Model Extraction Attack (모델 추출 공격): 쿼리·응답으로 모델 구조·파라미터 복제.
- 클라우드 보안
    - 주요 위협
        - 제3자 위탁 저장 관리 부주의 유출: 클라우드 사업자 내부자·관리 허점으로 인한 고객 정보 유출.
        - 공격 집중 타깃: 다수 고객사 데이터 집중으로 DDoS·APT 공격 주요 목표.
        - VM 간 도청·해킹: IaaS 이용 시 효과적 자원 분리 실패로 가상머신 간 Side-Channel·VM Escape 공격 피해.
        - 멀티테넌시 격리 실패: 하이퍼바이저 취약점·공유 자원 재할당 문제.
    - 클라우드 가상화 원리: 물리적 자원을 논리적으로 분할·추상화하여 다수 사용자 제공. "논리적 자원을 물리적으로 통합"은 방향이 반대인 오설명.
    - 서비스 모델 — IaaS (Infrastructure as a Service, 인프라형 서비스) / PaaS (Platform as a Service) / SaaS (Software as a Service).
    - 가상화 보안: 하이퍼바이저 보안 강화, VM 간 네트워크 격리, 보안 그룹·IAM 정책 적용.
- Zero Trust (제로 트러스트)
    - Zero Trust Principle (제로 트러스트 원칙): "Never Trust, Always Verify" — 절대 신뢰하지 말고 항상 검증. 네트워크가 이미 침해된 것으로 간주.
    - 핵심 원칙: Least Privilege (최소 권한), Continuous Authentication (지속 인증), Micro-segmentation (마이크로 세그멘테이션), Data-centric Protection (데이터 중심 보호).
    - 대조 개념: 경계 기반 보안(Perimeter Security, 내부 신뢰·외부 불신)에서 Zero Trust로 전환. 경계 강화가 아닌 경계 탈피가 핵심.
    - 표준 문서: NIST SP 800-207 — Zero Trust Architecture.
- IoT 보안 공격
    - De-synchronization Attack (비동기화 공격): 타임 서버·동기화 풀에 잘못된 시간 정보 지속 주입 → IoT 디바이스가 시간 재교정에 배터리·CPU·네트워크 자원 소모 (DoS 계열).
    - Sleep Deprivation Attack (수면 박탈 공격): 저전력 모드 진입 방해로 IoT 배터리 고갈.
    - Flooding (플러딩): 대량 패킷으로 네트워크·처리 자원 고갈.
    - Eavesdropping (이브스드로핑, 도청): 네트워크 트래픽 수동 수신으로 정보 탈취.
    - NTP (Network Time Protocol, 네트워크 시간 프로토콜): 시간 동기화 프로토콜, IoT 환경에서 De-synchronization 공격에 악용 가능.
