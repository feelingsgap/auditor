정보시스템감리사 필기시험의 '소프트웨어 공학' 영역은 SW 공학 개념부터 유지보수·운영까지 광범위한 주제를 다룹니다. 다음은 기출문서 분석을 바탕으로 소프트웨어 공학의 세부 범위별로 공부해야 할 내역을 정리한 것입니다.

### 소프트웨어 공학 세부 공부 내역

- **SW 공학 개념**
    - [SDLC] 폭포수 모델, V-모델, 프로토타이핑, 나선형 모델, 진화형, 점증형, 통합 프로세스(UP) 각 특성과 비교
    - [SW 공학 원론] SW 위기의 원인·정의, SW 공학 목적, TDD(테스트 주도 개발) 철학

- **SW 개발 방법론**
    - [전통 방법론] 구조적 방법론, 정보공학 방법론, 객체지향 방법론, CBD(Component Based Development) 성숙 단계(Burtler 모델)
    - [애자일] 애자일 생애주기(미완성 작업 피드백 수용, 완성된 인도물 제공, 점진적 방법 추구), 애자일 원칙·가치
    - [스크럼] 스크럼 구성 요소: 제품 백로그(해야 할 일 목록, 요구사항 포함), 스프린트(개발 반복, 주로 2~4주), 스크럼 마스터(팀이 프로세스를 따르도록 지원 — 프로젝트 관리자와 다름), PO(Product Owner)
    - [XP·린·칸반] XP 실무(단순 설계, 공동 소유권, 연속 통합, 리팩토링, 고객 온사이트), Planning Poker(착수 준비 → 추정 대상 토론 → 추정 수행), 린 소프트웨어 개발
    - [MSA 설계 절차] 마이크로서비스 아키텍처 설계 순서: 핵심 부분 스케치(개념화·개발조직·서비스 구성) → 정책·제약·원칙 정의 → 비기능 요구사항(신뢰성·보안) 식별 → 가시성 진단 및 영향 분석
    - [애자일 지표] 번다운 차트(시간 경과에 따른 남은 작업량), 번업 차트(전체 목표 대비 완료 현황), 속도(velocity), 배포그 진척률, 배포그 항목 수

- **계획**
    - [비용 산정] LOC 방식; COCOMO I(기본·중간·발전); COCOMO II 2000 계수조정모델(Scale of Economy, EQ·ED 복잡도 가중치, 재사용 모델 AAF=CM/DM/IM); 기능점수(IFPUG): 개발 FP(DFP), 개선 FP(EFP = (ADD+CHGA+CFP)×VAFA ± DEL×VAFB)
    - [일정·위험] CPM, PERT, 간트(Gantt) 차트, 크러싱, 패스트 트래킹, 임계 경로, 작업 병렬화; 위험 식별·평가·대응, 위험 노출도

- **요구분석**
    - [도출] 문화기술적 연구(ethnography, 사용자 일상 관찰), 프로토타이핑(고객이 요구사항을 구체적으로 파악 못할 때), 인터뷰, 워크숍, 브레인스토밍, 관찰(Observation)
    - [분류·명세] 기능/비기능 요구사항, 전환·비즈니스·이해관계자 요구사항; Use Case 다이어그램, SysML Requirement Diagram(요구사항의 계층 구조·모든 도메인과의 관계·개별 요구사항 식별·입출력의 흐름·시스템 특성 제약사항 수식 표현)
    - [검증·품질 속성] 요구사항 검증 점검 항목 — 유효성(사용자 실제 요구 반영), 일관성(상충·모순 없음), 완전성(모든 기능·제약 포함), 실현성(기술·예산 범위 내 구현 가능); 요구사항 추적성(Traceability)

- **설계**
    - [설계 원칙] SOLID: SRP(단일 책임), OCP(개방·폐쇄), LSP(리스코프 치환), ISP(인터페이스 분리), DIP(의존성 역전 — 추상 클래스·인터페이스에 의존)
    - [Clean Architecture] 의존성 방향: Entities → Use Cases → Interface Adapters → Frameworks and Drivers; 컴포넌트 결합도 3원칙: 비순환 의존성, 안정적인 컴포넌트에 의존, 추상화된 컴포넌트에 의존; 패키지 Instability = Fan-out / (Fan-in + Fan-out)
    - [SW 아키텍처 스타일] 파이프 앤 필터(데이터 흐름, 중간 처리, 최종 목적지 변환), MVC(UI 분리, 응집력 향상, 다중 UI 지원), Publish-Subscribe(이벤트 분배자 연결, 데이터 생성자·사용자 구분), 계층형, Client-Server, Master-Slave, 이벤트 기반
    - [SOA·웹 서비스] SOAP(메시지 교환 표준), WSDL(서비스 인터페이스 정의), WS-BPEL(여러 서비스 포함 프로세스 프로그램 정의 워크플로우 언어); REST(자원 기반, GET/POST/PUT/DELETE, Stateless)
    - [GoF 디자인 패턴] Strategy(알고리즘 캡슐화, 교체 가능), Observer(상태 변화 알림), Command(요청 객체화, LEDLight 예시), Decorator(동적 기능 추가), Bridge(구현과 추상화 분리), Flyweight(공유로 메모리 절약), Proxy(대리 접근), State(상태에 따른 행동 변화), Iterator(순회 추상화), Chain of Responsibility(핸들러 연결, alt-else 패턴), Template Method(알고리즘 골격 정의), Adapter(인터페이스 변환)
    - [소프트웨어 제품라인(SPL)] 공통 아키텍처와 공유 컴포넌트로 다양한 고객 요구에 적응하는 애플리케이션 집합; Feature diagram(필수/선택/OR/XOR 특성 모델)
    - [UML] Use Case(include·extend 차이 — extend는 선택 조건에서 확장, include는 항상 포함), Class diagram(연관·상속·인터페이스·중첩 클래스 Java 매핑), Sequence diagram(loop·alt·opt 프레임, 아키텍처 패턴 판별), State Machine(trigger [guard] / effect 전이 표기, exit/entry 액션 순서), Activity, Component diagram
    - [SysML] Block Definition diagram, Parametric diagram(시스템 변수 표현, 제약 조건 변수와 값 속성 바인딩, 수학적 관계 지정), Use Case·Class·Component(UML 공통 다이어그램)
    - [응집도·결합도] 응집도(좋음→나쁨): 기능적·순차적·통신적·절차적·시간적·논리적·우연적; CBO(Coupling Between Objects), LCOM(Lack of Cohesion in Methods), RFC(Response For a Class) — 계산 측정 문항은 2.9 품질표준으로
    - [아키텍처 평가] ATAM, CBAM(비용편익 분석); SEI 아키텍처 뷰 모델(C&C뷰, 모듈뷰, 할당뷰)

- **구현**
    - [코드 분석] Java `static` 메서드 내 인스턴스 변수·메서드 접근 시 컴파일 오류; Java 내부 클래스·익명 클래스 변수 스코프(final 선언 필요); Python list pop()/insert()/index() 메서드 동작; C 포인터 연산 (*ptr++, **ptr)
    - [프레임워크] Spring AOP 어드바이스 유형: AfterReturning(정상 반환 후), Around(메서드 전후 감싸기), AfterThrowing(예외 발생 시); Spring MVC: DispatcherServlet → HandlerMapping → Controller → ModelAndView → ViewResolver → View
    - [리팩토링 기법] Extract/Inline Method, Move Method, Substitute Algorithm, Rename Method, Hide Delegate, Remove Middle Man, Encapsulate Variable, Encapsulate Downcast, Split Temporary Variable(임시변수 분리), Remove Assignments to Parameters, Replace Conditional with Polymorphism(조건 다형성 대체), Replace Error Code with Exception, Replace Type Code with Subclasses(서브클래스 계층 생성), Pull Up Method
    - [코드 악취(Bad Smell)] Feature Envy(다른 클래스의 기능에 욕심), Shotgun Surgery(변경 시 많은 클래스 수정), Data Clumps(유사 데이터 중복), Divergent Change(확산적 변경), Switch/if 문 과다; Bad Smell↔리팩토링 매핑: Feature Envy→Move Method, Shotgun Surgery→Move Method+MergeClass, Data Clumps→독립 클래스 정의, Switch if→Replace Conditional with Polymorphism
    - [재공학] 역공학(Reverse Engineering), 재구조화(Restructuring), 이식(Migration); 레거시 시스템 결정(시스템 품질 × 비즈니스 가치 매트릭스): 고품질·고가치→유지, 저품질·고가치→재공학
    - [오픈소스 라이선스] GPL(소스코드 제공 의무, 파생물도 GPL), LGPL(라이브러리 형태 허용, 수정 시 고지·날짜 명기), MPL(라이브러리 배포 시 본 라이브러리 사용 명시), BSD(광범위한 재배포 허용, 저작권 안내 명시)
    - [클라우드 컨테이너] 독립된 환경, 운영체제 공유(하이퍼바이저 불필요), 쿠버네티스(컨테이너화된 워크로드·서비스 오케스트레이션), 컨테이너 간 다른 OS 사용 불가; 모놀리식 아키텍처 → MSA 전환 시 컨테이너 기반 개발이 유리

- **테스트**
    - [커버리지 체계] 문장(Statement)·결정(Decision)·조건(Condition)·결정/조건·변형 조건/결정(MC/DC)·다중 조건; 포함 관계: 다중 조건⊃MC/DC⊃결정/조건⊃(결정, 조건)⊃문장; Every Branch Coverage와 Every Statement Coverage 충족 여부를 코드+테스트 케이스로 분석
    - [블랙박스 기법] 동등 분할(Equivalence Partitioning), 경계값 분석, Cause-Effect Graphing(Cause 간 제약: Mask·Requires·Exclusive·One and only one — "Inclusive"는 없음), 결정 테이블(TC1~TC4 규칙 적용 커버리지 계산), 상태 전이(State Transition), 시나리오(End-to-End), 요구 기반(Requirement-based)
    - [조합 테스트] all combination(가장 많은 TC), pairwise(2개 인자 조합), base choice(기본값 기반, 가장 적은 TC); 기능점수 파라미터 조합 계산(글꼴이름×크기×스타일 = base choice 공식)
    - [AI·ML 테스트] 메타모픽 테스트(입력 변형에 따른 출력 변형 규칙 이용 — AI 강건성·양방향 그래프 거리 동일성 검증), 뉴런 커버리지(Neuron coverage — 활성화 뉴런 측정), 계층 커버리지(Layer coverage), Sign-sign coverage; 적대적 공격(Adversarial attack — 노이즈 포함 입력으로 잘못된 동작 유도); 거리 척도: 유클리드·체비쇼프·맨해튼·세이프 거리
    - [테스트 성숙도] TMMi 레벨 2 테스트 계획(test planning) 프로세스 영역 세부 목표: 제품 위험 평가(perform a product risk assessment), 테스트 접근 방식 설정(establish a test approach), 테스트 추정치 설정(establish test estimates) — "테스트 성능 지표 설정"은 레벨 2 목표가 아님
    - [표준·프레임워크] ISO/IEC/IEEE 29119 Part 2(전체적 적합성: 모든 프로세스 요구사항 만족, 맞춤형 적합성: 합의된 프로세스의 요구사항 만족); JUnit4 실행 순서: @BeforeClass → @Before → @Test → @After → (반복) → @AfterClass; Release Testing(외부 배포 이전 요구명세 기반 블랙박스)

- **유지보수 및 운영**
    - [유지보수 유형] 수정형(corrective, 결함 수정), 적응형(adaptive, 환경 변화 적응), 완전형(perfective, 성능·기능 향상 — 예: DB 재구성으로 검색 속도 향상), 예방형(preventive, 미래 문제 예방)
    - [유지보수 절차] 변경 요청 접수 → 영향 분석 → 릴리즈 계획 → 변경 구현·테스트 → 배포
    - [IT 거버넌스] COBIT(Control Objectives for IT), ISO/IEC 38500
    - [ITIL 4] Service Value System(SVS): 14개 General Management Practices(Change Control, IT Asset Management, Software Deployment and Management 포함), 17개 Service Management Practices, 3개 Technical Management Practices; "Business Analysis"는 General이 아닌 별도 영역

- **프로젝트 공통**
    - [형상관리] Baseline(특정 시점 컴포넌트 버전 집합); Configuration Identification(식별자 부여), Configuration Control(변경 평가·조정·승인), Configuration Status Accounting(변경 히스토리 기록), Configuration Verification(승인 명세와 일치 평가); 코드라인(A, B, C 병렬 개발)·베이스라인(공유 컴포넌트 묶음)·메인라인(통합 기준)
    - [Git] push(로컬→원격 전송), fetch(원격→로컬 복사), pull(원격 최신 가져오기+병합), clone(원격 복제), checkout(HEAD를 특정 브랜치로 이동), branch(브랜치 생성), stash(작업 임시 저장); 브랜치(커밋 가리키는 포인터), master(최초 커밋 생성, 최근 커밋 가리킴)
    - [품질 표준] ISO/IEC 9126/25010 품질 특성: 기능적합성·성능효율성·호환성·사용성·신뢰성·보안성·유지보수성(다른 하드웨어 적응=이식성, 수정 가능=변경성, 테스트 기준 충족=시험용이성)·이식성; ISO/IEC 25023 척도; ISO/IEC 5055 품질 특성(Maintainability, Security, Reliability, Performance Efficiency — Usability 제외); CMMI-DEV 22개 프로세스 영역 분류: 수준 3 정의(defined) 단계 포함 영역 = Risk Management, Integrated Project Management, Organizational Process Definition(다 수준 3) — Quantitative Project Management·OPP는 수준 4, Technical Solution은 엔지니어링 PA; SPICE(ISO 15504) 성숙도 레벨: L0 Incomplete → L1 Performed → L2 Managed → L3 Established → L4 Predictable → L5 Optimizing
    - [품질 지표·측정] 사이클로매틱 복잡도(McCabe CC: 결정 노드 수 + 1, 또는 간선 - 노드 + 2); POFOD(Probability of Failure on Demand: 인명·환경 피해 가능 시스템 신뢰성 척도 — MTBF·MTTF·ROCOF와 구분); PCE(Phase Containment Effectiveness: 단계별 생성 결함 중 해당 단계에서 발견 비율); 컴포넌트 안정성: Fan-in(의존 수)/Fan-out(의존하는 수), Instability=Fan-out/(Fan-in+Fan-out)
    - [신뢰성·Dependability] 가용성(Availability: 특정 시점에 작동 중이고 유용한 서비스 제공 확률), 복원성(Resilience: 장비 고장·사이버 공격 등 손상 사건에서도 중요 서비스 유지 추정); ISO 22989:2022 AI 신뢰성(Trustworthiness) 속성: AI Robustness, AI Resilience(사고 후 시스템 빠르게 복구·일부 유형 장애에서 저하된 수준 운영 제공), AI Controllability, AI Explainability; 결함 회피(fault avoidance: 강한 자료형, 포인터 최소화)·결함 감지 및 정정(fault detection: 체계적 테스팅·디버깅·정적 분석)
    - [SW 프로세스 인증] SP(소프트웨어 프로세스) 인증: 프로세스 평가(프로세스와 수행 능력 수준 단계 기준), NIA(한국지능정보사회진흥원) 담당, 5개 영역·14개 평가항목·60개 세부 평가항목; CWE(Common Weakness Enumeration: 악의적 행위자가 사용할 수 있는 SW 아키텍처·소스 코드 문제점 목록)
