정보시스템감리사 필기시험의 '소프트웨어 공학' 영역은 설계 원칙부터 프로그래밍 이해까지 광범위한 주제를 다룹니다. 다음은 기출문서(,) 분석을 바탕으로 소프트웨어 공학의 세부 범위별로 공부해야 할 내역을 정리한 것입니다.

### 소프트웨어 공학 세부 공부 내역

#### 1. 설계 원칙

- **객체 지향 설계 원칙**의 개념과 적용을 이해해야 합니다. 특히 **SRP(단일 책임 원칙)**, **OCP(개방-폐쇄 원칙)**, **DIP(의존성 역전 원칙)**, **LSP(리스코프 치환 원칙)**와 같은 SOLID 원칙에 대한 지식이 필요합니다.
    - 예시: 단위 테스트 시 테스트 대상 클래스를 변경하지 않고도 환경을 통제 가능하도록 하는 설계(, 2021(26)).
    - **DIP(의존성 역전 원칙)**는 추상 클래스와 인터페이스가 구체적인 클래스보다 덜 변하기 때문에, 외부에서 해당 클래스를 참조할 때 가능한 추상 클래스에 의존하는 것을 권장합니다(, 2024(46)).
- **클린 아키텍처(Clean Architecture)**는 외부 요소에서 내부 요소 방향만의 의존성을 강조하며, 구성 요소들의 적절한 순서(Entities – Use Cases – Interface Adapters – Frameworks and Drivers)를 파악해야 합니다(,, 2022(41), 2025(26)).

#### 2. 디자인 패턴

- **주요 디자인 패턴**의 목적과 적용 상황을 숙지해야 합니다. 객체의 생성, 관계 설정, 행동 지정 등을 통해 **재사용 가능한 해결책을 제시**하는 패턴들입니다().
    - **Strategy 패턴, Observer 패턴, Command 패턴, Decorator 패턴** 등을 이해합니다(, 2021(27)).
    - **Bridge 패턴, Decorator 패턴, Flyweight 패턴, Proxy 패턴**의 특징을 파악합니다(, 2021(31)).
    - 특정 동작(예: Button 클릭 시 LEDLight 동작)에 대한 설계에서 **Command 패턴과 Adapter 패턴**의 적용 예시를 이해해야 합니다(, 2022(43)).
    - Sequence diagram을 보고 **Iterator, Strategy, State, Chain of Responsibility 패턴** 중 어떤 패턴이 적용되었는지 파악할 수 있어야 합니다(, 2023(26)).
    - **파이프 앤 필터(Pipe and Filter)** 아키텍처 패턴의 특징을 이해해야 합니다 [2025(29)].
    - **State 패턴**과 **Template Method 패턴**의 차이점을 이해하고, 주어진 시나리오에 적합한 패턴을 선택할 수 있어야 합니다 [2025(48)].
    - **Replace Type Code with Subclasses** 리팩토링 기법을 이해해야 합니다 [2025(49)].

#### 3. 애자일 방법론

- **애자일 개발**의 품질 관리가 **문서 기반의 정형적인 방식이 아닌**, 코드에서 문제 발견 시 원래 개발자가 아니더라도 직접 수정할 수 있는 **유연한 방식**을 따른다는 것을 이해합니다(, 2021(28)).
- **애자일 생애주기(Agile Life Cycle)**의 개념(미완성 작업에 대한 피드백 수용, 완성된 인도물 제공, 작업 항목 파악 후 점진적 방법 추구)을 알아야 합니다(, 2023(2)).
- **스크럼(Scrum) 방법론**의 주요 요소와 그 역할을 학습합니다.
    - **스크럼 마스터**: 팀이 스크럼 프로세스를 효과적으로 따르도록 돕는 역할(, 2023(37)).
    - **제품 백로그**: 해야 할 일에 대한 목록으로 소프트웨어 요구사항 등이 포함됩니다(, 2023(37)).
    - **스프린트**: 개발에서 이루어지는 반복으로 주로 2~4주 길이로 구성됩니다(, 2023(37)).
    - **익스트림 프로그래밍(eXtreme programming)**의 실무(단순한 설계, 공동 소유권, 연속적 통합, 리팩토링)를 이해해야 합니다(, 2021(38)).
- **스토리 점수 추정 방법** 중 **플래닝 포커(Planning Poker)**의 절차(착수 준비, 추정 대상 토론, 추정 수행)를 알아야 합니다(, 2022(38)).
- **애자일 성과 지표**에는 백로그 진척률, 번업 차트, 번다운 차트, 속도 등이 있습니다(, 2024(50)).

#### 4. 테스트 기법

- **테스트 커버리지(Test Coverage)**의 종류와 포함 관계를 이해합니다.
    - **결정 커버리지(decision coverage) 100%** 만족 시 **문장 커버리지(statement coverage) 100%** 만족합니다(, 2021(29)).
    - **결정/조건 커버리지(decision/condition coverage) 100%** 만족 시 **결정 커버리지(decision coverage)와 조건 커버리지(condition coverage) 100%**를 모두 만족합니다(, 2021(29)).
- **변형 조건/결정 커버리지(MC/DC)**를 달성하는 최소한의 테스트 케이스 쌍을 생성하는 방법을 알아야 합니다(, 2022(33)).
- **조합 테스트 기법(Combinatorial Test Techniques)**을 비교하고, **all combination testing**이 가장 많은 테스트케이스를 생성하며, **base choice testing**이 가장 작은 테스트케이스를 생성한다는 것을 파악합니다(, 2021(44),, 2024(36)).
- **인공신경망 테스팅** 시 사용할 수 있는 커버리지(Neuron coverage, Layer coverage, Sign-sign coverage)를 이해합니다(, 2023(29)).
- **Cause-Effect Graphing**에서 Cause 간의 제약 사항(Mask, Requires, Exclusive, One and only one)을 파악해야 합니다(, 2024(34)).
- **릴리즈 테스팅(Release testing)**은 외부 배포 이전에 시스템의 **요구명세 기반 블랙박스 테스팅**을 수행하는 것을 의미합니다(, 2023(47)).
- **Metamorphic testing**은 입력 변형에 따른 출력 변형 규칙을 이용하는 테스트 기법입니다(, 2024(37)).
- **TMMi(Test Maturity Model Integration)** 레벨 2의 테스트 계획 프로세스 영역 세부 목표(제품 위험 평가 수행, 테스트 접근 방식 설정, 테스트 추정치 설정)를 숙지해야 합니다(, 2024(43)).
- **테스트 설계 기법**: **State transition testing, Decision table testing, Scenario testing, Requirement-based testing**의 특징을 구분할 수 있어야 합니다 [2025(28)].
- **인공지능 테스트 관련 설명**: **메타모픽 테스트, 뉴런 커버리지, 적대적 공격** 등의 개념을 이해해야 합니다 [2025(33)].
- **커버리지 기준**: **Every Branch Coverage**와 **Every Statement Coverage**의 충족 여부를 주어진 코드와 테스트 케이스를 통해 분석할 수 있어야 합니다 [2025(42)].
- **UML 표기법**: **include**와 **extend**의 차이점을 이해하고, 특정 시나리오에 맞는 표기법을 선택할 수 있어야 합니다 [2025(45)].

#### 5. 객체지향 프로그래밍 / UML

- **ISO 9126 품질 특성 분류 기준**에 따라 전화번호 관리 애플리케이션의 리스크를 식별하는 예시를 이해해야 합니다(, 2021(30)).
- Java 코드에서 `static` 메서드 내의 인스턴스 변수/메서드 접근과 같은 **컴파일 오류 발생 지점**을 식별하는 능력이 필요합니다(, 2021(32)).
- **스프링 프레임워크의 어드바이스 유형**(AfterReturning, Around, AfterThrowing 어드바이스)을 파악해야 합니다(, 2021(34)).
- **UML 상태머신 다이어그램**의 전이 표기법(`trigger [guard] / effect`)을 정확히 알아야 합니다(, 2021(39)).
- **UML 클래스 다이어그램**과 Java 코드 간의 매핑 관계(연관 관계, 상속, 인터페이스 구현, 중첩 클래스 등)를 이해해야 합니다(, 2021(42), 2025(30), 2025(44)).
- **ISO/IEC 25010 신뢰성**은 예상치 못한 프로그램 종료 후 재시작 시 기존 문서를 바로 보여주는 것과 같이 시스템이 의도한 기능을 지속적으로 수행할 수 있는 능력과 관련됩니다(, 2022(42)).
- **UML 시퀀스 다이어그램**을 기반으로 **클래스 다이어그램**의 문제점을 식별하는 능력이 중요합니다(, 2022(45)).
- **객체 지향 척도(Chidamber와 Kemerer)**인 **LCOM(Lack of cohesion in methods)**과 **RFC(Response for a class)**의 개념과 계산 방법을 알아야 합니다(,, 2022(40)).
- **순환 복잡도(CC)와 객체 간 결합도(CBO)**를 시퀀스 다이어그램 등에서 계산하는 방법을 이해해야 합니다(, 2022(50)).
- **SysML**은 UML을 차용하여 복잡한 시스템을 모델링하며, **Use case diagram, Class diagram, Component diagram**은 SysML과 UML에 공통적으로 존재하는 다이어그램입니다(, 2023(31)).
- **다형성(Polymorphism)**은 하나의 인터페이스나 추상 클래스를 통해 여러 다른 타입의 객체를 다룰 수 있는 객체지향 개념입니다(, 2024(30)).
- **SysML의 Parametric diagram**은 시스템 변수 값을 표현하고, 제약 조건 변수와 값 속성 간 바인딩을 통해 수학적 관계를 지정합니다(, 2024(31)).

#### 6. 리팩토링

- **리팩토링**은 코드 개선 사항 발견 시 지속적으로 코드를 개선하여 단순하고 유지보수하기 쉽게 만드는 활동입니다(, 2021(38)).
- **코드 악취(Bad Smells)**와 이를 해결하는 **리팩토링 방법**을 연결하여 이해해야 합니다.
    - **산탄총 수술(Shotgun Surgery)**: 변경 시 많은 클래스를 조금씩 수정해야 하는 문제(, 2021(41)).
    - **기능에 대한 욕심(Feature Envy)**, **데이터 덩어리(Data Clumps)**, **확산적 변경(Divergent Change)** 등 주요 코드 악취를 알아야 합니다(, 2021(41)).
    - **Inline Method, Extract Method, Move Method, Substitute Algorithm**과 같은 기본적인 리팩토링 기법들을 이해합니다(, 2021(35)).
    - **Hide Delegate, Remove Middle Man, Rename Method, Encapsulate Downcast** 등도 리팩토링 방법입니다(, 2021(50)).
    - **Replace Error Code with Exception**: 에러 코드를 예외 처리로 대체하는 리팩토링 기법입니다(, 2022(34)).
    - **Encapsulate Variable**: 변수를 캡슐화하고 접근 메서드를 제공하여 리팩토링하는 것입니다(, 2022(28)).
    - **Split Temporary Variable**(임시변수 분리), **Remove Assignments to Parameters**(매개변수 할당 제거), **Replace Conditional with Polymorphism**(조건 다형성 대체) 등의 리팩토링 방법도 알아야 합니다(, 2024(32)).

#### 7. 코드 품질 및 복잡도

- **응집도(Cohesion)**의 종류를 좋은 응집도에서 나쁜 응집도 순서로 파악합니다(기능적, 순차적, 통신적, 절차적, 시간적, 논리적, 우연적)(, 2021(36)).
- **CMMI-DEV**는 소프트웨어 개발 시 프로세스 품질을 향상시키는 가이드라인으로, 프로세스 영역(Configuration Management, Technical Solution, Decision Analysis and Resolution, Causal Analysis and Resolution)을 이해해야 합니다(, 2023(32)).
- **시스템 확실성(Dependability)** 속성인 **가용성(Availability)**과 **복원성(Resilience)**의 정의를 명확히 구분해야 합니다(, 2022(47)).
- 시스템의 신뢰성을 표현하는 척도로 **POFOD(probability of failure on demand)**의 개념을 알아야 합니다(, 2022(46)).
- **ISO/IEC 5055**에서 제시하는 품질 특성으로 Maintainability, Security, Reliability 등을 알아야 합니다(, 2024(39)).
- **사이클로매틱 복잡도(Cyclomatic Complexity)**를 함수의 논리 흐름 그래프를 보고 계산할 수 있어야 합니다(, 2022(29),, 2024(42)).
- **Phase Containment Effectiveness (PCE)**는 개발 단계에서 결함이 생성되고 발견되는 효율성을 나타내는 지표입니다(, 2023(41)).
- **ISO/IEC 25010의 신뢰성**에 대한 척도를 이해해야 합니다 [2025(31)].
- **McCabe의 Complexity Number**를 의사 코드를 보고 계산할 수 있어야 합니다 [2025(39)].
- **AI 시스템의 신뢰성(Trustworthiness)** 속성인 **AI Resilience**의 개념을 이해해야 합니다 [2025(41)].

#### 8. 형상관리

- **소프트웨어 버전 관리 시스템**인 Git과 Subversion의 특징을 비교하여 이해합니다(, 2021(40)).
- **베이스라인(Baseline)**은 시스템을 구성하는 컴포넌트 버전들의 집합이자 특정 시점이나 목적을 위해 만들어진 산출물의 집합입니다(, 2021(40)).
- **Git 명령어**의 사용법을 숙지합니다.
    - `git push`: 로컬 저장소의 변경 사항을 원격 저장소로 보냅니다(, 2023(42)).
    - `git fetch`: 원격 저장소의 모든 내용을 로컬 저장소로 복사합니다(, 2023(42)).
    - `git pull`: 원격 저장소의 최신 내용을 가져오면서 병합합니다(, 2023(42)).
    - `git clone`: 원격 저장소를 로컬로 복제합니다(, 2023(42)).
    - **브랜치(branch)**: 커밋을 가리키는 포인터(, 2022(35)).
    - **마스터(master)**: 최초 커밋 시 생성되며 가장 최근 브랜치를 가리킵니다(, 2022(35)).
    - `git stash`: 작업 내용을 임시 저장할 때 사용합니다(, 2022(35)).
    - `git checkout`: HEAD 포인터를 특정 브랜치로 이동시킵니다(, 2024(29)).
- **소프트웨어 형상 관리 활동**(Configuration Status Accounting, Configuration Identification, Configuration Verification, Configuration Control)의 내용을 이해해야 합니다(, 2023(49)).

#### 9. 요구공학

- **요구사항 도출 기법** 중 **문화기술적 연구(ethnography)**는 사용자의 일상 업무를 관찰하고 실제 작업을 기록하여 숨겨진 요구사항을 발견하는 데 도움을 줍니다(, 2021(37)).
- **요구사항 분류**(전환, 비즈니스, 비기능적, 이해관계자 요구사항)를 구분할 수 있어야 합니다(, 2022(3)).
- **요구사항 품질 속성**에는 성능 효율성, 기능 적합성, 가용성, 테스트 용이성 등이 있습니다(, 2021(47)).
- **프로토타이핑(Prototyping)**은 고객이 요구사항을 구체적으로 파악하지 못할 때, 시스템의 일부를 빠르게 구현하여 피드백을 조기에 받는 데 적합한 요구사항 추출 방법입니다(, 2024(44)).
- **SysML의 Requirement Diagram**이 표현하는 정보를 이해해야 합니다 [2025(32)].
- **요구사항 검증 과정**에서 수행하는 점검 항목과 그에 대한 설명을 연결할 수 있어야 합니다 [2025(43)].

    - Java의 `static` 메서드 내 인스턴스 변수/메서드 접근(, 2021(32)).
    - 이진 탐색 알고리즘의 동작(, 2022(30)).
