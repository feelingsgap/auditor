# 2.1 SW 공학 개념
- 기출 출제 0건 — 개요 참고용
    - SDLC (Software Development Life Cycle, 소프트웨어 개발 생명주기): 요구분석·설계·구현·테스트·유지보수로 이어지는 개발 주기 모델. 폭포수·V-모델·프로토타이핑·나선형·진화형·통합 프로세스(UP) 등으로 구분된다.
    - Waterfall Model (폭포수 모델): 요구→설계→구현→테스트→운영을 순차로 내려가는 선형 모델. 단계 간 산출물 승인 기반으로 진행되며 변경 수용이 어렵다.
    - V-Model (V-모델): 개발 단계별 테스트를 대칭 배치하여 요구분석↔인수, 설계↔시스템/통합, 구현↔단위 테스트가 짝을 이룬다.
    - Spiral Model (나선형 모델): 계획·위험분석·개발·평가를 반복하는 위험 중심 모델. 대규모·고위험 시스템에 적합.
    - UP (Unified Process, 통합 프로세스): Inception→Elaboration→Construction→Transition 4단계 반복 개발. RUP가 대표 구현체.
    - TDD (Test-Driven Development, 테스트 주도 개발): Red(실패 테스트)→Green(통과 코드)→Refactor(구조 개선)의 짧은 사이클을 반복하는 개발 철학.
    - Software Crisis (소프트웨어 위기): 하드웨어 발전 대비 소프트웨어 개발 생산성·품질·납기 한계로 드러난 구조적 문제. SW 공학이 태동한 배경.

# 2.2 SW 개발 방법론
- 애자일 프레임워크
    - XP (eXtreme Programming, 익스트림 프로그래밍) 핵심 실무
        - Simple Design (단순 설계): 현재 필요한 기능만 구현하고 과도한 사전 설계를 지양한다. YAGNI (You Aren't Gonna Need It) 원칙을 따르며 "미래 변경 대비 설계"는 XP 철학에 반한다.
        - Collective Ownership (공동 소유권): 팀 전체가 코드에 공동 책임을 지며 누구든 어떤 코드라도 수정할 수 있다. 코드에서 문제를 발견하면 원저자가 아니어도 직접 수정 가능.
        - Continuous Integration (연속적 통합): 작업이 끝나는 즉시 전체 시스템에 통합하고 모든 테스트를 통과해야 한다.
        - Refactoring (리팩토링): 코드 개선 사항 발견 시 지속적으로 구조를 개선해 단순성과 유지보수성을 유지한다.
        - On-site Customer (온사이트 고객): 고객이 개발팀에 상주하며 시나리오를 직접 토론하고 피드백을 제공한다.
        - Pair Programming (페어 프로그래밍): 두 개발자가 한 대의 컴퓨터에서 공동 작업하며 즉시 코드 리뷰가 이루어진다.
    - Scrum (스크럼)
        - Product Backlog (제품 백로그): 해야 할 일의 목록으로 소프트웨어 요구사항을 포함하며 제품 책임자(PO)가 우선순위를 관리한다.
        - Sprint (스프린트): 1~4주(보통 2~4주) 길이의 반복 개발 주기. 스프린트 계획→데일리 스크럼→스프린트 리뷰→스프린트 회고로 구성된다.
        - Scrum Master (스크럼 마스터): 팀이 프로세스를 따르도록 지원하는 서번트 리더·퍼실리테이터. 전통적 프로젝트 관리자와는 다르다.
        - Development Team (개발팀): 3~9명(PO·SM 제외) 규모 권장. 자기 조직화 기반으로 스프린트 목표를 달성한다.
        - Planning Poker (플래닝 포커): 피보나치 수열 카드를 동시 공개→최솟값·최댓값 설명→토론→합의 도출 방식의 상대적 스토리 포인트 추정 기법. 동시 공개는 앵커링 편향 방지 목적.
        - Sprint vs RUP 혼동 주의: Inception→Elaboration→Construction→Transition은 RUP 4단계이지 Scrum Sprint cycle이 아니다.
    - 애자일 품질 관리 특성
        - 집단적 코드 소유권, 지속적 리팩토링, TDD, 페어 프로그래밍 기반으로 결함을 실시간으로 제거한다.
        - 공식적 인스펙션·문서 중심 정형 방식은 폭포수적 접근이며 애자일과 거리가 멀다.
        - ISO 9001 같은 표준 기반 프로세스 준수는 전통적 품질관리이며 애자일 철학과는 다르다.
- 애자일 성과 지표
    - Velocity (속도): 스프린트마다 팀이 완료하는 작업량(스토리 포인트 합). 향후 스프린트 계획과 전체 진척 비교의 기준.
    - Burndown Chart (번다운 차트): X축 시간, Y축 남은 작업량. 스프린트 종료까지 0을 향해 내려가는 추세로 완료 예측에 사용.
    - Burnup Chart (번업 차트): X축 시간, Y축 완료된 작업량. 완료량이 누적 상승하며 스코프 변경을 시각화하기에 유리.
    - Backlog 진척률: 스프린트 내 완료된 백로그 항목들의 스토리 포인트 합으로 진행 상황을 측정.
- 전통·컴포넌트 방법론
    - CBD (Component-Based Development, 컴포넌트 기반 개발): 재사용 가능한 컴포넌트를 조립해 시스템을 구축하는 방법론.
    - Burtler CBD 성숙 6단계 (출제): GUI 콘트롤 이용 → 구현 개념 이용 → 비즈니스 개념 이용 → 컴포넌트 재사용 → 기업 표준 → CBD 성숙. 컴포넌트 활용 범위가 구현 수준에서 기업 수준으로 확장된다.
    - 구조적 방법론·정보공학 방법론·객체지향 방법론: 전통적 개발 패러다임의 주요 계열로 각각 프로세스·데이터·객체 중심의 분석·설계 접근을 제공한다.
- MSA 설계 절차
    - MSA (Microservice Architecture, 마이크로서비스 아키텍처): 독립 배포 가능한 소규모 서비스들로 시스템을 구성하는 아키텍처 스타일.
    - 설계 순서: 목표 식별(신뢰성·보안 등 달성 목표) → 원칙·정책 정의(의사결정 기준·제약) → 핵심 설계 스케치(개발조직·서비스 구성) → 현황 진단·영향 분석(작은 변경 수용 가능성 평가)의 4단계 순서로 진행한다.

# 2.3 계획
- 비용 산정 모델
    - COCOMO II (Constructive Cost Model II, 건설적 비용 모델 II): Boehm이 2000년에 발표한 소프트웨어 비용 추정 모델. 규모 인자(Scale Factors)로 지수를 결정한다.
    - Scale of Economy (규모의 경제): 지수가 1 미만이면 규모 증가 시 단위 노력이 감소(경제), 1 초과이면 불경제. COCOMO II는 규모 인자에 따라 지수가 달라진다.
    - COCOMO II 재사용 모델 AAF (Adaptation Adjustment Factor, 적응 조정 인자): AAF = 0.4×DM + 0.3×CM + 0.3×IM. 설계 변경률(DM)의 가중치가 코드 변경률(CM)·통합 노력률(IM)보다 높다.
    - DM (Design Modification), CM (Code Modification), IM (Integration Modification): 설계/코드/통합 각각의 변경률. DM·CM이 모두 0이면 이해도(SU)도 0이 된다.
    - IFPUG FPA (Function Point Analysis, 기능점수 분석): 기능 관점에서 소프트웨어 규모를 정량화하는 방법. IFPUG (International Function Point Users Group)가 표준화한다.
    - DFP (Development Function Point, 개발 기능점수): 신규 개발 프로젝트의 FP.
    - EFP (Enhancement Function Point, 개선 기능점수): 개선 프로젝트의 FP. 공식은 EFP = [(ADD + CHGA + CFP) × VAFA] + (DEL × VAFB). 추가·수정·변환은 VAFA, 삭제는 VAFB로 각각 별도 조정.
    - AFP (Application Function Point, 애플리케이션 기능점수): 운영 중인 애플리케이션 전체의 FP.
    - ADD / CHGA / DEL / CFP: 추가/수정 후/삭제/데이터 변환된 기능의 미조정 FP.
    - VAFA / VAFB (Value Adjustment Factor After/Before): 개선 이후/이전 상태의 값 조정 인자.
    - EI / EO / EQ (External Input/Output/inQuiry, 외부 입력/출력/조회): IFPUG 트랜잭션 유형. 복잡도 가중치는 Low/Avg/High 기준으로 구분된다.
- 일정 관리
    - Gantt Chart (간트 차트, 바차트): 작업과 기간을 가로 막대로 표시하는 일정 표기법. 마일스톤·여유 시간 시각화에 활용된다.
    - Critical Path (임계경로): 프로젝트 시작에서 종료까지 가장 긴 작업 경로로 Float(여유 시간)가 0인 작업의 연속. 임계경로 작업이 지연되면 프로젝트 전체가 지연된다.
    - CPM (Critical Path Method, 임계경로법): ES·EF·LS·LF를 계산해 Float = LS-ES = LF-EF로 임계경로를 식별하는 기법.
    - Float / Slack (여유 시간): 프로젝트 종료 지연 없이 작업이 지연될 수 있는 시간. 0인 작업이 임계경로.
    - PERT (Program Evaluation and Review Technique, 일정 평가·검토 기법): 낙관·비관·최빈 3점 추정으로 작업 기간을 확률적으로 산정.
    - Crashing (크래싱): 자원을 추가 투입해 임계경로 작업 기간을 단축하는 일정 압축 기법.
    - Fast Tracking (패스트 트래킹): 순차 작업을 병렬로 수행해 기간을 단축하는 기법.
- 위험 관리
    - 위험 관리 절차: 위험 식별 → 위험 분석·평가 → 위험 대응 계획 → 위험 모니터링 및 통제.
    - 위험 노출도 (Risk Exposure): 위험 발생 확률 × 영향도. 위험 우선순위 결정의 기본 지표.
    - 위험 대응 전략: 회피(avoid)·전가(transfer)·완화(mitigate)·수용(accept).

# 2.4 요구분석
- 요구사항 도출 기법
    - Ethnography (문화기술적 연구, 민족지학 관찰): 분석가가 사용자의 실제 업무 환경에 장기간 참여·관찰·기록하여 암묵적 요구사항을 발견한다. 공식 인터뷰로 드러나지 않는 실제 작업 방식을 파악하는 데 효과적이나 시간·비용 부담이 크다.
    - Prototyping (프로토타이핑): 불완전하지만 동작하는 시제품을 빠르게 제시→사용자 피드백 수집→요구사항 구체화를 반복한다. 고객이 요구를 명확히 모를 때, 초기 기능의 빠른 검증이 필요할 때, 아이디어 검증을 통한 새 아이디어 발굴 시 모두 적용 가능.
    - Horizontal Prototype (수평 프로토타입): UI 전반을 얇게 구현해 전체 흐름을 미리 보여준다.
    - Vertical Prototype (수직 프로토타입): 특정 기능을 심층 구현해 기술적 타당성을 검증한다.
    - Interview (인터뷰): 이해관계자와 대화로 요구를 수집하는 기본 기법.
    - User Story (사용자 스토리): "사용자로서 ~하고 싶다"는 짧은 문장으로 애자일에서 요구를 표현.
    - Brainstorming (브레인스토밍): 자유로운 아이디어 발산으로 새 요구를 발굴.
    - Observation (관찰): 사용자 작업 환경을 직접 관찰해 암묵적 요구를 파악.
    - Use Case (유스케이스): 시스템과 사용자 간 상호작용을 기술한 문서로, 시스템의 기능을 정의한다.
    - Scenario (시나리오): Use Case의 구체적인 실행 흐름을 기술한 문서.
    - Workshop (워크숍): 이해관계자들이 모여 요구사항을 논의하고 도출하는 기법.
- 요구사항 검증 4대 기준 (Sommerville)
    - Validity (유효성): 요구사항이 사용자의 실제 요구를 반영하는지 점검한다.
    - Consistency (일관성): 요구사항이 서로 상충하지 않고 모순된 제약조건이 없는지 점검한다.
    - Completeness (완전성): 사용자가 의도한 모든 기능과 제약조건이 빠짐없이 명세되었는지 점검한다.
    - Feasibility (실현성): 현재 기술·지식·예산으로 실제 구현이 가능한지 점검한다.
    - Verification vs Validation: 검증(Verification)은 "제대로 만들고 있는가", 확인(Validation)은 "올바른 제품을 만들고 있는가"로 구분된다.
- 요구사항 유형·연관관계
    - Business Requirement (비즈니스 요구사항): 조직 목표 관점의 최상위 요구. 사용자 요구사항의 원천.
    - User Requirement (사용자 요구사항): 이해관계자 관점에서 시스템이 수행해야 할 내용. 기능 요구사항의 원천.
    - Functional Requirement (기능 요구사항): 시스템이 제공해야 할 기능·동작. 시스템 요구사항으로 전개된다.
    - Non-functional Requirement (비기능 요구사항): 성능·신뢰성·보안 등 품질 속성. 기능 요구사항에 제약으로 작용.
    - Constraint (제약사항): 기술·법률·예산 등 기능 요구사항 구현을 제한하는 조건으로 기능 쪽으로 화살표가 향해야 한다.
    - Business Rule (비즈니스 룰): 업무 규칙이 기능 요구사항에 반영된다.
- 요구사항 패턴 (Withall)
    - Transaction Requirement Pattern (트랜잭션 요구 패턴): 특정 트리거에 의한 처리 행위에 대한 요구. 고유 ID 자동 부여·제한된 접근 제어가 특징(예: 계좌 금액 조정).
    - Chronicle Requirement Pattern (연대기 요구 패턴): 시간 순서에 따른 이력 기록 요구.
    - Living Entity Requirement Pattern (생명 엔티티 요구 패턴): 엔티티의 전체 생명주기(생성·조회·수정·삭제) 요구.
    - Configuration Requirement Pattern (구성 요구 패턴): 시스템 설정·구성 관련 요구.
- SysML Requirement Diagram
    - SysML Requirement Diagram (요구사항 다이어그램): 요구사항과 다른 모델 요소 간 관계를 시각화. 요구사항 계층 구조(ContainReqt 포함 관계), 다른 모델 요소와의 관계(Satisfy·Refine·Verify·DeriveReqt), 개별 요구사항 표현 3가지가 주된 역할.
    - 다이어그램 경계 구분: 입·출력·처리 흐름은 Activity Diagram이, 시스템 특성의 수식 제약은 Parametric Diagram이 담당한다(Requirement Diagram 역할 아님).
- ISO/IEC 25010 품질 속성 매핑
    - Capacity (용량성): 동시 사용자 수 수용 등 규모 한계. Performance Efficiency의 세부 특성.
    - Functional Suitability (기능 적합성): 요구 기능을 제공하는 능력.
    - Reliability (신뢰성): 다운타임 제한 등. Availability·Recoverability·Maturity·Fault Tolerance의 상위 특성.
    - Availability (가용성): 필요 시점에 서비스가 작동·접근 가능한 능력.
    - Testability (시험용이성): 테스트 기준 수립과 실행의 용이성.

# 2.5.1 객체지향 설계 원칙·디자인 패턴
- SOLID 원칙
    - SRP (Single Responsibility Principle, 단일 책임 원칙): 하나의 클래스는 변경 이유가 하나뿐이어야 한다. 두 가지 이상의 역할이 섞이면 각 역할별로 클래스를 분리한다.
    - OCP (Open-Closed Principle, 개방-폐쇄 원칙): 확장에는 열려 있고 수정에는 닫혀 있어야 한다. 추상 인터페이스를 통해 새 구현체를 추가하면 기존 코드를 건드리지 않는다.
    - LSP (Liskov Substitution Principle, 리스코프 치환 원칙): 하위 타입 객체는 상위 타입 변수에 대입해도 프로그램이 올바르게 동작해야 한다. is-a 관계를 행동 수준까지 보장한다.
    - ISP (Interface Segregation Principle, 인터페이스 분리 원칙): 클라이언트는 자신이 사용하지 않는 메서드에 의존하지 않아야 한다. 거대한 인터페이스 대신 역할별로 작은 인터페이스를 분리한다.
    - DIP (Dependency Inversion Principle, 의존성 역전 원칙): 상위·하위 모듈 모두 추상(인터페이스·추상 클래스)에 의존해야 하며 구체 클래스 직접 참조 시 위반이다. 5년간 가장 많이 출제된 원칙.
- OOP 기초 — 다형성
    - 동일 메서드명으로 서로 다른 타입 객체가 각자의 구현을 실행하는 성질이다.
    - 상위 타입 변수에 하위 객체를 대입한 뒤 메서드를 호출하면 실제 객체의 오버라이딩된 구현이 런타임에 결정된다(동적 디스패치).
    - 컴파일 타임에 시그니처로 구분되는 오버로딩(Overloading)과는 구분된다.
- GoF 디자인 패턴
    - 행위 패턴 (기출) — 객체 간 책임 분배·상호작용을 다루는 패턴
        - Strategy (전략 패턴): 알고리즘군을 캡슐화해 런타임에 교체 가능하게 만든다. Context가 Strategy 인터페이스에 의존하고 ConcreteStrategy를 주입받아 교체한다(장기 게임의 알고리즘 선택이 대표 예).
        - Command (명령 패턴): 요청을 객체로 캡슐화해 Invoker-Command-Receiver를 분리한다. Button→LightOnCommand→ILamp 구조로 실행·취소·큐잉을 지원한다.
        - Chain of Responsibility (책임 연쇄 패턴): 요청을 처리 가능한 핸들러까지 체인으로 전달한다. hasHandler() 조건으로 처리 여부를 판별하며 불가능하면 다음 핸들러로 위임한다.
        - State (상태 패턴): 내부 상태별로 행동 클래스를 분리하여 if/else 조건 분기를 제거한다. 상태 전환 시 상태 객체를 교체함으로써 동작이 바뀐다.
    - 행위 패턴 (학습 보강)
        - Observer (관찰자 패턴): Subject의 상태가 변경되면 등록된 Observer 전체에 자동으로 알린다. 발행-구독 구조의 기본 형태다.
        - Template Method (템플릿 메서드 패턴): 알고리즘 골격을 부모 클래스에 고정하고 변하는 일부 단계만 하위 클래스가 오버라이드한다.
        - Iterator (반복자 패턴): 컬렉션 내부 구조를 노출하지 않고 hasNext()/next() 인터페이스로 순회를 캡슐화한다.
    - 구조 패턴 (기출) — 클래스·객체를 조합해 더 큰 구조를 만드는 패턴
        - Decorator (데코레이터 패턴): 상속 없이 객체에 동적으로 기능을 추가한다. Component 인터페이스를 구현하면서 동시에 동일 인터페이스 타입 필드를 보유하는 것이 클래스 다이어그램 식별 핵심이다.
        - Adapter (어댑터 패턴): 호환되지 않는 인터페이스를 변환해 기존 클래스를 재사용한다. Adaptee를 Target 인터페이스로 감싸며 Command 패턴과 복합 구조로도 출제된다.
    - 구조 패턴 (학습 보강)
        - Bridge (브리지 패턴): 추상화(Abstraction)와 구현(Implementor)을 분리하여 각각 독립적으로 확장 가능하게 한다.
        - Proxy (프록시 패턴): 실제 객체 대신 대리가 접근을 제어한다. 보호(권한 확인)·원격(원격 객체 접근)·가상(지연 초기화) 프록시의 3종이 대표적이다.
        - Facade (퍼사드 패턴): 복잡한 서브시스템에 단순한 단일 창구 인터페이스를 제공하여 클라이언트가 내부 복잡성을 알지 않아도 된다.
    - 생성 패턴 (학습 보강) — 객체 생성 과정을 추상화·캡슐화하는 패턴
        - Singleton (싱글턴 패턴): 클래스 인스턴스를 하나로 제한하고 전역 접근점을 제공한다. 멀티스레드 환경의 동기화 처리가 주의 사항이다.
        - Factory Method (팩토리 메서드 패턴): 객체 생성을 하위 클래스가 결정하도록 위임한다. Creator가 팩토리 메서드를 선언하고 ConcreteCreator가 구체 Product를 반환한다.
        - Abstract Factory (추상 팩토리 패턴): 관련 객체군(예: UI 테마의 버튼+체크박스 세트)을 일관되게 생성하는 인터페이스를 제공한다.
        - Builder (빌더 패턴): 복잡한 객체를 단계적으로 조립하며 Director가 Builder 인터페이스를 통해 조립 순서를 제어하고 getResult()로 최종 객체를 반환한다.
- Software Product Line
    - 공통 아키텍처와 공유 컴포넌트를 바탕으로 다양한 고객 요구에 맞는 제품군을 관리하는 방법론이다.
    - 도메인 엔지니어링(공통 자산 개발)과 애플리케이션 엔지니어링(개별 제품 조립)으로 구성된다.
- Clean Architecture
    - 의존성 방향
        - Entities → Use Cases → Interface Adapters → Frameworks/Drivers 순으로만 의존하는 단방향 구조다.
        - 내부 계층은 외부 계층을 알지 못하며, 외부 계층이 내부에 의존한다.
        - 비즈니스 규칙이 프레임워크·DB·UI에 종속되지 않는 것이 핵심 가치다.
    - 컴포넌트 결합 3원칙
        - ADP (Acyclic Dependencies Principle, 비순환 의존 원칙): 컴포넌트 간 의존성 그래프에 순환이 없어야 한다. 순환이 생기면 릴리즈·빌드·테스트가 뒤엉킨다.
        - SDP (Stable Dependencies Principle, 안정 의존 원칙): 자신보다 안정적인(Fan-in이 높아 변경이 적은) 컴포넌트에 의존해야 한다. 불안정한 쪽이 안정 쪽을 참조하는 구조.
        - SAP (Stable Abstractions Principle, 안정 추상 원칙): 안정적인 컴포넌트일수록 추상화 수준이 높아야 한다(구체 클래스 비중이 낮아야 한다). 변경 영향 최소화 목적.
        - Instability(불안정도) 계산식: I = Fan-out / (Fan-in + Fan-out). 0에 가까울수록 안정, 1에 가까울수록 불안정.

# 2.5.2 소프트웨어 아키텍처·평가
- 아키텍처 스타일
    - Pipe and Filter (파이프와 필터): 데이터가 파이프를 통해 흐르며 각 필터 단계에서 변환된다. 필터는 독립·재사용 가능하고 대표 사례는 Unix 파이프라인·컴파일러·ETL 파이프라인.
    - MVC (Model-View-Controller, 모델-뷰-컨트롤러): UI(View)를 비즈니스 로직(Model)과 분리하고 Controller가 입력을 처리한다. 하나의 Model에 다수 View 연결이 가능해 View 간 결합도를 최소화한다. 파생 패턴은 MVP·MVVM.
    - Publish-Subscribe (발행-구독): Publisher가 이벤트를 발행하면 Subscriber가 수신하며 Event Distributor(분배자)가 중간에서 연결한다. 컴포넌트 간 느슨한 결합을 제공.
    - Master-Slave (마스터-슬레이브): Master가 작업을 분할(split)→여러 Slave에 위임→병렬(par) 실행→결과 수집·합산(merge)하는 구조. 시퀀스 다이어그램에서 split·par·merge 조합이 식별자.
    - Broker (브로커 패턴): 분산 시스템에서 중간자 Broker가 서비스 탐색·요청 전달·결과 반환을 담당한다. 위치 투명성·modifiability·availability·performance 향상에 기여.
    - Blackboard (블랙보드): 여러 지식 소스가 공유 저장소(블랙보드)를 통해 협력하는 구조. AI·음성 인식 등 비결정적 문제에 활용.
    - Microkernel (마이크로 커널): 핵심 기능만 커널에 두고 나머지를 플러그인으로 확장하는 plug-and-play 구조.
    - Reflection (리플렉션): 시스템의 구조·행동 메타 정보를 메타 객체에 분리 저장하여 런타임 적응을 가능하게 한다.
    - Peer-to-Peer (피어투피어): 동등한 노드들이 직접 통신하는 분산 구조.
    - Layered (계층형): 상위 계층이 하위 계층만 사용하는 계층적 분리 구조.
    - Client-Server (클라이언트-서버): 클라이언트 요청→서버 응답 구조.
- SOA·웹 서비스
    - SOAP (Simple Object Access Protocol, 단순 객체 접근 프로토콜): XML 봉투(envelope) 기반 메시지 교환 프로토콜. WS-* 표준(보안·트랜잭션·신뢰성) 지원으로 엔터프라이즈 환경에 적합하지만 복잡하다.
    - WSDL (Web Services Description Language, 웹 서비스 기술 언어): 웹 서비스의 오퍼레이션·메시지 형식·바인딩 등 인터페이스를 XML로 기술한다.
    - WS-BPEL (Web Services Business Process Execution Language, 비즈니스 프로세스 실행 언어): 여러 웹 서비스를 조합한 비즈니스 프로세스(워크플로우)를 정의하는 언어.
    - REST (Representational State Transfer, 표현 상태 전이): HTTP 메서드(GET/POST/PUT/PATCH/DELETE) 기반의 자원 지향 아키텍처 스타일. JSON·XML·텍스트 등 다양한 포맷을 지원하며 경량·확장성이 우수.
    - HTTP 표준 메서드: GET(조회)·POST(생성)·PUT(전체 수정/대체)·PATCH(부분 수정)·DELETE(삭제). "READ"·"UPDATE"·"CREATE"는 CRUD 개념이지 HTTP 메서드가 아니다.
    - SOAP vs REST: SOAP은 XML 전용·서비스 지향(RPC 스타일)·구현 복잡, REST는 다양한 포맷·자원 지향·구현 단순·높은 확장성.
- Microservice Architecture
    - MSA 핵심 원칙: 독립 배포 가능한 소규모 서비스 + 느슨한 결합 + 서비스별 독립 DB(Database per Service 패턴).
    - 결함 격리 (Fault Isolation): 특정 컴포넌트 장애가 애플리케이션 전체 중단으로 이어지지 않도록 격리한다.
    - Anti-pattern: 단일 데이터베이스 중심 통합은 모놀리식 특성이며 MSA 원칙에 반한다.
    - Orchestration (오케스트레이션): 중앙 조정자가 서비스 호출 순서를 제어.
    - Choreography (코레오그래피): 이벤트 기반으로 각 서비스가 자율적으로 협력.
- 응집도·결합도
    - Cohesion (응집도, 높은 순): 기능적(Functional) > 순차적(Sequential) > 통신적(Communicational) > 절차적(Procedural) > 시간적(Temporal) > 논리적(Logical) > 우연적(Coincidental).
    - Functional Cohesion (기능적 응집도): 하나의 문제만 수행하는 함수. 가장 높은 응집도.
    - Sequential Cohesion (순차적 응집도): 한 기능의 출력이 다음 기능의 입력이 되는 경우.
    - Communicational Cohesion (통신적 응집도): 동일한 데이터·속성을 사용하는 기능끼리 묶인 경우.
    - Procedural Cohesion (절차적 응집도): 일련의 절차를 수행하는 기능 묶음.
    - Temporal Cohesion (시간적 응집도): 시간적으로 연관(초기화·종료 등)된 기능 묶음.
    - Logical Cohesion (논리적 응집도): 논리적으로 연관된 기능 묶음.
    - Coincidental Cohesion (우연적 응집도): 서로 연관 없는 기능을 묶은 경우. 가장 낮은 응집도.
- 아키텍처 평가 방법
    - SAAM (Software Architecture Analysis Method, 소프트웨어 아키텍처 분석 방법): 직접/간접 시나리오 기반 아키텍처 평가. 비용 계산 절차는 없다.
    - ATAM (Architecture Tradeoff Analysis Method, 아키텍처 상충 분석법): 품질 속성 간 트레이드오프를 분석하여 위험·비위험·감도점·트레이드오프점을 도출하는 SEI 방법론.
    - ARID (Active Review for Intermediate Designs, 중간 설계 능동 검토): 설계 초기 단계의 능동적 검토.
    - CBAM (Cost Benefit Analysis Method, 비용-편익 분석법): ATAM을 보완해 품질속성 시나리오별 효용값·비용을 산정하고 ROI 기반으로 전략을 선정. 절차는 시나리오 선정→효용값 부여→비용 산정→ROI 계산→전략 선택.
- SEI 아키텍처 뷰 모델
    - Module View (모듈 뷰): 코드 단위 정적 구조(계층·부분·의존 등).
    - C&C View (Component & Connector, 컴포넌트-커넥터 뷰): 런타임 구조. Publish-Subscribe·Pipe and Filter·Client-Server 등이 이 뷰에 속한다.
    - Allocation View (할당 뷰): 소프트웨어와 환경(하드웨어·개발조직) 간 매핑.
- Clean Architecture 계층
    - 내부 → 외부 순서: Entities(핵심 비즈니스 규칙) → Use Cases(애플리케이션 비즈니스 규칙) → Interface Adapters(컨트롤러·프레젠터·게이트웨이) → Frameworks and Drivers(웹 프레임워크·DB 등 세부 구현).
    - Dependency Rule (의존성 규칙): 의존성은 항상 안쪽(Entities 방향)을 향하며 내부 계층은 외부를 알지 못한다. 비즈니스 로직을 UI·DB·프레임워크로부터 독립시키는 것이 핵심 가치.

# 2.5.3 UML·SysML·설계 표기법
- UML Class Diagram
    - 가시성 표기: private(-), public(+), protected(#).
    - Static 멤버: UML에서 밑줄(underline)로 표기하며 Java에서는 static 키워드에 대응.
    - readOnly 속성: UML에서 {readOnly}로 표기하며 Java에서는 final 키워드에 대응.
    - Generalization (일반화, 상속): 실선 + 빈 삼각형(▷) 화살표로 부모 클래스를 향한다. Java의 extends에 해당.
    - Realization (실현, 인터페이스 구현): 점선 + 빈 삼각형(▷) 화살표로 인터페이스를 향한다. Java의 implements에 해당. extends로 구현하면 오류.
    - Association (연관): 실선 + 열린 화살표(→). 지속적인 구조적 관계를 표현.
    - Dependency (의존): 점선 + 열린 화살표(→). 메서드 내부에서 다른 클래스를 사용하는 일시적 관계.
    - Composition (합성): 속이 찬 마름모. 강한 포함 관계로 생명주기가 종속된다.
    - Aggregation (집합): 속이 빈 마름모. 약한 포함 관계.
    - Multiplicity (다중성): 1, *, 0..1, 1..* 등 연관 관계의 수를 표기. 시퀀스 다이어그램에서 for each로 컬렉션을 순회하면 클래스 다이어그램에도 다중성이 반영되어야 한다.
- UML Sequence Diagram
    - Combined Fragment (결합 프래그먼트): loop(반복)·alt(선택)·opt(조건)·par(병렬)·break(탈출) 등 제어 흐름 표기.
    - par 블록: 병렬 실행을 표현. split·merge와 함께 나타나면 Master-Slave 패턴의 식별자.
    - 아키텍처 패턴 판별: 시퀀스 다이어그램의 메시지 흐름(pack/forward/find server/call service 등)으로 Broker·Master-Slave·Front Controller·MVC 구분.
- UML State Machine Diagram
    - 전이 표기법: trigger [guard] / effect. trigger는 이벤트, [guard]는 조건식, /는 구분자, effect는 수행 동작. guard·effect는 생략 가능.
    - 복합 상태 → 외부 전이 순서: 내부 하위 상태 exit → transition action → 복합 상태 exit → 목적지 entry 순서로 실행된다.
- UML Use Case Diagram
    - «include»: 기본 유스케이스가 포함 유스케이스를 반드시 실행. 화살표는 기본→포함 방향.
    - «extend»: 특정 조건에서만 기본 유스케이스를 확장. 화살표는 확장 유스케이스 → 기본 유스케이스 방향(확장하는 쪽에서 출발).
    - 핵심 구분: include는 항상 실행(필수), extend는 조건부 실행(선택).
- UML Activity Diagram
    - Activity Diagram (활동 다이어그램): 작업 흐름·제어 흐름·병렬 처리·동기화를 표현.
    - Fork / Join (포크·조인): 병렬 수행 분기·합류 표기.
    - Swimlane (스윔레인): 액터별 책임 구분을 시각적으로 표현. 시스템 수준에서 액터-시스템 상호작용 모델링 가능.
    - 적용 시나리오: 액터 관점 시스템 모델링과 제어 흐름·병렬 수행 파악을 동시에 지원하는 다이어그램.
- SysML 다이어그램
    - SysML (Systems Modeling Language): OMG가 UML 2.x를 기반으로 하드웨어 등 복잡한 시스템 모델링을 위해 정의한 언어. 9개 다이어그램으로 구성.
    - UML 공통 다이어그램: Use Case, Sequence, State Machine, Activity. 그대로 SysML에 채택.
    - Class Diagram은 SysML에서 Block Definition Diagram(BDD)으로 대체된다.
    - Requirement Diagram (요구사항 다이어그램): SysML 고유 다이어그램. 요구사항 계층·관계 시각화.
    - Block Definition Diagram (BDD, 블록 정의 다이어그램): 블록의 구조적 계층·속성 정의. UML Class의 대체.
    - Internal Block Diagram (IBD, 내부 블록 다이어그램): 블록 내부 부분과 연결 관계 정의.
    - Parametric Diagram (파라메트릭 다이어그램): 값 속성 간 수학적·물리적 제약을 Constraint Block과 Binding Connector로 표현. 시스템 변수의 유효 범위 분석과 예외 상황 탐지, 성능·신뢰성·트레이드오프 정량 분석에 활용된다.
- Feature Diagram
    - Mandatory (필수, 채운 원): 반드시 포함되는 피처.
    - Optional (선택, 빈 원): 포함 여부를 선택할 수 있는 피처.
    - Alternative/XOR (대안, 채운 삼각형): 여러 자식 중 정확히 하나만 선택.
    - OR (OR 그룹, 빈 삼각형): 여러 자식 중 하나 이상 선택.
    - 조건부 컴파일(#if defined) 코드에서 #if→#else는 XOR 관계, 상위 #if는 선택적 피처로 매핑된다.

# 2.6 구현
- 리팩토링 기법
    - 메서드 구조 조정
        - Extract Method (메서드 추출): 코드 일부를 별도 메서드로 분리한다. Inline Method의 역방향.
        - Inline Method (인라인 메서드): 메서드 본문이 이름만큼 명확할 때 호출부에 직접 삽입하고 원본 메서드를 제거한다.
        - Move Method (메서드 이동): 메서드를 실제 애용하는 데이터가 있는 클래스로 옮긴다. Feature Envy 해결책.
        - Pull Up Method (메서드 상향): 여러 서브클래스의 동일 메서드를 상위 클래스로 올린다.
        - Substitute Algorithm (알고리즘 대체): 메서드 본문 알고리즘을 더 명확한 구현으로 교체한다.
        - Rename Method (메서드 이름 변경): 이름을 더 명확하게 변경하는 기법.
    - 캡슐화·위임
        - Hide Delegate (위임 숨기기): 위임 체인(A→B→C)을 감추어 클라이언트가 중간 객체(B)에만 접근하도록 개선한다.
        - Remove Middle Man (중간자 제거): 불필요한 위임 메서드를 제거하고 직접 접근하도록 한다. Hide Delegate의 반대 방향.
        - Encapsulate Variable (변수 캡슐화): 전역 변수를 getter/setter 함수로 감싸 직접 접근을 차단한다.
        - Encapsulate Downcast (다운캐스팅 캡슐화): 다운캐스팅 코드를 메서드 내부로 숨긴다.
    - 매개변수·변수 처리
        - Split Temporary Variable (임시변수 분리): 하나의 임시변수가 여러 용도로 재사용될 때 각 용도별로 변수를 분리한다.
        - Remove Assignments to Parameters (매개변수 할당 제거): 메서드 내부에서 매개변수 값을 직접 수정하지 않고 지역변수에 복사 후 지역변수를 수정한다. 가독성과 부작용 방지를 위한 기법.
    - 오류·타입 코드 처리
        - Replace Error Code with Exception (오류 코드를 예외로 교체): 오류를 특수 반환값(-1, null)으로 표현하던 방식을 예외 객체 throw로 전환한다. 호출자가 반환값 확인을 누락해도 예외가 전파된다.
        - Replace Conditional with Polymorphism (조건문을 다형성으로 교체): if/switch 타입 분기를 다형성 구조로 대체한다.
        - Replace Type Code with Subclasses (타입 코드를 서브클래스로 교체): 타입 상수에 따른 switch 분기를 각 타입별 서브클래스로 대체해 다형성을 활용한다. 새 타입 추가 시 OCP를 충족.
        - Replace Type Code with Class (타입 코드를 클래스로 교체): 타입 코드를 별도 클래스로 객체화하되 switch 분기는 유지(열거형 대체 수준).
        - Replace Type Code with State/Strategy (타입 코드를 상태·전략으로 교체): 상태·전략 패턴으로 분리해 위임을 활용한다.
- 코드 악취(Bad Smell)와 처방 매핑
    - Feature Envy (기능 욕심): 메서드가 자신이 속한 클래스보다 다른 클래스의 데이터·기능을 더 많이 사용한다. 처방: Move Method(데이터가 있는 클래스로 이동).
    - Shotgun Surgery (산탄총 수술): 하나의 변경이 여러 클래스에 흩어진 소규모 수정을 유발한다. 처방: Move Method/Field로 흩어진 변경 지점을 한 곳에 응집.
    - Divergent Change (확산적 변경): 하나의 클래스가 서로 다른 이유로 자주 변경된다(Shotgun Surgery의 반대). 처방: 클래스 분리.
    - Data Clumps (데이터 뭉치): 항상 함께 등장하는 데이터 그룹이 별도 클래스로 캡슐화되지 않았다. 처방: Extract Class(독립 클래스 정의).
    - Switch/If Overuse (스위치·조건문 과다): 조건 분기가 여러 곳에 반복. 처방: Replace Conditional with Polymorphism.
- Java 언어 요소
    - Static 컨텍스트 제약: static 메서드 내에서는 static 변수·static 메서드만 직접 참조 가능. 인스턴스 변수·인스턴스 메서드는 인스턴스 참조 없이 직접 접근하면 컴파일 오류.
    - Anonymous Inner Class (익명 내부 클래스): 내부에서 this는 익명 클래스 자신의 인스턴스를 가리킨다(외부 클래스 인스턴스는 OuterClass.this.field로 접근).
    - Variable Shadowing (변수 섀도잉): 내부 스코프의 동명 변수가 외부 변수를 가리는 현상. 익명 클래스 필드 a와 run() 지역 변수 a가 공존 시 this.a는 필드를 참조.
- Python·C 언어 요소
    - list.pop(): 마지막 요소 제거·반환. list.pop(i)는 인덱스 i 요소 제거·반환.
    - list.insert(i, val): 인덱스 i 위치에 val 삽입(기존 요소 뒤로 밀림).
    - list.index(val): val의 첫 번째 인덱스 반환.
    - C 후위 ++ 연산자: 현재 값을 먼저 사용한 후 증가(b = **ptr++은 증가 전 ptr로 역참조).
    - int (*ptr)[4]: 4개 int 배열을 가리키는 포인터. ptr+1은 다음 4개 int 블록으로 이동.
- Spring 프레임워크
    - Spring Core Container: spring-core·spring-beans·spring-context·spring-expression(SpEL). IoC 컨테이너 구현과 DI 기반을 제공하며 모든 모듈이 의존한다.
    - IoC (Inversion of Control, 제어 역전): 객체 생성과 의존성 주입을 컨테이너가 담당.
    - AOP 어드바이스 유형
        - Before (전 어드바이스): 메서드 실행 전.
        - AfterReturning (정상 반환 후): 정상 반환이 발생했을 때만 실행.
        - AfterThrowing (예외 발생 후): 예외가 던져졌을 때만 실행.
        - After / Finally (종료 후): 정상·예외 무관하게 메서드 종료 후.
        - Around (감싸기): 메서드 호출 전후 모두 감싸며 예외 상황도 처리. 가장 강력한 어드바이스 유형.
    - Spring MVC 요청 처리 흐름
        - DispatcherServlet (디스패처 서블릿): 모든 요청의 진입점. Front Controller 패턴 구현.
        - HandlerMapping (핸들러 매핑): URL과 Controller 간의 매핑 정보 관리.
        - Controller (컨트롤러): 비즈니스 로직 처리 후 ModelAndView 반환.
        - ModelAndView (모델앤뷰): Model 데이터와 View 이름을 함께 담는 반환 객체.
        - ViewResolver (뷰 리졸버): View 이름을 실제 View 파일(JSP·Thymeleaf 등) 경로로 변환.
        - 처리 순서: DispatcherServlet → HandlerMapping → Controller → ModelAndView → ViewResolver → View.
- 재공학·레거시 시스템 포트폴리오
    - Reverse Engineering (역공학): 기존 시스템에서 설계·요구사항을 추출.
    - Restructuring (재구조화): 외부 동작은 유지하며 내부 구조를 개선.
    - Migration (이식): 다른 플랫폼·환경으로 시스템을 이관.
    - 레거시 시스템 매트릭스: X축 시스템 품질, Y축 비즈니스 가치.
    - 재공학 (좌상단): 비즈니스 가치 높음 + 품질 낮음 → 현대화 대상.
    - 유지보수 (우상단): 가치 높음 + 품질 높음 → 현 상태 유지.
    - 폐기 (좌하단): 가치 낮음 + 품질 낮음 → 제거.
    - 교체 (우하단): 가치 낮음 + 품질 높음 → 교체.
- 오픈소스 라이선스
    - GPL (General Public License, 일반 공중 사용 허가서): 강한 카피레프트. 파생물도 GPL로 공개해야 하며 소스코드 전체 공개 의무.
    - LGPL (Lesser GPL, 약화된 GPL): 약한 카피레프트. 라이브러리 수정 시에만 공개 의무, 링크만 하는 응용프로그램은 비공개 가능. 라이브러리 수정 시 수정사실·날짜 명기, 응용프로그램은 라이브러리 사용 사실 명시, 사용자가 라이브러리를 대체 가능하도록 허용. 정·동 링크 구분 없이 "재링크 가능성 보장"(앱 소스 또는 오브젝트 파일 제공)이 핵심 의무이며, 소스 비공개 자체는 위반 아님.
    - MPL (Mozilla Public License): 파일 단위 카피레프트. Mozilla 제품군에 사용.
    - BSD (Berkeley Software Distribution License): 허용적(permissive) 라이선스. 소스 공개 의무 없고 저작권 안내만 유지하면 재배포 가능.
- 클라우드 컨테이너
    - Container (컨테이너): 호스트 OS 커널을 공유하는 경량 격리 환경. VM과 달리 하이퍼바이저 불필요.
    - 하이퍼바이저: VM(완전 가상화)에서 사용. 컨테이너는 namespace·cgroups 기반 OS 레벨 가상화를 사용.
    - Kubernetes (쿠버네티스): 컨테이너화된 워크로드와 서비스를 관리하는 오케스트레이션 도구.
    - 제약: 컨테이너들은 호스트 OS 커널을 공유하므로 컨테이너마다 다른 OS를 사용할 수 없다.
    - 적합성: MSA에 적합. 모놀리식 아키텍처는 컨테이너의 독립 배포·확장성 이점을 충분히 활용하기 어렵다.

# 2.7 테스트
- 커버리지 체계
    - Statement Coverage (문장 커버리지): 실행된 문장 수 / 전체 문장 수. 모든 실행 가능 라인이 한 번 이상 실행되어야 만족.
    - Decision / Branch Coverage (결정·분기 커버리지): 각 결정의 True·False 분기를 모두 실행해야 만족. 문장 커버리지를 포함한다.
    - Condition Coverage (조건 커버리지): 각 개별 조건의 참·거짓을 모두 실행. 결정 커버리지를 보장하지는 않음(독립 관계).
    - Decision/Condition Coverage (결정/조건 커버리지): 결정 커버리지와 조건 커버리지를 동시에 충족.
    - MC/DC (Modified Condition/Decision Coverage, 변형 조건·결정 커버리지): 각 조건이 전체 결정에 독립적으로 영향을 미침을 보여야 한다. 조건 N개일 때 최소 N+1개 테스트로 달성 가능.
    - Multiple Condition Coverage (다중 조건 커버리지): 모든 조건 조합을 실행. 가장 강한 커버리지.
    - 포함 관계 (약→강): Statement ⊂ Decision ⊂ Decision/Condition ⊂ MC/DC ⊂ Multiple Condition. 안쪽 집합을 만족하면 바깥쪽도 자동 만족.
- 블랙박스 테스트 기법
    - Equivalence Partitioning (동등 분할): 입력 도메인을 동등한 결과를 내는 구간으로 분할하여 대표값을 선택.
    - Boundary Value Analysis (경계값 분석): 동등 분할 경계값에서 오류가 집중됨을 이용해 경계 입력을 집중 테스트.
    - Decision Table Testing (결정 테이블 테스트): N개 조건의 조합(2^N Rule)별 동작을 표로 정리. TC가 커버하는 고유 Rule 수 / 전체 Rule 수로 커버리지 계산.
    - Cause-Effect Graphing (원인-결과 그래프): 원인(Cause)과 결과(Effect)를 AND·OR·NOT 논리 게이트로 연결해 테스트케이스를 도출. 결정 테이블로 변환해 체계적 설계 가능.
    - Cause-Effect Constraints: Exclusive(둘 중 최대 하나 참), Inclusive(적어도 하나 참), One and only one(정확히 하나 참), Requires(A가 참이면 B도 참), Masks(일부 문헌). "Mask" 단독은 29119 Part 4 Cause 간 제약 표준 목록에 부적절.
    - State Transition Testing (상태 전이 테스트): 상태 머신의 상태 변화를 이벤트로 검증.
    - Scenario Testing (시나리오 테스트): 사용자 입·출력 시퀀스 모델 기반. 시스템 테스트·인수 테스트에서 End-to-End 기능 검증에 활용.
    - Requirement-based Testing (요구 기반 테스트): 요구사항 명세를 직접 근거로 테스트케이스 도출.
- 조합 테스트 기법
    - All Combination Testing (전조합 테스트): 모든 경우의 수 완전 열거. TC 수 = 파라미터 값 수의 곱(가장 많음).
    - Pair-wise Testing (페어와이즈): 모든 2개 파라미터 쌍의 조합을 최소 1회씩 포함.
    - Base Choice Testing (기본 선택): 기본값 1세트를 정한 뒤 한 번에 하나의 파라미터만 비기본 값으로 변경. 공식 = 1 + Σ(각 파라미터 값 수 - 1).
    - Each Choice Testing (각 선택): 각 파라미터의 각 값을 최소 1회 포함. TC 수 = max(파라미터별 값 수)(가장 적음).
    - TC 수 순서 (많은 순): All Combination > Pair-wise > Base Choice > Each Choice.
- AI·ML 테스트
    - Metamorphic Testing (메타모픽 테스트, 변성 테스트): 입력 변환과 출력 간의 메타모픽 관계(예: f(A,B) = f(B,A) 대칭성)를 정의해 새 테스트케이스를 생성. 테스트 오라클 문제 해결용으로 과학 계산·검색 엔진·ML 모델에 유효.
    - Neuron Coverage (뉴런 커버리지): 테스트 입력으로 활성화되는 뉴런 비율 측정. 비활성 뉴런을 커버하도록 데이터를 보강.
    - Layer Coverage (레이어 커버리지): 각 레이어의 활성화 여부를 기준으로 하는 커버리지.
    - Sign-sign Coverage (부호-부호 커버리지): 인접 레이어 뉴런 간 활성화 부호 조합을 커버.
    - Adversarial Attack (적대적 공격): 인위적 노이즈를 추가한 입력(adversarial example)으로 AI 모델 오동작을 유도.
    - Coverage Testing (일반 커버리지 테스트): 코드 실행 경로 커버율 측정이지 AI 강건성·신뢰 영역 측정 기법이 아님(혼동 주의).
    - Component Coverage는 전통적 SW 테스트 개념으로 신경망 테스트 기준이 아님.
- AI 강건성 거리 척도
    - Euclidean Distance (유클리드 거리): 두 점 사이의 직선 거리. 기하학적 최단 경로이며 AI 강건성 반경의 기본 지표.
    - Manhattan Distance (맨해튼 거리): 수직·수평 이동만 허용하는 L자형 거리. |x1-x2| + |y1-y2|.
    - Chebyshev Distance (체비쇼프 거리): 각 축 차이의 최댓값. max(|x1-x2|, |y1-y2|).
    - AI Robustness (강건성): 두 입력의 거리가 일정 범위 이내일 때 동일 출력을 보장하는 특성.
- 테스트 성숙도·표준
    - TMMi (Test Maturity Model integration, 테스트 성숙도 통합 모델): 테스트 프로세스 성숙도를 L1~L5로 평가. L1 Initial, L2 Managed, L3 Defined, L4 Measured, L5 Optimization.
    - TMMi Level 2 Test Planning 세부 목표: 제품 위험 평가(Perform a product risk assessment), 테스트 접근 방식 설정(Establish a test approach), 테스트 추정치 설정(Establish test estimates), 테스트 계획서 수립, 테스트 계획 합의 획득. 성능 지표(Performance Indicators) 설정은 Level 4 활동이므로 L2 목표 아님.
    - ISO/IEC/IEEE 29119 Part 2 적합성: Full Conformance(전체적 적합성)는 모든 프로세스의 모든 요구사항 충족, Tailored Conformance(맞춤형 적합성)는 조직이 선택한 일부 프로세스에 대해 해당 프로세스의 요구사항을 전부 충족. 맞춤형도 선택 범위 내에서는 요구사항 전부 만족해야 한다.
- JUnit4 실행 순서·Release Testing
    - @BeforeClass: 클래스 로딩 후 단 1회 실행(static 메서드).
    - @Before: 각 @Test 메서드 실행 전마다 실행.
    - @Test: 테스트 메서드 실행.
    - @After: 각 @Test 메서드 실행 후마다 실행.
    - @AfterClass: 모든 테스트 완료 후 단 1회 실행(static 메서드).
    - 실행 순서: BeforeClass → (Before → Test → After) × 테스트 수 → AfterClass.
    - Release Testing (릴리즈 테스팅): 외부 배포 전 수행하는 요구명세 기반 블랙박스 테스트. 개발팀과 독립된 테스트팀이 수행하여 객관성 확보.
- E2E (End-to-End) 테스트
    - 애플리케이션 흐름을 처음부터 끝까지
    - 시나리오 테스트와 유사하지만 더 포괄적인 개념
- Scenario Testing (시나리오 테스트)
    - 사용자가 실제로 사용하는 시나리오를 기반으로 테스트


# 2.8 유지보수 및 운영
- 유지보수 유형 (IEEE 14764)
    - Corrective Maintenance (수정형 유지보수): 소프트웨어의 버그·오류를 수정하는 유지보수.
    - Adaptive Maintenance (적응형 유지보수): 운영 환경(하드웨어·OS·법규 등) 변화에 맞춰 소프트웨어를 수정.
    - Perfective Maintenance (완전형 유지보수): 기존 기능 유지하며 성능 향상·기능 개선을 위한 유지보수. DB 재구성을 통한 검색 속도 향상·스토리지 절감이 대표 사례.
    - Preventive Maintenance (예방형 유지보수): 미래 문제 발생을 예방하는 유지보수. 리팩토링·문서화 개선 등.
- ITIL 4 Practice 분류
    - ITIL 4 SVS (Service Value System, 서비스 가치 시스템): IT 서비스 관리 프레임워크. Practice를 3가지 범주로 구분한다.
    - General Management Practices (14개): 전사적 관리 실천. Business Analysis, Architecture Management, Continual Improvement, Information Security Management, Knowledge Management, Measurement and Reporting, Organizational Change Management, Portfolio Management, Project Management, Relationship Management, Risk Management, Service Financial Management, Strategy Management, Supplier Management, Workforce and Talent Management.
    - Service Management Practices (17개): IT 서비스 관리 특화. Change Control, Incident Management, IT Asset Management, Service Level Management, Service Desk, Availability Management, Capacity & Performance Management, Monitoring & Event Management, Problem Management, Release Management, Service Catalogue Management, Service Configuration Management, Service Continuity Management, Service Design, Service Request Management, Service Validation & Testing, Business Analysis.
    - Technical Management Practices (3개): 기술 관련 실천. Deployment Management(= Software Deployment and Management), Infrastructure and Platform Management, Software Development and Management.
    - 분류 구분 주의: Software Deployment and Management는 Technical Management Practice이며 Service Management Practice가 아니다.
- 유지보수 절차·IT 거버넌스 (보강)
    - 유지보수 절차: 변경 요청 접수 → 영향 분석 → 릴리즈 계획 → 변경 구현·테스트 → 배포.
    - COBIT (Control Objectives for IT, IT 통제 목표): ISACA의 IT 거버넌스·관리 프레임워크.
    - ISO/IEC 38500 (IT 거버넌스 국제 표준): 조직의 IT 사용에 대한 거버넌스 원칙.

# 2.9.1 품질 표준·프로세스 성숙도
- ISO/IEC 25010:2023 품질 특성 (최신 개정, 2011판 대체)
    - **구조적 변화**: 기존 2011판의 '제품 품질 모델 + 사용 품질 모델' 통합 → **제품 품질(25010)**과 **사용 품질(ISO/IEC 25019)**로 분리.
    - **9개 품질 특성 (8→9개로 증가)**
        - 기능적합성 (Functional Suitability)
        - 성능효율성 (Performance Efficiency)
        - 호환성 (Compatibility)
        - **상호작용 능력 (Interaction Capability)** ← 구 Usability(사용성) 명칭 변경
        - 신뢰성 (Reliability)
        - 보안성 (Security)
        - 유지보수성 (Maintainability)
        - **유연성 (Flexibility)** ← 구 Portability(이식성) 명칭 변경
        - **안전성 (Safety)** ← 신규 추가 (현대 ICT·임베디드 시스템 대응)
    - **주요 명칭 변경**
        - Usability(사용성) → **Interaction Capability(상호작용 능력)**: 사용자-시스템 상호작용 역량 강조.
        - Portability(이식성) → **Flexibility(유연성)**: 확장성(Scalability) 하위 특성 추가 포함.
    - **하위 특성 주요 변경**
        - Reliability: Maturity(성숙성) → **Faultlessness(무결함성)**으로 명칭 변경.
        - Interaction Capability 신규 하위 특성: **Inclusivity(포용성)**, **Self-descriptiveness(자기 설명성)** 추가; Accessibility → Inclusivity + User assistance로 분리; User interface aesthetics → **User engagement(사용자 참여)**로 변경.
        - Security: **Resistance(저항성)** 하위 특성 추가.
        - Flexibility: **Scalability(확장성)** 하위 특성 추가.
    - **Safety(안전성) 주요 하위 특성**: 운영 제약(Operational constraint), 위험 식별(Risk identification), 페일 세이프(Fail-safe), 위험 경고(Hazard warning), 안전한 통합(Safe integration).
    - Reliability 하위 특성 (2023): Faultlessness(무결함성), Availability(가용성), Fault Tolerance(결함 허용성), Recoverability(복구 가능성).
    - 혼동 주의: 합리적 결과 비율 → Functional Correctness(기능성); 비인가 접근 데이터 변조 방지 → Security; Adaptability(적응성)는 Flexibility 하위 특성(구 Portability 하위).
    - Maintainability 하위 특성 (2023 유지): Modularity(모듈성), Reusability(재사용성), Analysability(분석성), Modifiability(변경성), Testability(시험가능성).
    - **함정**: 2011판 기준으로 작성된 문제에서 'Usability·Portability·Maturity' 용어 사용 시 구 버전 기준임을 인식하고 맥락 파악 필요.
- ISO/IEC 5055 (CISQ 표준)
    - CISQ (Consortium for IT Software Quality, IT 소프트웨어 품질 컨소시엄): ISO/IEC 5055의 개발·관리 주체.
    - 4대 소스코드 품질 특성: Reliability, Security, Performance Efficiency, Maintainability. 자동화된 정적 분석으로 측정.
    - Usability 제외: 사용자 인터페이스·경험은 소스코드 정적 분석으로 측정 어려워 5055 범위 밖. Usability는 ISO/IEC 25010(SQuaRE)에서 다룬다.
- CMMI-DEV 프로세스 성숙도
    - CMMI-DEV (Capability Maturity Model Integration for Development): SW 개발 프로세스 품질 향상 가이드라인. 22개 프로세스 영역.
    - 4가지 카테고리: Process Management, Project Management, Engineering, Support.
    - Level 3 (Defined, 정의) 프로세스 영역: Risk Management(RSKM), Integrated Project Management(IPM), Organizational Process Definition(OPD), Organizational Process Focus(OPF), Decision Analysis and Resolution(DAR) 등 조직 전체 표준 프로세스가 정의·조정되는 단계.
    - Level 4 (Quantitatively Managed, 정량 관리): Quantitative Project Management(QPM), Organizational Process Performance(OPP).
    - 카테고리 분류 예: Configuration Management(CM)·DAR·Causal Analysis and Resolution(CAR)은 Support, Technical Solution(TS)은 Engineering.
- SPICE (ISO/IEC 15504) 성숙도
    - SPICE (Software Process Improvement and Capability dEtermination): 프로세스 개선·능력 판정 국제 표준.
    - 프로세스 능력 레벨 6단계: L0 Incomplete(불완전) → L1 Performed(수행) → L2 Managed(관리) → L3 Established(확립) → L4 Predictable(예측 가능) → L5 Optimizing(최적화).
    - CMMI Level 3 Defined ↔ SPICE L3 Established로 명칭이 다르므로 혼동 주의.
- SP 인증 제도
    - SP (Software Process, 소프트웨어 프로세스) 인증: SW 개발 기업의 프로세스 품질 수준을 평가·인증하는 국내 제도. SPICE (ISO/IEC 15504) 기반.
    - NIA (National Information Society Agency, 한국지능정보사회진흥원): SP 인증 주관 기관.
    - 평가 척도: 2차원(프로세스 차원 × 능력 차원). 능력 차원은 수준 0~5단계.
    - 인증 등급: 1등급(최고) ~ 3등급. 3등급 신청 시 미달하면 인증 미부여(하향 자동 부여 없음).
    - 평가 기준 구성: 5개 영역, 14개 평가항목, 60개 세부 평가항목.
    - SP 2등급 영역: 프로젝트 관리, 요구사항 관리, 품질보증, 형상 관리, 협력업체 관리(SAM), 측정·분석(MA), 프로젝트 모니터링·제어(PMC).
    - SP 3등급 이상: 조직 차원 교육 훈련(OT), 조직 프로세스 정의·집중 등 조직 수준 활동 포함.
- 신뢰성·Dependability
    - Dependability (확실성): 시스템의 신뢰 가능성 종합 개념. Availability + Reliability + Safety + Security + Resilience로 구성.
    - Availability (가용성): 주어진 시점에 시스템이 작동 중이고 유용한 서비스를 제공할 확률. A = MTBF / (MTBF + MTTR).
    - Reliability (신뢰성): 정해진 조건에서 기대한 기능을 지속 수행하는 능력.
    - Resilience (복원성·회복 탄력성): 장비 고장·사이버 공격 등 손상 사건에서 중요 서비스 유지 능력.
    - Security (보안성): 승인되지 않은 접근으로부터 시스템을 보호.
    - Fault Avoidance (결함 회피): 강한 자료형 언어 사용·포인터 등 오류 유발 요소 최소화로 설계·코딩 단계에서 결함 유입 방지.
    - Fault Detection and Correction (결함 감지 및 정정): 체계적 테스팅·디버깅·정적 분석으로 기존 결함 제거.
    - Fault Tolerance (결함 내성): 결함이 있어도 서비스를 유지(이중화·체크포인트·예외 처리).
    - Fault Forecasting (결함 예측): 결함 빈도·가능성을 사전 예측·평가.
- ISO 22989:2022 AI 신뢰성
    - AI Trustworthiness (AI 신뢰성): AI 시스템의 신뢰할 수 있는 특성. 여러 속성으로 구성된다.
    - AI Robustness (강건성): 다양한 운영 조건·입력 변화에도 성능이 유지되는 능력(사전 내성).
    - AI Resilience (회복 탄력성): 사고 후 운영 상태를 빠르게 복구하며 이해관계자가 허용 가능한 일부 장애 유형에서 저하된 수준(degraded level)의 운영을 제공하는 능력(사후 복구).
    - AI Controllability (제어 가능성): AI 시스템의 동작을 인간이 의도적으로 조작·중단할 수 있는 능력.
    - AI Explainability (설명 가능성): AI 의사결정 과정과 결과를 인간이 이해할 수 있도록 설명하는 능력.
- CWE·보안 약점 체계
    - CWE (Common Weakness Enumeration, 공통 약점 열거): MITRE가 관리하는 SW·HW 약점 분류 체계. 악의적 행위자가 무단 진입·악의적 행동에 사용할 수 있는 아키텍처·소스코드 문제점 목록이며, 최근 보안을 넘어 신뢰성·안전성까지 범위가 확장된다.
    - CVE (Common Vulnerabilities and Exposures): 개별 취약점 사례 식별자.
    - CVSS (Common Vulnerability Scoring System): 취약점 심각도 점수화 체계.
    - 연결 흐름: CWE(약점 유형) → CVE(취약점 사례) → CVSS(심각도).

# 2.9.2 품질 지표·메트릭·신뢰성 측정
- McCabe Cyclomatic Complexity
    - Cyclomatic Complexity (순환 복잡도, CCN/V(G)): McCabe가 제안한 소프트웨어 제어 흐름 복잡도 지표.
    - 공식 1: V(G) = 결정 포인트(Decision Point) 수 + 1. 결정 포인트는 if·else if·for·while·do-while·case·삼항 연산자·&&·||. else는 이미 if에서 카운트되므로 추가 카운트 금지.
    - 공식 2: V(G) = E - N + 2P. E는 엣지 수, N은 노드 수, P는 연결 컴포넌트 수(단일 그래프면 1).
    - 의의: V(G) = 독립 경로 수 = 최소 테스트케이스 수. V(G) ≤ 10 권장, 초과 시 리팩토링 고려.
    - 시퀀스 다이어그램 적용: loop·alt·opt·break 등 결합 프래그먼트가 각각 결정 노드 1개로 카운트된다.
- CK (Chidamber & Kemerer) 객체지향 메트릭
    - LCOM (Lack of Cohesion in Methods, 메서드 응집도 부족): 공유 속성이 없는 메서드 쌍 수 - 공유 속성이 있는 메서드 쌍 수. 값이 클수록 응집도가 낮다.
    - RFC (Response For a Class, 클래스 응답): 클래스의 객체가 메시지를 받아 실행할 가능성이 있는 메서드 수. 값이 클수록 복잡도·오류 가능성 증가.
    - WMC (Weighted Methods per Class, 가중 메서드 수): 클래스 내 모든 메서드의 복잡도 합계.
    - CBO (Coupling Between Objects, 객체 간 결합도): 해당 클래스가 직접 결합(사용)하는 고유 클래스 수. Class1을 이용하는 다른 클래스는 포함하지 않는다.
    - DIT (Depth of Inheritance Tree, 상속 깊이): 루트로부터의 상속 계층 깊이.
    - NOC (Number Of Children, 자식 수): 직접 상속받는 자식 클래스 수.
- 컴포넌트 안정성 지표
    - Fan-in (입력 결합도): 해당 컴포넌트에 의존하는 외부 컴포넌트 수. 높을수록 안정적.
    - Fan-out (출력 결합도): 해당 컴포넌트가 의존하는 외부 컴포넌트 수. 높을수록 불안정.
    - Instability (불안정도, I): I = Fan-out / (Fan-in + Fan-out). 0이면 완전 안정(들어오는 의존성만), 1이면 완전 불안정(나가는 의존성만).
    - 안정적 패키지 특성: 다른 패키지가 많이 의존하고 자신은 적게 의존.
- 신뢰성 척도
    - MTBF (Mean Time Between Failure, 평균 고장 간격): 수리 가능한 시스템에서 고장과 고장 사이의 평균 시간. 지속 운영 시스템의 가용성 측정에 적합.
    - MTTF (Mean Time To Failure, 평균 고장 시간): 수리 불가능한 시스템에서 처음 고장이 날 때까지의 평균 시간. 수명 예측에 사용.
    - MTTR (Mean Time To Recovery/Repair, 평균 복구 시간): 장애로부터 복구될 때까지의 평균 시간. Recoverability(회복성) 척도.
    - ROCOF (Rate of Occurrence of Failure, 고장 발생 비율): 단위 시간당 고장 빈도. 운영 중 시스템의 고장 빈도 측정.
    - POFOD (Probability of Failure on Demand, 온 디맨드 고장 확률): 서비스 요청 시 실패 확률. 원자력 긴급 정지 시스템·항공기 안전장치 등 안전 필수(safety-critical) 시스템의 신뢰성 척도.
    - 가용성 공식: A = MTBF / (MTBF + MTTR).
- PCE·유지관리 용이성
    - PCE (Phase Containment Effectiveness, 단계 내 결함 포함 효과성): PCE = 해당 단계에서 생성되어 같은 단계에서 발견된 결함 수 / 해당 단계에서 생성된 전체 결함 수. 값이 클수록 결함이 다음 단계로 누출되지 않아 품질이 높다.
    - 유지관리 용이성 향상 원칙: 단일 진입/단일 출구, 높은 응집도, 낮은 결합도, 정보 은닉, 추상화, 하향식 계층 분해, 코딩 표준·지침 준수, 예외·부작용 대비 상세 설계.
    - 안티 패턴: 복수 엔트리·복수 엑시트 구조는 제어 흐름 복잡도를 증가시켜 유지관리 용이성을 저하시킨다. 구조적 프로그래밍의 단일 진입·단일 출구 원칙에 위배.

# 2.9.3 형상관리·Git·베이스라인
- SCM 4대 활동
    - SCM (Software Configuration Management, 소프트웨어 형상 관리): 소프트웨어 산출물의 변경을 체계적으로 관리하는 활동.
    - Configuration Identification (형상 식별): 베이스라인 설정 및 모든 형상 항목에 식별자를 부여.
    - Configuration Control (형상 통제): 변경 요청 평가·조정·승인 활동.
    - Configuration Status Accounting (형상 상태 보고): 형상 항목의 변경 히스토리 기록 및 보고.
    - Configuration Audit / Verification (형상 감사): 개발된 제품이 승인된 명세와 정확히 일치하는지 평가.
- 베이스라인·코드라인·메인라인
    - Codeline (코드라인): 개별 컴포넌트의 버전 시퀀스(예: A1.1→A1.2→A1.3). 단일 컴포넌트의 버전 이력.
    - Mainline (메인라인): 여러 코드라인이 통합되어 릴리즈를 향해 진행되는 전체 통합 라인. 통합 개발의 주요 흐름.
    - Baseline (베이스라인): 특정 시점에 공식적으로 승인된 컴포넌트들의 특정 버전 집합. 소프트웨어 개발 중 특정 시점·목적을 위해 만들어진 산출물의 집합.
- Git vs Subversion 저장 방식
    - Git (분산 버전 관리): 스냅샷(snapshot) 기반으로 변경된 파일 전체를 저장하며 변경 없는 파일은 이전 스냅샷 링크를 저장. 로컬 저장소 보유로 오프라인 작업 가능.
    - Subversion (중앙집중형 버전 관리): 델타(delta·차이) 기반 저장. 하나의 마스터 저장소에 모든 컴포넌트 버전을 유지.
    - 공통: 두 시스템 모두 commit으로 새 버전을 생성. Git은 로컬, SVN은 중앙 저장소에 커밋.
- Git 명령어
    - git clone: 원격 저장소를 로컬에 최초 복제.
    - git fetch: 원격 변경사항을 로컬로 가져오되 병합(merge)은 하지 않음.
    - git pull: git fetch + git merge 결합. 원격 저장소의 최신 내용을 로컬로 가져와 현재 브랜치에 병합.
    - git push: 로컬 변경사항을 원격 저장소로 전송.
    - git branch <name>: 새 브랜치 생성. HEAD는 현재 브랜치에 그대로 유지.
    - git checkout <name>: 지정 브랜치로 전환해 HEAD가 해당 브랜치를 가리키도록 한다. git switch <name>도 동일 역할.
    - git checkout -b <name>: 생성과 전환을 동시에 수행(git switch -c <name>과 동일).
    - git merge <name>: 현재 브랜치에 다른 브랜치를 병합.
    - git stash: 미완성 작업을 스택에 임시 저장. git stash pop으로 복원.
- Git 개념
    - Branch (브랜치): 특정 커밋 SHA를 가리키는 가변 포인터.
    - master / main: 기본 브랜치. 최초 커밋 시 생성되며 새 커밋마다 자동으로 최신 커밋을 가리킨다.
    - HEAD: 현재 작업 중인 브랜치·커밋을 가리키는 포인터.
    - 병합 원칙: 토픽 브랜치의 변경은 git merge 또는 Pull Request로 명시적 병합이 필요하며 커밋만으로 자동 병합되지 않는다.
