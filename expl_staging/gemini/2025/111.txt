정답: 2 (②)

**해설:**
웹 어플리케이션과 뒷단 데이터베이스 서버를 잇는 SQL 코딩에서, 보안 필살기인 **Prepared Statement(선처리 바인딩 컴파일)**와 **Stored Procedure(저장 프로시저)** 콤보를 썼을 때 가장 기적적으로 철통 방어가 닫혀 해결되는 단골 해킹 취약점을 묻습니다.

**상세 분석:**
* 해커가 웹사이트 로그인 아이디 칸에 `admin' OR '1'='1` 처럼 요상한 따옴표 해킹 주문을 넣었을 때 일반 사이트의 처리 결과는 끔찍합니다.
* 일반적인 String 텍스트를 그냥 더하기(+)로 얼기설기 붙여 조립한 쿼리문은 저 문장을 DB의 실제 명령어 구조(조건식 무조건 참 판단)로 인식해버려 덜컥 우회 프리패스 문을 뚫어주는데, 이 참사가 바로 그 유명한 OWASP 1위 **SQL 인젝션(삽입)** 공격입니다.
* **1. Prepared Statement (선처리 구문)의 마법 방어막:** DB 엔진에 "야 내가 `SELECT * FROM USERS WHERE ID = ?` 형태의 빵꾸 뚫린 틀판 쿼리 뼈대를 먼저 던져둘게 준비해. 그리고 나중에 아이디 칸에 입력 들어온 외부 빌런 놈(`admin' OR '1'='1`)을 그 빵꾸에 총알 파라미터로 장전할 텐데, 저놈이 무슨 기괴한 따옴표 마법 주문을 쓰든 간에 **명령어가 아닌, 오직 순수 텍스트 값 자체로만 치환 취급 보호해!**" 라고 선언해버립니다. 해커의 악의적 SQL 제어 문장은 데이터화 되어 그냥 무해하게 씹혀버립니다.
* **2. Stored Procedure (저장 프로시저):** 아예 쿼리를 앱 소스 구동부에 안 두고, DB 서버 커널 안에 컴파일된 C언어 함수 캡슐 조각처럼 미리 박아놔 숨겨 호출만 시켜, 외부 변조 논리의 중간 강제 템플릿 삽입을 구조상 원천 봉쇄합니다.

**📌 핵심 방어막 요약:**
* Prepared Statement (물음표 파라미터 바인딩) / Stored Procedure (프로시저 로직 블록화 사전 컴파일) 구현의 가장 절대적이고 유일한 도입 목적 = 웹 해킹의 무덤이라 불리는 **SQL Injection (통역 변조 삽입 쿼리)** 을 데이터 단순 치환으로 원천 컷 기법 차단.
