정답: 3 (③)

**해설:**
법적 의무화로 붐이 크게 인 '데이터베이스(DB) 암복호화 적용 솔루션 프레임워크' 2종류인 **Plug-In(플러그인) 방식**과 **API 방식**의 인프라 구축 특징 및 타겟 아키텍처 설명을 크로스시켜 거짓말을 하는 눈속임 지문을 색출합니다.

**상세 분석:**
* **API 방식:** WAS(웹/애플리케이션 서버)단 안의 개발자들 소스코드에 암호화 로직 함수인 API 라이브러리(Jar 등)를 직접 임포트해 넣습니다. ➔ 장점: 앱 서버가 뼈 빠지게 암호화를 돌리고 DB 서버엔 이미 암호화 완료된 짐만 시키므로, 메인 센터인 DB 서버 자체의 성능 지연 과부하 영향은 거의 없습니다. ➔ 단점: 개발자가 소스 프로그램 대공사 수정을 모조리 뜯어고쳐야 하는 막노동 필수. (②번 설명 완벽 매칭 O).
* **Plug-In(플러그인) 방식:** 앱 서버 소스는 안 건드리고, DB 서버 엔진 내부 골수단에 암호 프레임워크 칩(플러그인 모듈 뷰)을 아예 스윽 끼워 이식해 박아버리는 방식. ➔ 장점: 개발자는 소스 수정 거의 없이 기존처럼 평문으로 짜고 던지면, DB 내부에서 알아서 가로채 투명하게(TDE처럼) 암/복호화가 돌아감 (소스 배포 수정 최소화 쌉이득, ①번 매칭 O). ➔ 단점: 그러나 무거운 DB가 스스로 암호화 땀방울까지 뻘뻘 흘리며 돌리므로 DB 병목 서버 성능이 박살나는 터짐 과부하 현상 발생 위험성 존재.
* **하이브리드 병용 (④번):** 위 두 장점을 스까서 대용량 성능이 터질 것 같은 구간 트랜잭션엔 API 방식 연동을 앱서버로 위임시키고, 안 무거운 구간엔 앱 개발 안 편하도록 그냥 플러그인 뷰를 돌리는 유동방식. (적절 O)
* **③ 지문 해부 검거 (정답/오답 속임수):** "Plug-In 방식은 솰랴솰라... `암복호화 모듈이 라이브러리 형태로 각 애플리케이션 서버에 설치되고`, 응용 프로그램에서 해당 모듈을 호출 하는 방식이다." 
  * 아오, 이것은 방금 위에서 제일 처음 설명한 개발자가 뼈빠지게 호출하는 전형적인 **API 방식**의 거동 매커니즘을 토씨 하나 안 틀리고 그대로 묘사한 것입니다. 앞에 Plug-In 이라는 주어를 걸어둬서 눈속임 사기 거짓말을 시전 중입니다.

**📌 핵심 아키텍처 요약:**
* Plug-in 모델 = 암호화 도구를 "DB 서버" 심장에 인서트로 이식한다 ➔ 소스 수정 필요 이득. 단 DB 심박수 부하 터짐.
* API 모델 = 암호화 라이브러리를 "App(웹/애플리케이션) 서버" 소스 단에 함수 코드로 호출 탑재한다 ➔ 소스 수정 개노가다 헬. 단 DB 부하 무풍지대 보호.
