정답: 2 (②)

**해설:**
SQL DDL(데이터 정의어) 테이블 생성 쿼리문에서 부모-자식 간의 참조 무결성을 견고하게 유지하기 위해 세팅해 놓은 제약 조건 트리거(`ON DELETE` / `ON UPDATE`)의 방어 기제 동작 원리를 묻는 문제입니다.

**상세 분석:**
* **FOREIGN KEY (DNO) REFERENCES DEPARTMENT(DNO)** 외래키 제약을 살펴봅시다.
* `ON DELETE SET DEFAULT`: 이것은 부모 부서(DEPARTMENT) 테이블 장부에서 어떤 부서 한 개가 통째로 삭제되어 폐쇄될 경우, 그 부서에 속해 일하던 자식 직원(EMPLOYEE)들의 모가지도 연달아 뎅겅 날리는 연쇄 폭파(CASCADE)를 하지 말고, 애초 테이블 스키마에 세팅해둔 **`DEFAULT 값(=1)`으로 그들의 남겨진 소속 컬럼을 자동/강제로 편입 갈아 끼워 넣어 버려라** 라는 일종의 구명조끼 방어선 의미입니다.
* `ON UPDATE CASCADE`: 반대로 부모 부서 번호 자체가 다른 문자로 갱신/승격되어 업데이트 칠해지면, 자식 직원들의 가슴 번호표도 모조리 다 수동 작업 없이 알아서 연쇄 변경 갱신(Cascade)해 주라는 뜻입니다.
* **②번 (정답: 틀린 설명):** "100번 직원이 속한 부서 조직이 폐쇄/삭제 날아가면 그에 소속된 100번 직원도 연쇄적으로 삭제된다." 
  * 만약 DDL 스크립트에 `ON DELETE CASCADE` 라고 적혀있었다면 윗물이 증발할 때 아랫물도 가차 없이 날아가는 게 맞습니다. 
  * 그러나 스크립트에 `ON DELETE SET DEFAULT` 로 방어 세팅을 다르게 걸어두었으므로, 직원의 생명은 해고당하지 않고 살려 두면서 무소속 미아 대기 발령소인 **1번(DNO=1 디폴트) 부서로 자동 좌천배치만 되는 것**이 올바른 메커니즘 전개입니다.

**📌 핵심 요약:**
* `ON DELETE / UPDATE CASCADE` = 윗물 부모가 삭제되거나 값 변형되면, 아랫물 자식 튜플들도 죽거나 같이 변형됨 (연쇄 동기화 폭발).
* `ON DELETE SET DEFAULT` / `SET NULL` = 윗물 부모가 삭제 죽어버리면, 아랫물 전체가 죽는 게 아니라 그 자식 속성값 칸에만 Default나 비버림(NULL)값 구명조끼가 씌워져 빈칸만 남고 생존함.
