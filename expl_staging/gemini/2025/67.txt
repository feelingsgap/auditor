정답: 1 (①)

**해설:**
페타바이트(PB) 단위 빅데이터 조각들을 거대한 수만 대 머신 클러스터 군단에 수평 쪼개어 분산 보관 관리하는 **HDFS (Hadoop Distributed File System)** 생태계 구조 특성 내에서, Read (데이터 빼오기) 액션과 Write (데이터 쑤셔 넣기 저축 완료) 사이의 극명한 속도 성능 레이턴시 차이 원인 아키텍처 특성과 병목 딜레이 유발 사유 메커니즘을 날카롭게 판별합니다.

**상세 분석:**
* 기본적으로 하둡 구조의 뇌 조직(Headquarters): 네임노드(NameNode)는 데이터 위치 족보 메타 정보 주소록 관제소. / 체력 조직(Workers): 데이터노드(DataNode)들은 실제 어리바리하게 물리 데이터 파일 덩어리를 품에 안고 구르는 머신 일꾼 창고들.
* **HDFS 데이터 읽기(Read)의 축복받은 쾌속 루트 구조:** 하둡 네임노드 중앙 대빵이 "너가 찾는 그 파일 조각 퍼즐들 C노드 창고, H노드 창고, K노드 창고에 흩어져 있어"라고 3곳 주소를 딱 찍어 스케줄링해주면, 데이터 찾으러 들어간 앱(클라이언트)은 그중에서 아파트 복도 동선이 제일 가깝고 쾌적한 최단 거리 네트워크 통신 홉(Rack Awareness 거리) 상의 **가장 빠른 데이터 노드 창고 한 군데만 다이렉트 콕 집어서 몰래 들어가 초고속 순차 스트리밍 땡기듯 텍스트를 쭉 읽어오면 상황 종료 끝납니다.** 여러 군데 들쑤시며 결재 도장 기다릴 필요가 전혀 없어 체감상 아주 **가볍고 빠릅니다.** 
* **HDFS 데이터 쓰기(Write)의 분노 폭발 고통스러운 병목 지연:** 그러나! 앱(클라이언트)이 창고에 새 데이터 장부를 적어 넣고 쓸 때(저장할 때)는 장애가 나서 하드가 불타는 디재스터 상황 시 복원 구명조끼 대비를 위해 하둡 헌법상 무조건 3개의 완전히 다른 외딴 노드 서버 창고(기본 Replication Factor = 디폴트 3배수 세팅)에 동일한 오리지널 파일 쌍둥이 앵무새 복사 덩어리들을 안전하게 뿌려 심어 안치해야 데이터 저장 완료라는 승인을 줍니다. 
* 결정적인 느림의 원인은 이거를 한 번에 동시 샷건으로 빵 쏘는 게 아닙니다! 노드 일꾼 A에 저장 ➔ A가 노드 창고 B에 복사 파이프라인 토스 ➔ B가 노드 창고 C 창고 순서로 파이프라인 릴레이 전달을 연쇄적으로 타고 가며, 저장을 징검다리 끈끈이 복사 처리합니다. 
* 그리고 최후방 후위대인 마지막 C가 하드디스크에 물리 도장 쾅 잘 찍고 저장이 완벽하게 100% 성공(ACK 수령 사인백) 할 때까지 앞단에서 기다리는 클라이언트 앱은 영겁의 시간 동안 먹통 정지 락에 걸려 계속 끝날 때까지 바닥에 주저앉아 **목 빠져라 멍 때리며 대기 해야 합니다.** 이 파이프라인 릴레이 병목 딜레이 체인이 필연적으로 발생하며 이 현상 때문에 체감 대기 속도가 Write시 어마 무시하게 **지연되고 심각하게 느립니다.** (강박적 데이터 안전 저축 보장의 대가)
* 고로, 데이터 유실, 오염 방지 및 무결성 담보를 위한 무식한 3중 복제본 파이프 연쇄 뿌리기 저장 대기열 병목(Replication 3 Pipeline Delay) 강제가 Write 성능 대폭락 체감 저하의 결정적 핵심 범인이자 아키텍쳐 딜레이 이유입니다.

**📌 핵심 요약 필살기:**
* HDFS Read 액션 = 최단 거리 인접 레이턴시 가장 낮은 1개 노드 만 패스트 픽업 ➔ 초고속 순방향 스트림 전송 뽑기 (다이렉트 스루풋이라 체감상 매우 빠름).
* HDFS Write 액션 강압 = 장애 방어 용 3개 외딴 격오지 노드 쌍둥이 복사본을 일렬 징검다리 연쇄 건너뛰며 파이프라인 복사 뿌리기를 거치고, 라스트 맨의 완벽 저축 완료 동기화 컨펌 떨어지는 걸 락 걸리고 강제 대기 ➔ (파이프라인 통신 대기로 인해 개체 수증가할수록 더 개판으로 느려짐).
