정답: 3 (③)

**해설:**
웹 화면이나 입력 폼에서 개발자가 멍청하게 사용자 입력 파라미터 값(문자열 길이, 필터링 등)의 악의적 장난질 치환 여부를 대충 검사 안 하고 통과시켰을 때 불거지는 입력 폼 해킹이 **'아닌'**, 운영체제의 완전 다른 프로세스 스케줄링 원리 카테고리에 속하는 버그를 찾습니다.

**상세 분석:**
* **① 포맷 스트링 삽입 결함 (입력 검증 계열 O):** C언어 등에서 사용자가 입력창에 `%x %s %n` 같은 괴상망측한 포맷 지정자 문구를 미친 듯이 악의적으로 때려 넣었을 때 처리 치환 검증을 안 막아두면 메모리 안감이 열리며 정보가 탈탈 뒤집어지며 박살 납니다. 전형적인 입력 검사 부실 해킹.
* **② 미검증 자동접속 리다이렉트 취약점 (입력 계열 O):** 유저 로그인 폼 파라미터에 사용자 입력으로 들어오는 `return_url=https://해커의구라은행.com` 이란 목적지를 화이트리스트 묻지도 따지지도 않고 그대로 넘겨서 해커 낚시 사이트로 정상 고객을 떨궈버리는 OWASP 톱 입력 검증 취약점.
* **④ 메모리 버퍼 오버플로우 폭주 (입력 계열 O):** 댓글에 10글자만 할당해 넣으랬는데 길이 검증 스크립트 미적용 생략 스킵을 노려 특수문자 10만 글자 폭탄을 한 폼에 밀어 넣어 C프로그램 스택 버퍼 힙 경계를 무너뜨려 크래시 시키는 짓. 전형적 입력 공격.
* **③ 경쟁 조건 (Race Condition / TOCTOU) (정답 지정 / 기계적 계열 탈락):**
  * 보안 경쟁 조건은 해커가 입력값을 이상한 특수문자 등으로 줘서 생기는 문제가 절대 아닙니다! 
  * 서버에서 여러 가닥의 쓰레드(Thread)나 프로세스가 하나의 공유 변수 파일 자원에 **"서로 간발의 타이밍을 다투어 코드를 동시에 들쑤시고 쓸 때(0.001초 차이 레이싱 병목 렉)"**, 선착순 상호배제 문 잠금장치 락(Mutex)을 안 걸어놔서 연산 박자가 물리적으로 꼬여 출금 수치나 권한이 오류나는 **OS 운영체제 하부 동기화 결함 버그**입니다. 입력 데이터 필터링 부족 이슈와는 궤가 아예 다른 시스템 스케줄링 구조론적 이슈입니다.

**📌 핵심 발현 메커니즘 요약:**
* 입력 데이터 검증 누락 계열: 웹 방화벽 필수 영역. 버퍼오버플로우 터짐, XSS 팝업, 포맷스트링 유출, SQL 인젝션, 미검증 리다이렉트 이동, 커맨드 파라미터 인젝션.
* 코드 프로세스 상호배제 타이밍/접근 스케줄링 오류 계열: 경쟁 조건(Race Condition). 잠금락 교착상태(Dead Lock).
