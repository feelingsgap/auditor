정답: 4 (④)

**해설:**
JAVA 애플리케이션 코딩에서 관계형 DB(Oracle, MySQL 등)에 접속해 데이터를 찌르고 받아오는 중간 다리 표준 인터페이스인 JDBC(혹은 ODBC) 코어 API 함수들의 라이프 사이클(작동 제어 순서도) 흐름을 묻는 문제입니다.

**상세 분석:**
* 데이터베이스 접속 프로세스는 현실에서의 "자동응답기 전화 통화 걸어서 목적 묻고, 대답 파일 받아오기"와 완전히 빼다 박은 동일한 알고리즘을 가집니다.
  1. 먼저 무전기 전원을 켜고 내 주파수 세팅과 상대방 접속 IP 주소를 맞춰야 함 ➔ **다. (환경 변수 클라이언트 설정, DB 연결 Connect 초기화)**
  2. 서버로 보낼 통일문(SQL) 양식을 "내가 이러이러한 Insert/Select 뼈대 문장 보낼 거니까 미리 파싱해서 조립 완료 스탠바이 하고 있어라" 라고 쏴줌 ➔ **마. (SQLPrepare, 보안과 효율을 위한 PreparedStatement 객체 초기화 등 문장 준비)**
  3. "저번 뼈대 서식에 빈칸 뻥 뚫어둔 그 물음표(?) 구멍에 들어갈 암호는 '홍길동' 이야" 라고 매개변수 실탄 장전 ➔ **나. (SQLBindParameter 맵핑 작업)**
  4. "장전 끝났다. 완벽하다, 발사!! 쿼리 실행하고 실행 결과 뱉어내라!" ➔ **가. (SQLExecute 명령 엑티브)**
  5. 서버에서 쿵 하고 떨어진 거대한 데이터 결괏값(Result Set) 메모리 덩어리의 열람 추출을 위해 "DB 1열은 내 Java String 변수에, 2열 부서번호는 Java Int 변수에 기입해라" 라우팅 쫙쫙 연결 수확하기 ➔ **라. (SQLBindCol로 자바/C 언어 프로그램 변수에 응답 결과 연결)**
* 결국 이 작업 공정의 가장 논리적이고 빈틈없는 퍼펙트 스텝은 **다 ➔ 마 ➔ 나 ➔ 가 ➔ 라** 테크트리가 됩니다.

**📌 핵심 요약 메모:**
* JDBC 순서도 대원칙: DB 커넥트(환경 세팅 다) ➔ Prepared Statement 골조 세워두기(마) ➔ 파라미터 값 매핑 바인딩(나) ➔ Execute 발사(가)! ➔ Result 결과 변수에 Fetch 적재(라)!
